# Architecture Decision Records

Append-only лог зафиксированных решений. Прошлую запись не переписываем — при изменении помечаем `superseded` и добавляем новую со ссылкой. Записи ниже **кодифицируют ранее принятые** решения (собраны из кода, `CLAUDE.md`, продуктовых доков и аналитики); дата кодификации — 2026-07-02, если явно не указано иное.

### ADR-001 · LCOS — это точка ввода накладных, не POS
- **Статус:** accepted
- **Дата:** 2026-07-02 (pre-existing)
- **Контекст:** соблазн разрастись в операционный учёт/POS.
- **Решение:** LCOS собирает и анализирует данные поверх ERP; он **не** POS и **не** операционный учёт. Tablet/POS-интеграция и бухгалтерия остаются в ERP (Esupl). Конкретно система — write-point накладных: фото → OCR → маппинг на SKU → валидация → payload Esupl → локальное сохранение + (за гейтом) запись в POS.
- **Следствия:** зарплаты, смены, финотчётность — вне объёма. Граница продукта чёткая.
- **Связи:** 00_PRODUCT, 01_ARCHITECTURE.

### ADR-002 · Human-in-the-loop
- **Статус:** accepted · **Дата:** 2026-07-02 (pre-existing)
- **Контекст:** доверие владельца к AI, риск ошибочного действия.
- **Решение:** AI показывает данные и аргументы — **решает человек**. Нет автозаказов; нет записи в POS без подтверждения человека.
- **Следствия:** запись в ERP всегда за подтверждением; UI обязан давать sanity-проверки.
- **Связи:** ADR-009 (ERP write gating), 00_PRODUCT.

### ADR-003 · Фазовая дисциплина и Wife-Gate
- **Статус:** accepted · **Дата:** 2026-07-02 (pre-existing)
- **Контекст:** классическая ловушка «строим для рынка — никто не покупает».
- **Решение:** Фаза 1 — личный инструмент для Customer Zero (жена, владелец кофейни). Переход в Фазу 2 гейтится критерием: жена сама говорит «без этого хуже» + сэкономлено ≥3 ч/нед после 4 недель использования. «Сначала помочь жене. Потом — деньги.»
- **Следствия:** коммерческая амбиция — опция, не обязательство; нет преждевременного давления рынка.
- **Связи:** 00_PRODUCT, 03_ROADMAP.

### ADR-004 · Esupl как первичная ERP
- **Статус:** accepted · **Дата:** 2026-07-02 (pre-existing)
- **Контекст:** Customer Zero работает на белорусском POS Esupl.
- **Решение:** первичная (единственная на Фазе 1) ERP — Esupl. `organization ↔ ровно один Esupl team`; `subdivision ↔ Esupl warehouse`. iiko и прочие — Фаза 2.
- **Следствия:** ERP-провайдер выбирается статически (`ERP_PROVIDER` из env); seam для других ERP существует, но не реализуется до триггера.
- **Связи:** ADR-008, 01_ARCHITECTURE.

### ADR-005 · Трёхуровневое разделение конфигурации/секретов, без env-фолбэка
- **Статус:** accepted · **Дата:** 2026-07-02 (pre-existing, config-arch-review)
- **Контекст:** секреты в коде/env и необходимость редеплоя ради смены ключа.
- **Решение:** три уровня. (1) `.env` — только boot/trust-root (DB, KEK, JWT/session, admin-оператор, статический ERP-выбор, URL, cookie-флаги, VPN-декларатив). (2) `system_settings` (БД) — несекретные рантайм-настройки. (3) `integration_credentials` (БД, шифр.) — интеграционные секреты. Для уровней (2)/(3) **нет фолбэка на env**.
- **Следствия:** каждое значение имеет ровно один источник; AI-ключи/toggles/POS-токен ушли из env в БД, управляются супер-админкой.
- **Связи:** REQ R1, ADR-006, ADR-010.

### ADR-006 · Fail-closed везде
- **Статус:** accepted · **Дата:** 2026-07-02 (pre-existing)
- **Контекст:** тихая деградация опаснее явного отказа.
- **Решение:** отсутствие/недоступность зависимости → жёсткая ошибка, не деградация. VPN down (при `ai_vpn_enabled`) → `VpnUnavailableError`, без тихого direct-egress. Нет AI-ключа → `AiUnavailableError`. Нет POS-токена → Esupl 401. Шифртекст при пустом keyring → `RuntimeError`. `erp_write_enabled` по умолчанию False.
- **Следствия:** нет «запасных путей»; предсказуемость важнее доступности.
- **Связи:** REQ R8, BACKLOG A1 (открытый plaintext-фолбэк `encrypt()` — исключение, подлежит устранению).

### ADR-007 · Две независимые плоскости аутентификации
- **Статус:** accepted · **Дата:** 2026-07-02 (pre-existing, CLAUDE.md §11)
- **Контекст:** смешение админ- и пользовательской auth = риск эскалации.
- **Решение:** app-пользователи (`users`, argon2, JWT access + opaque refresh, куки) и SQLAdmin-оператор (env `ADMIN_USERNAME`+bcrypt, session-cookie, нет строки в `users`) — раздельны и не смешиваются.
- **Следствия:** два набора hash/verify; оператор не заводится в `users`.
- **Связи:** REQ R3, R4.

### ADR-008 · Organization-as-tenant; multi-tenant-ready, single-tenant-first
- **Статус:** accepted · **Дата:** 2026-07-02 (pre-existing)
- **Контекст:** нужно расшириться в Фазу 2 без переписывания.
- **Решение:** иерархия `organization → subdivision → membership(user↔subdivision+role)`; `users` глобальна. `organization_id` денормализован на каждую операционную строку (`ondelete=RESTRICT`); tenant-репозитории требуют scope в конструкторе; scope только из подписанного JWT. Фаза 1 — один дефолтный tenant.
- **Следствия:** жёсткая изоляция с первого дня, минимальная реализация.
- **Связи:** REQ R5.

### ADR-009 · Одна реализация на provider-seam до реального триггера
- **Статус:** accepted (с активным исключением) · **Дата:** 2026-07-02 (pre-existing)
- **Контекст:** мёртвый код «на будущее».
- **Решение:** интерфейсы (OCR/ERP за Protocol+registry) существуют, но активна одна реализация: OCR=claude, ERP=esupl. Альтернативы не пишутся без триггера (рост объёма, гео/стоимостной риск).
- **Следствия:** LLM-транспорт сейчас — модульные функции, не Protocol.
- **Исключение/долг:** зарегистрирован второй OCR/AI-вендор `gemini` — нарушает решение. Требует резолюции: claude-only (рекоменд.) или достроить seam. → **BACKLOG DEC-01**.
- **Связи:** REQ R7, BACKLOG DEC-01.

### ADR-010 · Шифрование секретов at-rest (Fernet-конверт, версионируемый KEK)
- **Статус:** accepted · **Дата:** 2026-07-02 (pre-existing)
- **Контекст:** секреты в БД открытым текстом = утечка при дампе.
- **Решение:** Fernet, формат `enc:v2:<key_id>:<token>`; KEK(и) только в `.env` (`SECRETS_ENC_KEY` + `_KEY_ID` + `_KEYS_OLD`); ротация промоушеном нового KEK; `validate_keyring()` на старте.
- **Следствия:** дамп БД без `.env` бесполезен; ротация без потери старых шифртекстов.
- **Связи:** REQ R2, ADR-005.

### ADR-011 · Секреты читаются без кэша
- **Статус:** accepted · **Дата:** 2026-07-02 (pre-existing)
- **Контекст:** кэш секрета делает ротацию неотличимой от бага.
- **Решение:** `get_active_credential` читает+дешифрует на каждый вызов (без кэша) — ротация в супер-админке мгновенна. Один активный секрет на (scope, provider, org, subdivision) — partial-unique индекс.
- **Следствия:** лишний запрос к БД принят как цена детерминизма.
- **Связи:** REQ R6.

### ADR-012 · Фронт не хранит секретов
- **Статус:** accepted · **Дата:** 2026-07-02 (pre-existing)
- **Контекст:** XSS-кража токенов, ключи в браузере.
- **Решение:** auth только HttpOnly-куками; все секреты (LLM-ключи, ERP-токены) — на бэке, шифр., за write-only/read-masked API. `VITE_*` — только несекретные endpoint/provider-переключатели. Живые пути провайдеров — только `backend`/`mock`.
- **Следствия:** браузер-директ LLM/ERP удаляется как мёртвый код (BACKLOG A2).
- **Связи:** REQ R9, BACKLOG A2.

### ADR-013 · Photo-first поток накладной
- **Статус:** accepted · **Дата:** 2026-06-29 (recognition-feature)
- **Контекст:** поставщик неизвестен до распознавания.
- **Решение:** поставщик не выбирается заранее — OCR читает имя с фото, система матчит его на список POS (`bestNameMatch`, ≥0.6 авто, иначе uncertain+dropdown). Маппинги supplier→SKU+packing запоминаются и авто-применяются на следующей накладной того же поставщика (learning loop).
- **Следствия:** supplier коммитится в том же dispatch, что читает его id.
- **Связи:** 01_ARCHITECTURE, 03_ROADMAP (F1/F2).

### ADR-014 · OCR через vision-LLM, а не классический OCR-движок
- **Статус:** accepted · **Дата:** 2026-06-22 (scan-preprocessing-plan)
- **Контекст:** выбор конвейера предобработки изображения.
- **Решение:** распознавание — vision-LLM (Claude Vision; Gemini рассматривался). Поэтому **без** бинаризации/трешолда/жёсткого грейскейла (вредят VLM). Предобработка: тихая нормализация (EXIF, downscale ≤1568px, один JPEG-проход, fail-open), crop — «auto-first, manual-optional», не обязателен.
- **Следствия:** нет UI ручной разметки полей; ставка на tight-crop + resize + confidence-gate.
- **Связи:** 01_ARCHITECTURE, work/scan-preprocessing-plan.md.

### ADR-015 · Регламент документации (порядок доверия, архив инертен)
- **Статус:** accepted · **Дата:** 2026-07-02
- **Контекст:** осадочные слои пивотов, конфликтующая документация, агент читает устаревшее.
- **Решение:** документация ведётся как локальный Confluence-lite: 6 канонических живых документов + `work/` + `archive/`, front-matter с версией/датой/статусом, явный порядок доверия (код > нормативное > описательное > продукт > work > archive), архив агент не читает.
- **Следствия:** этот файл и весь `mvp.docs/` подчиняются README-регламенту.
- **Связи:** README.
