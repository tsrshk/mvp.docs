# Local OS — Функциональные стадии разработки

> **Версия:** 0.1 · **Дата:** Май 2026 · **Спутник к:** Local_OS_Specification_v04.md
>
> Декомпозиция на **функционально достаточные** стадии. Каждая стадия — самостоятельный работающий инструмент, который жена использует с первого дня после её завершения.

---

## Принцип декомпозиции

В отличие от классического "сначала вся архитектура, потом фичи", здесь каждая стадия:

1. **Закрывает одну конкретную боль полностью** — не «частично всё», а «полностью одно»
2. **Запускается в продакшен сразу** — после её завершения жена реально начинает пользоваться
3. **Имеет измеримую ценность** — мы можем сказать «эта стадия сэкономила X часов» или «дала Y инсайтов»
4. **Технически автономна** — не требует переписывания предыдущих стадий
5. **Длится 2–4 недели** — быстрый цикл фидбека

Это значит, что **Local OS уже на стадии F1 — это работающий продукт**, который приносит пользу. Не «MVP в конце 6 месяцев», а **«первый полезный инструмент через 3-4 недели»**.

---

## 1. Сводная таблица функциональных стадий

| Стадия | Название | Длительность | Боль которую закрывает | Ценность с первого дня |
|---|---|---|---|---|
| **F0** | Разговор с женой | 1 нед | Не знаем точные приоритеты | Список болей по важности |
| **F1** | OCR накладных | 3 нед | Ручной ввод накладных | -2 часа в неделю |
| **F2** | Маппинг и Esupl-приход | 3 нед | Ручное сопоставление позиций с SKU | -1.5 часа в неделю |
| **F3** | Справочник поставщиков | 2 нед | Цены поставщиков в голове | Структурированная база, история |
| **F4** | Сравнение поставщиков | 2 нед | «У кого лучше заказать?» | 1-3 рекомендации/мес с экономией |
| **F5** | Базовая аналитика продаж | 3 нед | «Что продавалось хорошо?» | Еженедельный дайджест |
| **F6** | Локальный контекст | 2 нед | Прогнозы без учёта реальности | Учёт погоды и событий |
| **F7** | Анализ конкурентов: меню | 3 нед | «А сколько у них?» | База цен 5-10 ближайших |
| **F8** | Анализ конкурентов: отзывы | 2 нед | Не знаем что говорят о нас и о них | Тренды отзывов еженедельно |
| **F9** | Cross-recipe рекомендации | 2 нед | «Что ещё можно добавить в меню?» | 3+ идеи новых блюд в месяц |
| **F10** | Стратегические инсайты | 3 нед | Принятие решений на ощупь | Еженедельные 3 главных вопроса |

**Что важно:** между F1 и F10 — каждая стадия добавляет новую функциональность, а **предыдущие продолжают работать**. После F1 у жены уже есть инструмент. После F10 — полноценный AI-помощник.

## 2. Детальное описание стадий

### F0 · Разговор с женой

**Длительность:** 1 неделя

**Боль:** мы не знаем какие из её болей самые острые и в каком порядке решать.

**Что делаем:**
- Серьёзный разговор без детей (2 часа)
- Подготовленные вопросы: текущая рутина, что бесит, что хотелось бы автоматизировать, что бы изменило её рабочую неделю
- Запись и систематизация

**Результат:**
- Список 5–10 болей с приоритетами **самой жены** (не нашими предположениями)
- Согласие на проект
- Договорённость о ролях

**Ценность с первого дня:** нет (это исследование). Но без F0 все последующие стадии могут пойти не туда.

**Технологии:** блокнот и чай.

---

### F1 · OCR накладных

**Длительность:** 3 недели

**Боль:** жена тратит ~2 часа в неделю на ручной перенос накладных в Esupl. Это самая частая и времязатратная рутина.

**Что делает продукт:**
- Жена отправляет фото накладной в Telegram-бот
- Бот через Claude Vision API извлекает структуру: поставщик, дата, позиции (название, количество, цена), сумма
- Жена видит результат в боте, может поправить ошибки
- Накладная сохраняется в нашей БД с историей

**Что НЕ делает на этой стадии:**
- Не пишет в Esupl (пока)
- Не мапит на SKU автоматически (пока ручной маппинг через бот)
- Не сравнивает с прошлыми ценами

**Технологии:**
- Telegram Bot (node-telegram-bot-api или aiogram)
- Claude API + Vision
- PostgreSQL для истории накладных
- Hetzner/DO для хостинга

**Ценность с первого дня:**
- Жена больше не печатает накладные руками
- Все накладные в одном месте, можно искать
- Базовая основа для всех следующих стадий

**Метрика успеха:**
- Точность OCR ≥95% позиций
- Жена обрабатывает накладные в боте, а не вручную
- Экономия: ~2 часа в неделю

**Gate-критерий:** жена 4 недели подряд использует бот для накладных вместо ручного ввода.

---

### F2 · Маппинг позиций и приход в Esupl

**Длительность:** 3 недели

**Боль:** позиция в накладной называется по-разному у каждого поставщика. Жена тратит ~1.5 часа в неделю на сопоставление и ввод прихода в Esupl.

**Что делает продукт:**
- При импорте новой накладной для каждой позиции:
  - Проверяет справочник «{поставщик} + {название} → SKU». Если есть — мап автоматически
  - Если нет — AI предлагает top-3 кандидата из SKU Esupl
  - Жена выбирает или вводит новый SKU
  - Связь запоминается на будущее
- После полного маппинга накладной → жена нажимает «Записать в Esupl» → приход создаётся через Esupl API

**Что добавляется к F1:**
- Справочник маппингов (растёт со временем)
- Семантический поиск по SKU (embeddings)
- Интеграция с Esupl API

**Технологии:**
- Esupl API (исследовать в первую неделю)
- pgvector в PostgreSQL для embeddings
- Claude embeddings или OpenAI

**Ценность с первого дня:**
- Жена больше не сопоставляет позиции вручную
- Приход автоматически попадает в Esupl
- После 1-2 месяцев работы 90% позиций мапятся автоматом

**Метрика успеха:**
- 70% позиций мапятся автоматически после первого месяца
- 90% — после второго
- Экономия: дополнительные ~1.5 часа в неделю

**Gate-критерий:** жена больше не открывает Esupl для ручного ввода накладных.

---

### F3 · Справочник поставщиков и прайс-листов

**Длительность:** 2 недели

**Боль:** у каждого поставщика свои цены, скидки, условия. Жена держит это в голове, в WhatsApp-переписках и в записях. Сравнить невозможно.

**Что делает продукт:**
- UI/бот для ведения справочника поставщиков:
  - Контакты, условия доставки, минимальная партия
  - Прайс-лист (вводится вручную или импортируется из накладных)
  - История изменения цен по времени
- При каждой новой накладной — автоматически:
  - Обновляет цены поставщика в справочнике
  - Сравнивает с прошлой ценой → уведомление если изменилась
- Жена может через бот спросить: «какая цена на молоко у Беларусьмолоко?» — получить ответ

**Что НЕ делает на этой стадии:**
- Пока не сравнивает поставщиков между собой (это F4)
- Пока не парсит прайс-листы из Excel/PDF (это Phase 2)

**Технологии:**
- Расширение БД (таблицы suppliers, supplier_prices, price_history)
- Интерактивный UI в Telegram (inline keyboards)

**Ценность с первого дня:**
- Все цены поставщиков в одном месте
- История изменений видна
- Если поставщик поднял цены — жена узнаёт сразу

**Метрика успеха:**
- 100% поставщиков жены в справочнике
- Все цены обновляются автоматически при новой накладной

**Gate-критерий:** жена смотрит цены в боте, а не в WhatsApp/блокноте.

---

### F4 · Сравнение поставщиков

**Длительность:** 2 недели

**Боль:** жена не знает «у кого лучше заказать в этот раз». Решения на основе привычки, не цифр.

**Что делает продукт:**
- Команда в боте «сравнить молоко» (или любой SKU)
- Бот показывает:
  - Текущий поставщик и цена
  - Альтернативные поставщики из справочника (F3) с их ценами
  - Учёт минимальной партии и условий доставки
  - Расчёт экономии за месяц/квартал
- Проактивные уведомления:
  - «Поставщик A поднял цену на 8% — у B теперь дешевле на 50 ₽/кг»
  - «Зерно эфиопии: у тебя средняя цена 1350. Год назад была 1200 — рост 12.5%»

**Что НЕ делает:**
- Не оформляет заказ автоматически
- Не связывается с поставщиками сам
- Не делает межклиентскую агрегацию цен (это Phase 2+)

**Технологии:**
- Бизнес-логика поверх F3
- Несколько LLM-запросов для генерации естественного объяснения

**Ценность с первого дня:**
- Жена принимает решения о закупках на основе сравнения, не интуиции
- Видит динамику цен своих поставщиков

**Метрика успеха:**
- 1–3 валидных рекомендации в месяц с экономическим эффектом
- За квартал — минимум 1 случай реальной смены/перераспределения поставщика

**Gate-критерий:** жена использует команду сравнения перед каждой крупной закупкой.

---

### F5 · Базовая аналитика продаж

**Длительность:** 3 недели

**Боль:** жена не имеет регулярного обзора что хорошо/плохо продавалось. Тратит ~3 часа в неделю на анализ в Esupl.

**Что делает продукт:**
- Автоматический sync продаж из Esupl каждые несколько часов
- Раз в неделю (например, понедельник 9:00) жена получает в Telegram:
  - Выручка за неделю vs прошлая vs аналогичный период
  - Топ-5 позиций по выручке и марже
  - Аномалии: что упало, что выросло
  - Маржа по группам товаров
- Команды в боте:
  - «Как дела с матчей?» — динамика конкретной позиции
  - «Что было в субботу?» — детализация дня

**Технологии:**
- Расширение БД (таблицы sales, daily_aggregates)
- Базовая статистика на Python/Java (без AI)
- LLM только для генерации текста отчёта

**Ценность с первого дня:**
- Жена получает картину недели без открытия Esupl
- Аномалии видны сразу, не через месяц
- Экономия: ~3 часа в неделю

**Метрика успеха:**
- Жена не открывает Esupl для weekly review
- Реагирует на аномалии в течение недели, не месяца

**Gate-критерий:** жена ждёт понедельничный дайджест.

---

### F6 · Локальный контекст

**Длительность:** 2 недели

**Боль:** прогнозы и интерпретации без учёта реальности (погода, события). Когда упала выручка в среду — это локальная проблема или общий тренд?

**Что делает продукт:**
- Подключение OpenWeather API (прогноз 14 дней)
- Парсер афиш Минска (концерты, фестивали в радиусе 1–3 км)
- В еженедельном дайджесте (из F5) добавляется:
  - «В четверг выручка упала на 30% — был ливень с 14:00 до 19:00»
  - «На следующей неделе фестиваль в парке Челюскинцев — ждите всплеск в субботу-воскресенье»
- Корректировка прогноза потребности с учётом погоды

**Технологии:**
- OpenWeather API ($0 для базового тарифа)
- Парсер афиш (relax.by, kp.by, telegram-каналы)
- Связывание событий с продажами

**Ценность с первого дня:**
- Жена понимает «почему» в дайджестах, не только «что»
- Может заранее подготовиться к событиям

**Метрика успеха:**
- Минимум 2 предупреждения о событиях в месяц
- Объяснения аномалий продаж стали в 90%+ случаев

**Gate-критерий:** жена использует прогноз для планирования закупок на неделю.

---

### F7 · Анализ конкурентов: меню и цены

**Длительность:** 3 недели

**Боль:** жена не знает «а как у соседей?». Цена капучино — высокая или низкая для района?

**Что делает продукт:**
- Список 5–10 ближайших кофеен района (из Google Maps API)
- Жена/founder делает обход с фото меню → OCR (на готовой инфраструктуре из F1) → база цен конкурентов
- Бот по запросу:
  - «Цены капучино в районе» — таблица: у нас 5.50, средняя 5.30, диапазон 4.80–6.20
  - «Что есть у Бернини чего нет у нас?» — сравнение ассортимента
- Раз в квартал — напоминание «пора обновить базу, конкуренты могут поменять меню»

**Что НЕ делает:**
- Не парсит сайты конкурентов автоматически (юр. серая зона)
- Не делает полный конкурентный анализ (это уже консалтинговая работа)

**Технологии:**
- Google Places API ($0 в рамках Free tier)
- Переиспользование OCR из F1
- Визуализация через простые таблицы в боте

**Ценность с первого дня:**
- Жена знает где её цены относительно района
- Может обоснованно поднимать/опускать цены
- Видит возможности (то что есть у соседей, чего нет у нас)

**Метрика успеха:**
- База покрывает 80% топ-позиций по 5+ конкурентам
- 1–3 обоснованных изменений цен/меню за квартал

**Gate-критерий:** жена обращается к данным конкурентов при пересмотре меню или цен.

---

### F8 · Анализ конкурентов: отзывы

**Длительность:** 2 недели

**Боль:** жена не отслеживает регулярно что говорят о нас в отзывах, и не знает что хвалят/ругают у конкурентов.

**Что делает продукт:**
- Подключение Google My Business API (OAuth от имени кофейни)
- Парсинг публичных отзывов конкурентов с Google Maps (раз в неделю)
- AI-анализ:
  - Sentiment (положительные/нейтральные/отрицательные)
  - Тренды жалоб («стали чаще жаловаться на громкую музыку»)
  - Упоминания конкретных позиций («матчу хвалят 8 раз за месяц у Бернини»)
- В еженедельном дайджесте — секция «Отзывы»

**Технологии:**
- Google My Business API
- LLM для sentiment и categorization
- Хранение истории отзывов

**Ценность с первого дня:**
- Жена реагирует на жалобы быстро
- Видит сильные стороны конкурентов
- Может перенять успешные позиции

**Метрика успеха:**
- 100% отзывов нашей кофейни обработаны в течение 48 часов
- Минимум 1 практическое решение в месяц на основе анализа

**Gate-критерий:** жена использует анализ отзывов для решений по сервису/меню.

---

### F9 · Cross-recipe рекомендации

**Длительность:** 2 недели

**Боль:** «что ещё можно добавить в меню без больших затрат?». Сейчас — гипотезы и пробы.

**Что делает продукт:**
- Анализ существующих SKU в Esupl
- AI-генерация: «из ваших текущих ингредиентов можно сделать ещё 7 позиций»
- Расчёт ROI каждой:
  - Какие SKU уже есть (затраты $0)
  - Какие нужно докупить (стоимость с учётом F4)
  - Прогноз продаж на основе бенчмарков конкурентов из F7
- Идеи на сезон: «лето — наоми, лимонады, smoothies. У вас есть базовая база»

**Что НЕ делает:**
- Не создаёт рецепт автоматически (жена решает по дегустации)
- Не учитывает тренды соцсетей (это Phase 2+)

**Технологии:**
- LLM с хорошим промптом (база знаний о кофейном бизнесе)
- Использование данных из F3, F4, F7

**Ценность с первого дня:**
- Регулярный поток идей для меню
- Каждая идея с экономическим обоснованием

**Метрика успеха:**
- 3+ идеи новых блюд в месяц
- Минимум 1 реально добавленная позиция в квартал

**Gate-критерий:** жена использует рекомендации при сезонном пересмотре меню.

---

### F10 · Стратегические инсайты

**Длительность:** 3 недели

**Боль:** жена принимает стратегические решения «на ощупь». Когда поднять цены? Когда сменить поставщика? Когда добавить позицию?

**Что делает продукт:**
- Объединение всех предыдущих стадий в **еженедельный стратегический разговор**:
  - 3 главных вопроса для обсуждения недели (не отчёт, а вопросы)
  - Контекст из F5 (продажи), F4 (поставщики), F7-F8 (конкуренты), F6 (внешняя среда)
  - AI задаёт уточняющие вопросы, жена отвечает в свободной форме
- Поддержка диалога:
  - «Что думаешь о повышении цены капучино на 10%?»
  - AI: «Текущая 5.50, средняя по району 5.30, у Бернини 6.00. У вас маржа упала на 4% за квартал. Если поднять до 6.00 при текущем объёме — это +$X в месяц. Риск — потеря 5-10% объёма. Хочешь смоделировать?»
  - Жена: «А что если только на специальные сорта?»

**Технологии:**
- LLM с длинным контекстом (Claude Sonnet 4.6 — 200K)
- Промпт-инжиниринг для разговорного режима
- Сохранение истории разговоров для контекста

**Ценность с первого дня:**
- Жена получает «думающего партнёра» раз в неделю
- Решения принимаются на основе данных, не интуиции

**Метрика успеха:**
- WIFE-GATE: жена сама говорит «без этого хуже»
- Минимум 1 стратегическое решение в месяц на основе разговора с ботом

**Gate-критерий (WIFE-GATE):** жена считает что продукт стал необходимой частью её рабочей недели.

## 3. Зависимости между стадиями

Не все стадии последовательны. Некоторые можно делать параллельно (если есть ресурс):

```
F0 (разговор) → определяет приоритеты для всех остальных
                ↓
F1 (OCR) ──────────────┬─→ F2 (маппинг + Esupl)
                       │       ↓
                       └─→ F3 (поставщики) ─→ F4 (сравнение)
                                                  ↓
F5 (продажи) ←─── (Esupl уже подключён в F2)
   ↓
F6 (контекст) добавляется к F5
   ↓
F7 (конкуренты-меню) ──┬─→ F9 (cross-recipe) — нужны F3+F7
                       └─→ F10 (инсайты) — объединяет всё
F8 (конкуренты-отзывы) ─→ F10
```

**Критический путь:** F0 → F1 → F2. После F2 — гибкость в порядке.

**Параллелизуемые ветки:**
- F3-F4 (поставщики и сравнение) можно делать параллельно с F5-F6 (аналитика и контекст)
- F7-F8 (конкуренты) можно делать после F2, независимо от F3-F4

## 4. Приоритезация стадий

Стадии можно перепрыгивать в порядке, если приоритеты жены показывают другую важность.

### Приоритеты по объёму экономии времени (часов/неделю)

1. **F1 + F2 (OCR + Маппинг + Esupl):** ~3.5 часа/нед — главная экономия
2. **F5 (аналитика продаж):** ~3 часа/нед
3. **F4 (сравнение поставщиков):** ~1 час/нед + экономия $$
4. **F10 (стратегические инсайты):** ~2 часа/нед + лучшие решения

### Приоритеты по «вау-эффекту» (что жена сразу почувствует)

1. **F1 (OCR):** жена впервые видит «магию» — фото в чёрную накладную, JSON на выходе
2. **F5 (дайджест):** первое понедельничное письмо с пониманием недели
3. **F10 (диалог):** AI как партнёр, не инструмент

### Рекомендуемый порядок (если делать всё последовательно)

```
F0 → F1 → F2 → F3 → F4 → F5 → F6 → F7 → F8 → F9 → F10
```

**Альтернативный порядок (если жена скажет что аналитика важнее закупок):**

```
F0 → F1 → F5 → F6 → F2 → F3 → F4 → F7 → F8 → F9 → F10
```

Финальный порядок определяется после F0.

## 5. Таймлайн (рекомендуемый порядок)

При 10–15 ч/нед разработки:

| Неделя | Стадия | Что происходит |
|---|---|---|
| 0 | F0 | Разговор с женой |
| 1–3 | F1 | OCR накладных |
| 4–6 | F2 | Маппинг + Esupl-приход |
| 7–8 | F3 | Справочник поставщиков |
| 9–10 | F4 | Сравнение поставщиков |
| 11–13 | F5 | Аналитика продаж |
| 14–15 | F6 | Локальный контекст |
| 16–18 | F7 | Конкуренты: меню |
| 19–20 | F8 | Конкуренты: отзывы |
| 21–22 | F9 | Cross-recipe |
| 23–25 | F10 | Стратегические инсайты |
| **26** | **WIFE-GATE check** | **Решение Phase 1 → Phase 2** |

**Итого:** ~26 недель = ~6 месяцев до полноценного продукта для жены.

**После каждой стадии — пауза на 3–7 дней** для использования женой и сбора фидбека перед следующей. Это не «потерянное время», это критическая часть процесса.

## 6. Что важно понимать про этот подход

### 6.1 Каждая стадия — это релиз

Не «допишем потом», не «сначала всё, потом запустим». Каждая F-стадия идёт в продакшен на нашу собственную кофейню сразу после завершения.

### 6.2 Можно остановиться на любой стадии

Если на F4 (сравнение поставщиков) жена скажет «всё, мне этого достаточно, остальное не нужно» — мы останавливаемся. Это не провал, это нормальный исход. Дальше — Phase 2 на основе работающего продукта.

### 6.3 Порядок не сакрален

Если в F0 окажется, что главная боль жены — аналитика конкурентов, а не закупки — переставляем стадии. Документ — гайдлайн, не закон.

### 6.4 Стадии маленькие специально

2–3 недели на стадию — это против соблазна делать «всё сразу». Маленькая стадия = быстрая обратная связь = меньше риска потратить месяц на ненужное.

### 6.5 Это работающий продукт с 4-й недели

После F1 у нас уже есть **работающий инструмент**, который снимает реальную боль. Не «прототип», не «POC» — продукт в продакшене на одной кофейне. С F2 — два инструмента. С F10 — десять.

## 7. Связь с Phase 2 (рынок)

Когда WIFE-GATE пройден (после F10) — мы переходим в Phase 2.

Что нужно сделать дополнительно для рынка:
- **Multi-tenant архитектура** — одна система для N клиентов (S3 в основной спецификации)
- **Onboarding flow** — без участия founder (S3)
- **Биллинг** — Stripe или белорусский эквивалент (S4)
- **Web-кабинет** — в дополнение к Telegram (S4)
- **iiko и Poster коннекторы** — расширение TAM (S4)

Но **продукт уже работает**. Мы не строим Phase 2 с нуля — мы упаковываем для рынка то, что работает у нас.
