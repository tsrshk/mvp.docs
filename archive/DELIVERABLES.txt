ENDPOINT SPECIFICATION
======================

Route: POST /api/v1/ingredients/mappings

Request Body:
{
  "source_key": string (required, non-empty, will be normalized)
  "pos_ingredient_id": string (required, non-empty)
  "method": "manual" | "fuzzy" | "ai" (required enum)
  "confidence": float (optional, range 0.0..1.0)
  "confirmed_by": UUID (optional, user ID confirming the mapping)
}

Response (200 OK):
{
  "id": UUID,
  "source_key": string (normalized: lowercase, trimmed, spaces collapsed),
  "pos_ingredient_id": string,
  "method": string,
  "confidence": float | null,
  "confirmed_by": UUID | null,
  "created_at": ISO 8601 datetime,
  "updated_at": ISO 8601 datetime
}

Error Response (400 Bad Request):
{
  "detail": "Error message describing validation or database failure"
}

Authentication:
- Required: Valid JWT access token in HttpOnly cookie
- Authorization: Requires active subdivision context in TenantContext
- Returns 401 if not authenticated
- Returns 403 if no active subdivision context


IMPLEMENTATION DETAILS
======================

Database Table: sku_mapping (already exists, migration 0004)
- Unique constraint on (scope_type, scope_id, source_key)
- Scope type: "subdivision" (not "org")
- Scope ID: subdivision_id from TenantContext
- Automatic timestamps: created_at, updated_at (managed by TimestampMixin)

Source Key Normalization:
- Strip leading/trailing whitespace
- Convert to lowercase
- Collapse multiple consecutive spaces to single space
- Example: "  МОЛОКО   3.2%  " → "молоко 3.2%"

Upsert Logic:
1. Normalize source_key
2. Query sku_mapping table: WHERE scope_type='subdivision' AND scope_id=? AND source_key=?
3. If found: UPDATE pos_ingredient_id, method, confidence, confirmed_by
4. If not found: INSERT new record with UUID primary key
5. Both operations in same transaction (idempotent)
6. Return response with all fields and timestamps

Multi-Tenancy:
- Each mapping is scoped to a specific subdivision
- Same source_key in different subdivisions creates separate records
- Prevents cross-tenant data leakage (verified by tests)
