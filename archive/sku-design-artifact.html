1Factory Pattern: SkuHandlerFactory

Core Idea: Different entity types (suppliers, ingredients, products) have different selection contexts and grouping strategies. The factory abstracts these differences into a single interface, so the component never cares about the specific entity type.

The Problem It Solves

  • Today: SkuSelect and SkuSearchSelect only handle ingredients (SKUs).
  • Tomorrow: We'll need supplier selection (multi-step flow) and product selection (supplier catalogue).
  • The risk: Duplicating search/grouping logic for each entity type.
  • The solution: One interface (SkuHandler), multiple implementations. Same component works for all.

Architecture Diagram

SkuHandlerFactory.create(context) ↓ (Detects entityType) ↓ ┌───┬───────────┬──────────┐ ↓ ↓ ↓ ↓ 'ingredient' → IngredientSkuHandler 'supplier' → SupplierSkuHandler 'product' → ProductSkuHandler ↓ SkuHandler interface ├─ fetchSkus() ├─ fetchSupplierProductIds() ├─ searchSkus() └─ groupSkus() ↓ SkuDropdown (same component, all entity types!)

interface SkuHandler {

IngredientSkuHandler

  • Fetches from POS provider (backend)
  • Searches by name + unit
  • Groups by supplier (if supplier selected)
  • Current primary use case

SupplierSkuHandler

  • Fetches from POS provider
  • Searches by name + tax ID
  • No sub-grouping (single list)
  • Future: multi-step supplier picker

ProductSkuHandler

  • Fetches supplier-specific products
  • Requires supplierId in context
  • Org/subdivision aware
  • Future: mapping supplier products → SKUs

2Dropdown Component: SkuDropdown

Design principle: A single, reusable component that works with any handler. Manages search state, keyboard navigation, focus, and rendering grouped results.

Props Interface

interface SkuDropdownProps {
  value: string | null
  handler: SkuHandler           // Created by factory
  supplierProductIds: Set<string>
  invalid?: boolean
  placeholder?: string
  onPick: (skuId: string | null) => void
  onCreate?: () => void
  className?: string
}

Component Lifecycle

Mount → handler.fetchSkus() → setState(allSkus)
User types → handler.searchSkus() → filtered
Supplier changes → handler.groupSkus() → grouped results
User picks → onPick(skuId) → close dropdown

Key Features

Feature Implementation
Live Search Client-side filtering (no API calls on keystroke)
Keyboard Nav Arrow up/down, Enter to pick, Esc to close
Click Outside Closes dropdown automatically
Grouped Rendering Optgroups by handler result groups
Accessibility Proper ARIA labels, focus management, WCAG 2.1

3Grouping Logic & Supplier Detection

The core grouping strategy: "Supplier items first, then all others, alphabetical." But this only happens if a supplier is selected. Without a supplier, it's just "all items, alphabetical."

Supplier Detection Decision Tree

Where does the supplier come from? 1. Line's own supplier (least common) ↓ 2. Invoice header supplier (most common) ↓ 3. Redux explicit selectedSupplierId ↓ 4. None selected (show all, no grouping) Priority: Line > Header > Redux > None

Grouping Algorithm (Pseudocode)

function groupSkusIntel(
  skus: Sku[],
  supplierId: string | null,
  supplierProductIds: Set<string>,
  locale: string = 'en'
): GroupedSkus {
  const collator = new Intl.Collator(locale)

  if (supplierId && supplierProductIds.size > 0) {
    // Split into two groups
    const supplierSkus = skus.filter(sku =>
      sku.productId && supplierProductIds.has(sku.productId)
    )
    const restSkus = skus.filter(sku =>
      !sku.productId || !supplierProductIds.has(sku.productId)
    )

    // Sort each group
    supplierSkus.sort((a, b) => collator.compare(a.name, b.name))
    restSkus.sort((a, b) => collator.compare(a.name, b.name))

    // Return groups
    return {
      groups: [
        {
          id: 'supplier',
          label: 'Ингредиенты этого поставщика',
          skus: supplierSkus,
          isDefault: true
        },
        {
          id: 'rest',
          label: 'Остальные ингредиенты',
          skus: restSkus
        }
      ],
      totalCount: skus.length
    }
  } else {
    // Single group: all SKUs, alphabetical
    const sorted = [...skus].sort((a, b) =>
      collator.compare(a.name, b.name)
    )
    return {
      groups: [{
        id: 'all',
        label: 'Все ингредиенты',
        skus: sorted,
        isDefault: true
      }],
      totalCount: skus.length
    }
  }
}

Key Data Structure: Sku.productId

What it is: An ID that links a SKU (ingredient) to a supplier's product catalogue entry.

Why it matters: It's the join key. When a supplier is selected, we ask "which SKUs have productIds in this supplier's delivery history?" Those appear first.

Scenario Behavior
Supplier selected, SKU.productId in supplier's history → Appears in "supplier group" (first)
Supplier selected, SKU.productId NOT in history → Appears in "rest group" (second)
Supplier selected, SKU.productId is null → Appears in "rest group"
No supplier selected → All SKUs in single group, no sub-grouping

4Integration Points

Integration 1: Invoice Form → SKU Selection

Line Row Component (Desktop View)
export function LineRow({
  line,
  onSkuPick,
  supplierProductIds,
  // ...
}: LineRowProps) {
  const supplier = useAppSelector(s => s.invoiceSession.supplier)

  // Create handler context from invoice state
  const handlerContext: SkuSelectionContext = {
    entityType: 'ingredient',
    orgId: auth.orgId,
    subdivisionId: auth.subdivisionId,
    supplierId: supplier?.id
  }

  const handler = useMemo(
    () => SkuHandlerFactory.create(handlerContext),
    [handlerContext]
  )

  return (
    <SkuDropdown
      value={line.skuId || null}
      handler={handler}
      supplierProductIds={supplierProductIds}
      onPick={(skuId) => onSkuPick(line.id, skuId)}
    />
  )
}

Integration 2: Supplier Selection → SKU Grouping

Invoice Header Component
const onSupplierSelect = (supplier: Supplier | null) => {
  // Update invoice supplier
  dispatch(supplierSelected(supplier))

  // Load product IDs for grouping
  if (supplier) {
    (async () => {
      const handler = SkuHandlerFactory.create({
        entityType: 'ingredient',
        orgId: auth.orgId,
        subdivisionId: auth.subdivisionId,
        supplierId: supplier.id
      })
      const productIds = await handler.fetchSupplierProductIds(supplier.id)
      // Dispatch to Redux so all pickers re-group
      dispatch(supplierProductIdsLoaded(productIds))
    })()
  }
}

Integration 3: Org/Subdivision Scope

Every handler is scoped to org + subdivision. This ensures:
• SKUs from handler.fetchSkus() are only for this org/subdiv
• Backend auth validates scope via session cookie
• Supplier product IDs are also scoped (ERP owns the data)
• Future multi-tenant scenarios are built-in

Integration 4: Async Search & Caching

Caching strategy:

  • SKUs: Loaded once on component mount (via handler.fetchSkus())
  • Search: Happens client-side (no API calls on every keystroke)
  • Supplier product IDs: Cached per supplier (loaded when supplier changes)

Why? Instant feedback, no jank, no server load.

6Real-World Example: Mobile Card SKU Selection

LineCard Component (Current Use Case)
export function LineCard({
  line,
  onSkuPick,
  supplierProductIds,
  // ...
}: LineCardProps) {
  const supplier = useAppSelector(s => s.invoiceSession.supplier)
  const allSkus = useAppSelector(s => s.skus.items)

  // 1. Create handler context (depends on supplier & auth)
  const handlerContext: SkuSelectionContext = useMemo(
    () => ({
      entityType: 'ingredient',
      orgId: useAuth().orgId,
      subdivisionId: useAuth().subdivisionId,
      supplierId: supplier?.id
    }),
    [supplier?.id]
  )

  // 2. Create handler (factory does the work)
  const handler = useMemo(
    () => SkuHandlerFactory.create(handlerContext),
    [handlerContext]
  )

  return (
    <div className="p-4 border rounded-lg">
      <div className="mb-3">
        <label className="text-xs font-semibold">Товар</label>
        {/* 3. Pass handler to dropdown (same component works for all entity types!) */}
        <SkuDropdown
          value={line.skuId || null}
          handler={handler}
          supplierProductIds={supplierProductIds}
          invalid={line.errors?.includes('skuId')}
          onPick={(skuId) => onSkuPick(line.id, skuId)}
          className="mt-1 w-full"
        />
      </div>
      {/* Other line fields... */}
    </div>
  )
}

What Happens When User Types "ящ"

User types "ящ" → handleChange('ящ')
setState(searchQuery: 'ящ') → useEffect triggered
handler.searchSkus(allSkus, 'ящ') → [ящик, ящик-10л, …]
handler.groupSkus(filtered, supplierProductIds) → GroupedSkus with groups
Render grouped results → User sees filtered items

What Happens When Supplier Changes

User selects supplier → dispatch(supplierSelected(s))
handler context changes → supplierId is now set
handler.fetchSupplierProductIds() → ProductId[] for this supplier
ALL dropdowns re-group → Supplier items appear first

Summary

What You Get

  • ✓ Factory Pattern: Extensible to new entity types (ingredients, suppliers, products, …).
  • ✓ Single Component: SkuDropdown works with any handler.
  • ✓ Intelligent Grouping: Detects supplier selection, prioritizes relevant items.
  • ✓ Fast Search: Client-side, instant feedback, no jank.
  • ✓ Accessibility: Keyboard nav, focus management, ARIA labels.
  • ✓ Org/Subdiv Scope: Built-in multi-tenancy support.
  • ✓ Testable: Handlers are stateless, easy to mock.

Migration Path

  1. Phase 1: Create factory + IngredientSkuHandler (mirrors current logic)
  2. Phase 2: New SkuDropdown component (side-by-side with old)
  3. Phase 3: Supplier & Product handlers (for future features)
  4. Phase 4: Redux integration (automatic re-grouping)
