# Spec-kit workflow — мост между vault и spec-kit

> Этот документ — bridge между двумя системами документации проекта LCOS: **vault `mvp.docs`** (постоянная база знаний, SSOT) и **spec-kit** (одноразовые рабочие артефакты конкретной фичи). Он отвечает на вопросы: где что живёт, какой артефакт чему соответствует, как выглядит жизненный цикл новой фичи и как не породить второй SSOT. Порядок доверия проекта неизменен: **код + `CLAUDE.md` > adr/ > requirements/architecture > overview/product** (см. [[ADR-015]], [[README]]); артефакты spec-kit стоят в этой иерархии **ниже** vault — это рабочие документы уровня `work/`.

## 1. Зачем spec-kit и где он живёт

**Зачем.** Vault `mvp.docs` фиксирует *что построено и почему* (as-built SSOT: эпики, фичи, сущности, требования, ADR). Spec-kit закрывает другую потребность — *дисциплинированный процесс поставки одной фичи*: спецификация → уточнение → план → задачи → реализация, с шаблонами, чек-листами и конституцией как входным контрактом для агента-исполнителя. Раньше эту роль играли рукописные ТЗ в `mvp.docs/work/` (например `TZ__STABILIZATION_2026-07-09__ALIGNED.md`); spec-kit стандартизирует тот же процесс.

**Где живёт** (все пути — от корня воркспейса `d:\_work\mvp`):

| Расположение | Содержимое |
|---|---|
| `.specify/` | Каркас spec-kit: `memory/constitution.md` (конституция проекта), `templates/` (spec/plan/tasks/checklist), `scripts/powershell/` (вспомогательные скрипты), `workflows/` |
| `specs/` | Рабочие артефакты фич: по каталогу `specs/NNN-slug/` на фичу (`spec.md`, `plan.md`, `tasks.md`, чек-листы). Создаётся первым запуском `/speckit-specify` (на момент написания каталога ещё нет) |
| `.claude/skills/speckit-*` | Слэш-команды spec-kit (skills), см. §6 |
| `mvp.docs/` | Vault — постоянный SSOT. Spec-kit его **не заменяет и не дублирует** |

Важно: `.specify/` и `specs/` лежат в **корне воркспейса**, который сам по себе не git-репозиторий (см. §5) и не входит в репозиторий `mvp.docs`. Поэтому на данный момент они **не находятся под контролем версий** — в отличие от кода `mvp.be`/`mvp.fe` и самого vault. Если процессные артефакты нужно версионировать, их следует перенести в репозиторий (например `mvp.docs/work/speckit/`).

## 2. Соответствие артефактов: vault ↔ spec-kit

| Артефакт spec-kit | Аналог в vault `mvp.docs` | Кто SSOT | Комментарий |
|---|---|---|---|
| `.specify/memory/constitution.md` | `adr/` (ADR-001…021, DEC-0011/0013) + `requirements/` ([[global-requirements]], [[fail-closed]], [[sku-identity-resolver]], …) | **vault** | Конституция — *дистиллят* действующих ADR и нормативных требований для агента-исполнителя. Она не порождает новых норм: новая норма = новый ADR или правка requirements, затем синхронизация конституции |
| `specs/NNN-slug/spec.md` | `features/LCOS-F#-…` (например [[LCOS-F8-ocr-recognition]]) | **vault** (после сдачи) | `spec.md` — рабочая спецификация *до и во время* разработки; после сдачи фичи её выжимка (описание, AC по Backend/Frontend, связи) переносится в документ фичи LCOS-F# |
| `specs/NNN-slug/plan.md` | `work/plan/` (`00_IMPLEMENTATION_PLAN.md`, `PHASE_*.md`, `ARCH_SUPPLIER_PRICELISTS.md`) | spec-kit (на время работы) | Технический план реализации — эквивалент фазовой спеки `PHASE_*.md`; долгосрочный roadmap остаётся в [[roadmap]] и `work/plan/00_IMPLEMENTATION_PLAN.md` |
| `specs/NNN-slug/tasks.md` | `work/` (декомпозиции вида `09_PHASE1_TASKS.md`, журналы ТЗ) | spec-kit (на время работы) | Декомпозиция на задачи с зависимостями; после сдачи имеет статус закрытого журнала-evidence, как `TZ__FIX_DISCREPANCY_BUCKET1_2026-07-09.md` |
| чек-листы spec-kit (`/speckit-checklist`) | `## Критерии приёмки (AC)` в документе фичи (`AC-BE-*` / `AC-FE-*`), merge-gate тесты | **vault + код** | Проверяемые инварианты в итоге живут как AC фичи и merge-blocking тесты, не в чек-листе. Генерируются командой `/speckit-checklist` (§6) |

## 3. Жизненный цикл новой фичи

1. **`/speckit-specify`** — сформулировать фичу; создаётся каталог `specs/NNN-slug/` со `spec.md`. Спецификация обязана ссылаться на действующие нормы vault ([[global-requirements]], релевантные ADR), а не пересказывать их.
2. **`/speckit-clarify`** (по необходимости) — снять неоднозначности в `spec.md` до планирования.
3. **`/speckit-plan`** — технический план `plan.md`: архитектура решения, затрагиваемые сущности/сервисы, миграции. План проходит doc-review-гейт *до кода* (практика проекта — как `ARCH_SUPPLIER_PRICELISTS.md` для F72–F75).
4. **`/speckit-tasks`** — декомпозиция в `tasks.md` (задачи, зависимости, порядок). При желании — `/speckit-checklist` для чек-листа качества требований.
5. **`/speckit-implement`** — реализация по задачам в `mvp.be`/`mvp.fe` (ветки — вручную, см. §5). Сквозные инварианты (fail-closed, tenant-изоляция, human-in-the-loop, гейт `ERP_WRITE_ENABLED`) действуют без напоминания; merge-gate тесты обязаны быть зелёными. При необходимости `/speckit-converge` сверяет фактический код с `spec/plan/tasks` и **дописывает недостроенное новыми задачами в `tasks.md`** — цикл `implement ⇄ converge` повторяется, пока не закрыты все задачи.
6. **Обновление SSOT в `mvp.docs`** — обязательный завершающий шаг, целиком ручной и **вне команд spec-kit**; без него фича не считается сданной:
   - создать/обновить документ фичи `features/LCOS-F#-…` (front-matter: `epic`, `status`, `entities`, `requirements`, `adrs`, `sources` с путями к коду; AC по `### Backend` / `### Frontend`);
   - обновить документ родительского эпика (таблица фич) и [[MOC]];
   - обновить/создать документы `entities/` при изменении схемы БД;
   - при новом архитектурном решении — **новый ADR** в `adr/` + строка в [[index]] (журнал append-only: старый ADR не переписывается, только помечается superseded);
   - при изменении норм — синхронизировать `.specify/memory/constitution.md` с новым состоянием adr/requirements.

## 4. Правила против дублирования (анти-второй-SSOT)

- **SSOT остаётся в `mvp.docs`.** Требования, схема сущностей, архитектурные решения записаны там ровно один раз. `specs/` — эфемерные рабочие артефакты конкретной фичи.
- **Ссылки вместо копирования.** В `spec.md`/`plan.md` нормы цитируются ссылкой (путь до документа vault или ID нормы, например «fail-closed per [[fail-closed]] / R8»), а не пересказом. Пересказ неизбежно устаревает и порождает конфликтующий текст.
- **Конституция — производная, не источник.** Обнаружили конфликт конституции с adr/requirements → правится конституция (или создаётся новый ADR, если решение действительно новое). Конституция никогда не «побеждает» vault.
- **AC живут в фиче LCOS-F#.** После сдачи фичи проверяемые критерии переезжают в `## Критерии приёмки (AC)` документа фичи; `specs/NNN-slug/` после этого — закрытый журнал (evidence), его не редактируют задним числом.
- **При конфликте побеждает код** (порядок доверия [[README]]): расхождение spec.md ↔ код решается в пользу кода, документ исправляется.

## 5. Мульти-репо: особенности воркспейса

- Корень воркспейса `d:\_work\mvp` — **не git-репозиторий**; git-репозитории — это `mvp.fe`, `mvp.be`, `mvp.docs` по отдельности.
- Поэтому **встроенные git-операции spec-kit не используются**: `/speckit-specify` не создаёт веток, скрипты `create-new-feature.ps1` работают только с файловой структурой `specs/`.
- Ветки под фичу создаются **вручную** в затронутых репозиториях (`mvp.be` и/или `mvp.fe`), с обычной дисциплиной проекта: коммитит владелец (Ivan), агент не коммитит.
- Один каталог `specs/NNN-slug/` может покрывать изменения в обоих репозиториях — `tasks.md` явно помечает, к какому репо относится задача (BE/FE), в духе разбиения AC на `AC-BE-*`/`AC-FE-*`.

## 6. Слэш-команды spec-kit: когда какую использовать

| Команда | Когда использовать | Результат |
|---|---|---|
| `/speckit-constitution` | Создание/синхронизация `.specify/memory/constitution.md` после изменения adr/ или requirements/ в vault | Обновлённая конституция (производная от vault, см. §4) |
| `/speckit-specify` | Старт новой фичи: есть продуктовое намерение, нужен `spec.md` | Каталог `specs/NNN-slug/` со `spec.md` |
| `/speckit-clarify` | В `spec.md` остались неоднозначности/открытые вопросы; выполняется до `/speckit-plan` | Уточнённый `spec.md` (секция clarifications) |
| `/speckit-plan` | Спецификация стабильна, нужен технический план реализации | `plan.md` (архитектура, сущности, миграции) |
| `/speckit-tasks` | План принят, нужна декомпозиция для исполнителя | `tasks.md` (задачи, зависимости, BE/FE-разметка) |
| `/speckit-checklist` | Нужен чек-лист качества требований для фичи по заданному фокусу (см. строку «чек-листы spec-kit» в §2) | Кастомный чек-лист под указанную область |
| `/speckit-analyze` | Перед реализацией: кросс-проверка согласованности spec ↔ plan ↔ tasks ↔ конституция (аналог adversarial-верификации ТЗ из практики проекта) | Отчёт о расхождениях, без правки артефактов |
| `/speckit-implement` | Артефакты согласованы, можно писать код | Реализация по `tasks.md` в `mvp.be`/`mvp.fe` |
| `/speckit-converge` | После первой реализации: сверка фактического кода со `spec`/`plan`/`tasks` | Недостроенное **дописывается новыми задачами в `tasks.md`** для повторного `/speckit-implement` (цикл implement ⇄ converge). Синхронизацию vault команда **не** делает — это ручной шаг §3.6 |
| `/speckit-taskstoissues` | Опционально: конвертация `tasks.md` в GitHub issues по фиче | Issues по задачам. В текущем мульти-репо процессе не применяется — задачи ведутся в `tasks.md`, issues не заводятся |

Типовая последовательность: `specify → (clarify) → plan → tasks → (checklist) → (analyze) → implement ⇄ converge → обновление mvp.docs (вручную)`. Пара `implement ⇄ converge` — цикл: `converge` находит недостроенное и дописывает задачи, `implement` их закрывает. Перенос SSOT в `mvp.docs` (§3 шаг 6) — отдельный ручной шаг вне команд spec-kit. `/speckit-constitution` запускается вне цикла фичи при изменении норм; `/speckit-taskstoissues` — опционально.
