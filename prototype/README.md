# LCOS — интерактивный прототип

Кликабельный прототип приложения LCOS для четырёх доменов MVP-скоупа:
**накладные**, **поставщики**, **планирование закупок** и **сравнение поставщиков**.
Он служит **визуальной и поведенческой основой для разработки** — договорённостью о том,
как должны выглядеть и работать экраны, прежде чем писать production-код в `mvp.fe`.

> ⚠️ Это прототип, а **не** SSOT. Источник истины по требованиям остаётся в `mvp.docs`
> (эпики, фичи, сущности, ADR). Прототип трассируется к ним — см. §«Покрытие требований».
> При конфликте: **код + `CLAUDE.md` > документы требований > прототип**.

---

## Как открыть

Никакой сборки не нужно — чистый HTML/CSS/JS без зависимостей.

- **Проще всего:** открыть `index.html` двойным кликом в браузере (работает по `file://`).
- **Или** поднять статику:
  ```bash
  cd mvp.docs/prototype
  python -m http.server 8080   # → http://localhost:8080
  ```

Прототип **mobile-first**: на узком экране — верхняя панель + нижний таб-бар; на широком (≥768px) —
боковое меню (как в реальном приложении). Ресайзните окно, чтобы увидеть оба режима.

## Что можно потрогать (сценарии)

| Домен | Маршрут | Ключевые сценарии |
|---|---|---|
| **Накладные** | `#/invoices` | Список со статус-фильтром и метриками; деталь накладной со статус-машиной. |
| ↳ приёмка | `#/invoices/import` | Фото → распознавание → **workbench**: авто-матч поставщика, сопоставление строк со SKU, кнопка «Подобрать товары» (AI), блокировка отправки при несопоставленных строках, «Отправить в Esupl». |
| **Заказы** | `#/orders` | Список заказов по статусам; **«Предложить заказ»** (детерминированный планировщик: остаток ≤ порога → округление до упаковок + причина); **«Из истории»** (prefill); **индикатор минимального заказа** («нужно ещё X BYN»); редактирование количеств; **«Подтвердить» → копируемое сообщение** поставщику с подсказкой канала → «Отправлен». |
| **Поставщики** | `#/suppliers` | Список карточек; карточка с контактами и условиями; редактирование с **динамической формой критериев** (объём, срок, дни доставки, режим оплаты, отсрочка); мягкое скрытие. |
| **Сравнение** | `#/compare` | Лента **ценовых алертов** (рост цены / дешевле рядом, счётчик непрочитанных); таблица цен по SKU (тренд 90 дн., бейджи «текущий»/«лучший», устаревшие цены); **расчёт экономии ≈ X BYN/мес** с раскрытием формулы; **«Объяснить рекомендацию»** (AI по кнопке). |
| Настройки | `#/settings` | Контекст, статус интеграции POS (fail-closed: запись выключена), тумблеры модулей. |

Сквозной принцип продукта соблюдён во всех потоках: **AI показывает данные и аргументы —
решает и отправляет человек**. Никаких автозаказов, никакой записи в POS без подтверждения.

## Структура

```
prototype/
├── index.html                 точка входа (подключает всё по порядку)
├── README.md                  этот файл
└── assets/
    ├── styles.css             дизайн-токены (1:1 из mvp.fe @theme) + компоненты
    ├── data.js                мок-данные всех доменов + форматтеры (BYN, ru-RU)
    ├── app.js                 оболочка (sidebar/topbar/bottombar), hash-роутер,
    │                          общие UI-хелперы (badge, toast, sheet, иконки)
    ├── views.invoices.js      Накладные: список · workbench · деталь
    ├── views.orders.js        Заказы: список · выбор поставщика · редактор · сообщение
    ├── views.suppliers.js     Поставщики: список · карточка · форма критериев
    ├── views.compare.js       Сравнение: алерты · таблица цен · экономия · explain
    └── views.settings.js      Настройки
```

Каждый view-модуль регистрирует свои маршруты через `route('/path', handler)` и возвращает
`{ html, mount?, title, section, container? }`. Роутер выбирает самый специфичный маршрут
(меньше wildcard-сегментов), поэтому `/invoices/import` побеждает `/invoices/:id`.

## Дизайн-система

Токены и компоненты унаследованы **из `mvp.fe/src/index.css` (`@theme`)** без изменений —
прототип намеренно выглядит как будущее приложение, а не как отдельный концепт:

- Палитра: page `#f4f5f7`, panel `#fff`, line `#e2e5ea`, accent `#2f6df0`,
  good `#1a9b56`, warn `#c47f0a`, danger `#d23b3b`, info `#0d8a8a`.
- Системный шрифт, радиусы 8–10px, `shadow-card`, пилюли-бейджи (soft-bg/strong-fg).
- Mobile-first: крупные тач-цели, нижний таб-бар, bottom-sheet, safe-area insets.
- Язык — **русский**, валюта — **BYN** (`1 234,56 BYN`), даты — `ДД.ММ.ГГГГ`.
- Бренд «Local**OS**» («OS» — акцентным синим), как в реальной оболочке.

Совместим со светлой темой (dark mode в продукте пока нет — прототип это отражает).

## Покрытие требований (трассировка)

| Экран прототипа | Домен-документы в `mvp.docs` | Статус в продукте |
|---|---|---|
| Накладные + workbench | [[LCOS-E2-invoice-intake]], [[invoices]], `invoice-status-machine`, [[LCOS-F9-line-matching]] | built |
| Поставщики + критерии | [[LCOS-E4-suppliers]], [[suppliers]], [[LCOS-F17-supplier-cards]], [[LCOS-F18-supplier-criteria]] | built |
| Заказы (закупки) | [[LCOS-E8-purchasing]], [[LCOS-F37-purchase-orders]], [[LCOS-F38-orders-ui]], [[LCOS-F39-order-message]], [[LCOS-F40-ai-order-proposal]], [[purchase_orders]], [[purchase_order_lines]] | **planned** |
| Сравнение + алерты | `plan/PHASE_F4_SUPPLIER_COMPARE`, [[LCOS-F75-supplier-price-analytics]] | **planned** |

Прототип особенно ценен для двух **planned**-доменов (Заказы и Сравнение): он фиксирует UX
до реализации бэкенда — редактор заказа, индикатор минимума, машину статусов PO,
формат сообщения поставщику, таблицу сравнения, расчёт экономии и границы AI.

### Осознанные упрощения (не переносить в код бездумно)
- Все данные — в памяти (`data.js`); нет бэкенда, авторизации, мультитенантности.
- «Распознавание» и «Объяснить» — имитация задержки, а не реальные LLM-вызовы.
- Планировщик заказа считает по мок-остаткам/порогам из `ingredients` (в проде — `stock_levels`).
- Тренд 90 дн. и среднемесячный объём — упрощённые прокси; в проде — из `invoice_lines`.
- `packing` в мок-данных кодирует множитель строкой («кор. × 12»); в проде — `packings.factor`.

## Как развивать прототип

- **Данные:** правьте `assets/data.js` (поставщики, ингредиенты, цены, алерты, заказы).
- **Новый экран:** добавьте `route('/path', () => ({ html, mount }))` в подходящий view-модуль
  (или новый файл + `<script>` в `index.html`).
- **Стили:** используйте существующие классы/токены из `styles.css` — не вводите новые цвета,
  чтобы прототип оставался зеркалом продукта.
- При переносе в `mvp.fe`: экраны ложатся на FSD (`pages/orders`, `entities/purchase-draft`,
  `entities/price-compare`), RTK Query с провайдер-паттерном `backend|mock`, переиспользование
  `SkuSelect` / `LineRow` из `features/lines-table` (см. AC в [[LCOS-F38-orders-ui]]).
