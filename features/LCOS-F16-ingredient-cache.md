# LCOS-F16 · Кэш ингредиентов (только для черновика, неавторитетный)

**Эпик:** [[LCOS-E3-sku-identity]] · **Статус:** built · **Фаза:** Phase 1

## Описание

`ingredient_cache` — это **неавторитетный, пересобираемый** снимок master-данных ингредиентов POS. Он существует, чтобы дёшево обслуживать толерантный уровень черновика/отображения, и архитектурно определён так, что **никогда** не может скомпрометировать «ров». Всю фичу несут два инварианта дизайна (`[[DEC-0011]]`):

1. **Нет FK от «рва» к кэшу.** `sku_mapping.pos_ingredient_id` — это долговечная *строка* POS id без внешнего ключа к `ingredient_cache.id` (суррогатный PK). Поэтому кэш можно целиком сбросить и пересобрать, не осиротив ни один маппинг — «ров» переживает устаревший каталог, потому что он якорится на долговечный POS id, а не на кэшированные атрибуты.
2. **Отсутствует на пути коммита.** Fail-closed commit-резолвер (`_resolve_commit_identities`, см. [[LCOS-F13-sku-identity-resolver]]) читает авторитет *только* из `sku_mapping` и делает live POS-валидацию; он никогда не читает кэш. Это закрывает **VER-022** — старая scope-асимметрия «кэш только для организации» не может повториться, потому что кэш вообще не является уровнем коммита (`[[DEC-0013]]`).

Кэш scope-aware (`scope_type`, `scope_id`) с `UNIQUE(scope_type, scope_id, pos_ingredient_id)`, держит снимок (`name`, `unit`, `category`), столбцы обнаружения изменений (`pos_version`, `content_hash`), флаг soft-delete (`is_active`, если POS удалил позицию) и `synced_at`.

> **Заметка о состоянии as-built.** Что здесь *построено и обеспечено* — это **архитектура** долговечной идентичности: таблица, её уникальность, инвариант no-FK и её намеренное отсутствие на пути коммита. Живой путь *наполнения/чтения* **ещё не подключён** — ни один сервис сейчас не читает и не пишет `ingredient_cache`, а сегодняшний draft-resolve читает таблицу `ingredients` [[LCOS-F15-sku-catalog]] напрямую. Кэш — это подготовленный, охраняемый инвариантами шов; стратегия синхронизации (ленивое TTL-обновление + принудительное обновление при открытии инвойса + перевалидация на коммите) специфицирована в `[[DEC-0011]]` и отложена, при этом delta-sync/webhooks отслеживаются как DEFER-016.

## Возможности

- Неавторитетный снимок master-данных ингредиентов POS: `name`, `unit`, `category`, `pos_version`/`content_hash` (обнаружение изменений), `is_active` (soft-delete), `synced_at`.
- Scope-aware изоляция: `scope_type`/`scope_id` с `UNIQUE(scope_type, scope_id, pos_ingredient_id)`.
- Якорится на долговечный `pos_ingredient_id`; **нет** FK от `sku_mapping` → кэш полностью пересобираем (drop + rebuild) без потери маппингов.
- По замыслу только для черновика: предназначен для толерантного уровня черновика/отображения; явно **не** авторитет пути коммита (`[[DEC-0011]]` / `[[DEC-0013]]`).
- VER-022 закрыт: кэш полностью вне пути коммита, так что никакой scope-асимметрии между черновиком и коммитом невозможно.

## Доступ по ролям

| Роль | Что может делать |
|---|---|
| [[member]] | Нет прямого UI; при наполнении кэшированные master-данные лишь информируют толерантное отображение черновика внутри его scope. |
| [[admin]] | То же, внутри своей subdivision. |
| [[superadmin]] | Межарендаторский доступ. |
| [[sqladmin-operator]] | Не участвует. |

Кэш — внутренний шов ускорения/отображения; у него нет выделенного endpoint-а, обращённого к арендатору, и он никогда не гейтирует отправку.

## Задействованные сущности

- [[ingredient_cache]] — неавторитетный снимок; scope-aware; `UNIQUE(scope_type, scope_id, pos_ingredient_id)`; суррогатный PK никогда не является целью маппинга.
- [[sku_mapping]] — намеренно **не** имеет FK к кэшу; якорится на долговечный POS id, так что переживает пересборку кэша.
- [[invoice_lines]] — снимает долговечный `pos_ingredient_id` на коммите, так что закоммиченная строка тоже переживает пересборку кэша.
- [[subdivisions]], [[organizations]] — `(scope_type, scope_id)`, по которым кэш партиционирован.

## Зависимости / связи

- **Требования:** [[sku-identity-resolver]] (определяет кэш как уровень черновика, никогда авторитет коммита), [[fail-closed]] (коммит читает авторитет только из подтверждённого `sku_mapping` + live POS), [[erp-esupl-integration]] (read-only источник, из которого тянула бы будущая синхронизация).
- **Фичи:** [[LCOS-F13-sku-identity-resolver]] (commit-резолвер, который намеренно исключает кэш), [[LCOS-F15-sku-catalog]] (таблица `ingredients`, которая фактически обеспечивает draft-resolve сегодня), [[LCOS-F14-learning-loop]] («ров», который должен пережить пересборку кэша), [[LCOS-F11-esupl-read]] (read-only-шов POS).
- **ADR:** [[DEC-0011]] (POS = SoT; кэш неавторитетен + пересобираем; нет FK), [[ADR-018]] (коммит доверяет только подтверждённым маппингам), [[DEC-0013]] (вариант A — точное совпадение в кэше никогда не авто-коммитит/авто-создаёт маппинг).

## Критерии приёмки (AC)

### Backend
- [ ] AC-BE-1. `ingredient_cache` существует с `scope_type`, `scope_id`, `pos_ingredient_id`, полями снимка (`name`/`unit`/`category`), обнаружением изменений (`pos_version`/`content_hash`), `is_active`, `synced_at` и `UNIQUE(scope_type, scope_id, pos_ingredient_id)`.
- [ ] AC-BE-2. `sku_mapping` **не** имеет FK к `ingredient_cache` (схема-тест `test_mapping_has_no_fk_to_cache`).
- [ ] AC-BE-3. Путь коммита не читает авторитет (существование / единицу) из кэша — commit-резолвер читает только `sku_mapping` и валидирует вживую против POS (`[[DEC-0011]]` / `[[DEC-0013]]`).
- [ ] AC-BE-4. Сброс и пересборка `ingredient_cache` оставляют `sku_mapping` нетронутым и разрешимым (`test_cache_rebuild_preserves_mappings`).
- [ ] AC-BE-5. Регресс VER-022: кэш отсутствует на пути коммита, так что не существует scope-асимметрии draft/commit.
- [ ] AC-BE-6. Точное совпадение в кэше без commit-eligible маппинга **не** авто-коммитит и **не** создаёт маппинг (вариант A, `[[ADR-018]]`) — покрыто merge-gate тестом.

### Frontend
- [ ] AC-FE-1. Нет выделенного UI кэша: кэшированные master-данные (когда путь наполнения появится) могут лишь информировать толерантное отображение черновика и никогда не гейтируют и не авто-подтверждают отправку.

### Прочее
- [ ] AC-OTHER-1. Инварианты кэша закреплены тестами `merge_gate` (`test_mapping_has_no_fk_to_cache`, `test_cache_rebuild_preserves_mappings`).

## Открытые вопросы / гейты

- **Наполнение/чтение не подключены** — ни один сервис сейчас не читает и не пишет `ingredient_cache`; draft-resolve использует таблицу `ingredients` [[LCOS-F15-sku-catalog]] напрямую. Таблица — подготовленный, охраняемый инвариантами шов.
- **Стратегия синхронизации отложена** — ленивое TTL-обновление + принудительное обновление при открытии инвойса + перевалидация на коммите специфицированы в `[[DEC-0011]]`, но не реализованы; delta-sync/webhooks = DEFER-016 (до триггера по объёму/частоте).
- **VER-021** — долговечность `pos_ingredient_id` — краеугольный камень, на котором держится вся неавторитетная модель; эмпирически не подтверждена, запуск владельцем, merge гейтирован.
