# LCOS-F31 · Автообрезка краёв документа (OpenCV.js, на стороне клиента)
**Epic:** [[LCOS-E6-ocr-quality]] · **Status:** planned · **Phase:** Phase 1

## Описание

Фото накладной на столе обычно включает поверхность стола, руки и фоновый хлам. Отправка всего кадра в vision-модель тратит бюджет 1568px на недокументные пиксели и снижает качество распознавания. Сегодня пользователь должен вручную перетаскивать четыре угловых маркера `ImageCropper`. Эта фича добавляет **автоматическое** обнаружение краёв документа, так что кадр предлагается за него, с ручной коррекцией, всегда доступной как фоллбэк («auto-first, manual-optional»).

Обнаружение выполняется полностью на клиенте лениво загружаемым OpenCV.js (~8MB WASM). Он загружается только при открытии сканера, поэтому не должен попадать в основной бандл. Пайплайн — классический поиск четырёхугольника: grayscale → blur → Canny/Otsu края → крупнейший четырёхугольник → `warpPerspective`; результат передаётся существующему `ImageCropper` с пред-размещёнными угловыми маркерами. Согласно [[ADR-014]] бинаризация/пороговая обработка финального изображения не вводится — модель хочет естественное фото, просто обрезанное и выпрямленное.

Критически поток **fail-open**: если OpenCV не загрузился (офлайн, битый WASM) или не нашёл уверенного четырёхугольника, пользователь просто получает сегодняшний ручной кроппер, и распознавание никогда не блокируется. Принудительный шаг подтверждения появляется **только** при слабом сигнале качества (низкая уверенность краёв, сильный перекос, документ заполняет <60% кадра); иначе обрезка применяется тихо. Порядок пайплайна фиксирован: crop → существующий `normalizeImage` (единый проход resize+JPEG), так что обрезанная страница кодируется ровно один раз (контракт препроцессинга [[provider-abstraction]]).

## Возможности

- Автоматическое обнаружение границ документа на захваченном изображении лениво загружаемым OpenCV.js (grayscale → blur → Canny/Otsu → крупнейший четырёхугольник → `warpPerspective`).
- Обнаруженный четырёхугольник показывается как пред-размещённые угловые маркеры в существующем `ImageCropper`; ручная корректировка всегда доступна.
- Тихое применение при сильном сигнале; принудительный шаг подтверждения только при слабом сигнале (низкая уверенность краёв, сильный перекос или документ <60% кадра).
- Порядок пайплайна crop → `normalizeImage` (один resize до `MAX_LONG_EDGE = 1568` + один JPEG-encode) — без наложенной компрессии; зеркалит существующий однопроходный `cropImage`.
- Fail-open: любая ошибка загрузки OpenCV/WASM или обнаружения откатывается к ручной обрезке; распознавание идёт независимо.
- OpenCV.js code-split и лениво загружается — отсутствует в начальном/основном бандле.
- Нет бинаризации/пороговой обработки выходного изображения (ADR-014).

## Задействованные сущности

- [[invoices]] — автообрезка улучшает байты страницы, которые становятся `InvoiceDraft` на `/recognize`; на этом шаге ничего не персистится.

## Зависимости / связи

- **Требования:** [[provider-abstraction]] (контракт препроцессинга: страница — это один намеренный проход resize+JPEG перед тем, как её получит OCR-шов).
- **Фичи:** питает [[LCOS-F8-ocr-recognition]] / [[LCOS-F29-multipage-recognize]] (более чистые страницы на запрос); в паре с [[LCOS-F32-camera-capture]] (фото с камеры больше всего нуждаются в обрезке). Существующий `ImageCropper` (ручная обрезка) — фоллбэк-поверхность.
- **ADR:** [[ADR-014]] (нет бинаризации/порога в подготовке изображения).

## Критерии приёмки (AC)

### Frontend
- [ ] AC-FE-1. На 5 реальных фото накладных со стола четырёхугольник документа обнаруживается и предлагается автоматически как минимум в 4; угловые маркеры пред-размещены в `ImageCropper`.
- [ ] AC-FE-2. Ручная корректировка углов всегда доступна и по подтверждению производит готовую для провайдера страницу через тот же однопроходный crop→normalize (`cropImage` → `MAX_LONG_EDGE`, один JPEG-encode).
- [ ] AC-FE-3. Принудительный шаг подтверждения появляется только при слабом сигнале качества (низкая уверенность краёв / сильный перекос / документ <60% кадра); при сильном сигнале обрезка применяется тихо.
- [ ] AC-FE-4. Fail-open: при недоступном OpenCV.js (офлайн / битый WASM) поток деградирует к сегодняшнему ручному кропперу, и распознавание не блокируется.
- [ ] AC-FE-5. OpenCV.js code-split и лениво загружается только при открытии сканера — проверка размера чанка `vite build` подтверждает, что его нет в основном бандле.
- [ ] AC-FE-6. Бинаризация/пороговая обработка не применяется к выходному изображению (ADR-014); страница остаётся естественным обрезанным фото.

### Backend
- [ ] AC-BE-1. Изменений на бэкенде нет: `/invoices/recognize` по-прежнему получает единый нормализованный JPEG на страницу, и его контракт MIME/размера без изменений; автообрезка не должна производить байты, превышающие `MAX_LONG_EDGE` или лимит размера.

### Прочее (QA/perf)
- [ ] AC-OTHER-1. Бандл/perf: открытие сканера триггерит ленивую загрузку OpenCV один раз; time-to-interactive начального маршрута приложения не затронуто (OpenCV отсутствует в entry-чанке).

## Открытые вопросы / гейты

- **Fail-open обязателен** — автообрезка — вспомогательное средство, никогда не гейт; любой сбой молча откатывается к ручной обрезке.
- **Пороги слабого сигнала** (уверенность краёв, угол перекоса, <60% кадра) нуждаются в тюнинге против реальных фото; зафиксировать настроенные значения в `work/phase-s2.md`.
- **Вес WASM** — ~8MB должны оставаться вне основного бандла; подтвердить кэширование, чтобы повторные сканы не перескачивались.
