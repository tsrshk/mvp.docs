# LCOS-F74 · Ассортимент поставщика и свежесть цен

**Эпик:** [[LCOS-E4-suppliers]] · **Статус:** built · **Фаза:** Phase 1

## Описание

Фича даёт read-model «ассортимент поставщика» поверх позиций прайса, распарсенных фичей [[LCOS-F73-price-list-parsing]]. Ассортимент — это то, что поставщик **предлагает сейчас**: последняя `price_list_line` на пару (`supplier`, разрешённый SKU / нормализованный `raw_name`), с ценой и **свежестью** наблюдения.

Свежесть считается по `observed_at` позиции: карточка показывает «обновлено N дней назад» и подсвечивает **устаревшие** позиции по настраиваемому порогу (по умолчанию 30 дней). Так внутренний пользователь видит не только цены, но и то, насколько им можно доверять — устаревший прайс визуально отделён от свежего.

Read-model учитывает только **последнюю версию** прайса на каждую позицию: несколько загрузок (`version` в [[price_list_upload]]) дают несколько наблюдений в append-only `price_list_line`, но в ассортименте берётся самое позднее по `observed_at` наблюдение на пару (`supplier`, SKU / `raw_name`). Историю цен во времени несёт `supplier_prices` (проекция из [[LCOS-F73-price-list-parsing]]); эта фича не пишет данные, а только читает.

## Возможности

- Read-model «ассортимент»: последняя `price_list_line` на СЫРУЮ ПОЗИЦИЮ поставщика (`supplier_id`, `raw_name`) внутри арендатора — позиции показываются как есть (сторона поставщика).
- Привязка к нашему SKU (`pos_ingredient_id`/`ingredient_id` + `resolution_method`) — **аннотация** к позиции («→ SKU X» / «не привязано»), а НЕ ключ группировки: варианты поставщика (напр. 2.5%/3.2%) с одним нашим SKU НЕ схлопываются (решение владельца). Привязка/подбор — на экране строк ([[LCOS-F73-price-list-parsing]]).
- Цена позиции: `price_per_base_unit` (нормализованная к базовой единице), с fallback на `price` + `currency`, если фасовка не разрешена.
- Свежесть по `observed_at`: «обновлено N дней назад»; признак актуальности (`is_stale`) по порогу устаревания.
- Подсветка устаревших позиций: порог по умолчанию 30 дней, конфигурируем через `system_settings` при необходимости.
- Учитывается только последняя версия прайса на позицию; более ранние наблюдения остаются в append-only истории, но в ассортимент не попадают.

## Задействованные сущности

- [[price_list_line]] — источник ассортимента; группировка по (`supplier_id`, `pos_ingredient_id` / нормализованный `raw_name`), выбор последней по `observed_at`; поля цены `price_per_base_unit` / `price` / `currency` и `is_available`.
- [[suppliers]] — локальный SSOT карточки поставщика ([[ADR-021]]); ассортимент рендерится вкладкой на карточке.

## Зависимости / связи

- **Требования:** [[sku-identity-resolver]] (разрешённый SKU даёт стабильный ключ группировки ассортимента), [[erp-esupl-integration]] (matching поставщика; карточка — локальный SSOT).
- **Фичи:** [[LCOS-F73-price-list-parsing]] (наполняет `price_list_line`, которую эта фича читает), [[LCOS-F13-sku-identity-resolver]] (резолв `pos_ingredient_id` для группировки).
- **ADR:** [[ADR-021]] (поставщик — локальный SSOT; список/карточка читаются локально).

## Критерии приёмки (AC)

### Backend
- [ ] AC-BE-1. `GET /suppliers/{id}/assortment` возвращает позиции ассортимента с ценой (`price_per_base_unit` / `price` + `currency`), `observed_at` и признаком актуальности (`is_stale`).
- [x] AC-BE-2. Каждая СЫРАЯ позиция поставщика (`supplier_id`, `raw_name`) представлена **ровно одной** строкой — последней по `observed_at` (только последняя версия на позицию).
- [x] AC-BE-3. Позиции НЕ схлопываются по нашему SKU; привязка к SKU (`pos_ingredient_id`/`resolution_method`) возвращается как аннотация к позиции.
- [ ] AC-BE-4. `is_stale` вычисляется по порогу устаревания (по умолчанию 30 дней) относительно `observed_at`.
- [ ] AC-BE-5. Тест изоляции арендатора: ассортимент одного `organization_id` не виден из-под другого (`test_assortment_tenant_isolation`).
- [ ] AC-BE-6. Эндпоинт за гейтом модуля suppliers; несуществующий/чужой `supplier_id` → 404.

### Frontend
- [ ] AC-FE-1. На карточке поставщика есть вкладка «Ассортимент».
- [ ] AC-FE-2. Таблица: SKU/название → цена, дата обновления, бейдж «устарело» для устаревших позиций.
- [ ] AC-FE-3. Устаревшие позиции визуально выделены (подсветка/бейдж) по признаку `is_stale` из ответа.

### Прочее
- [ ] AC-OTHER-1. Порог устаревания зажурналирован (напр. 30 дней) и, при необходимости, вынесен в `system_settings`.

## Открытые вопросы / гейты

- **Ключ группировки при неразрешённом SKU** — нормализация `raw_name` для fallback-ключа должна быть детерминированной; правила нормализации разделяются с резолвером [[LCOS-F13-sku-identity-resolver]].
- **Порог устаревания** — 30 дней как дефолт; вынос в `system_settings` (глобально / на организацию) уточняется при появлении требования.
- **Зависит от F73** — ассортимент пуст, пока [[LCOS-F73-price-list-parsing]] не наполнит `price_list_line`.
