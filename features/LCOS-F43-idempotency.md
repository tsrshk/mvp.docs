# LCOS-F43 ┬Ě Server-side submit idempotency (DEFER-04)
**Epic:** [[LCOS-E8-purchasing]] ┬Ě **Status:** planned ┬Ě **Phase:** Phase 1

## Description

Makes invoice submit safe to retry, on the server. Today a duplicate-submit guard lives only in the browser (`shared/pos/sentRegistry.ts`) ÔÇö fragile against reloads, multiple tabs and races. DEFER-04 moves the guarantee to the backend: the frontend sends an `Idempotency-Key` (uuid) header on `POST /invoices`; the backend stores it (a column on `invoices` with a unique constraint) and, on a repeat with the same key, returns the **original result without a second ERP call**. Once this exists, the browser guard is deleted (along with its re-export in `shared/pos/index.ts`).

This is a Phase-1 prerequisite for turning on permanent live writes ([[LCOS-F44-live-closeout]]): with the ERP write path armed, a retried submit must never create a second invoice in Esupl. The unique key on `invoices` is what enforces "one submit Ôćĺ one invoice Ôćĺ one ERP call" even under retries.

## Capabilities

- `Idempotency-Key` (uuid) header accepted on `POST /invoices`, generated by the frontend.
- Key persisted on `invoices` with a unique constraint; the domain `InvoiceDraft` body stays unchanged.
- Repeat submit with the same key Ôćĺ original result returned, no second ERP call.
- Frontend retry with the same key is safe.
- Browser-side `shared/pos/sentRegistry.ts` guard removed (with its re-export in `shared/pos/index.ts`).
