# organizations · tenant

**Scope:** org (isolation root) · **Status:** built

## Purpose
An organization is the **tenant** and the root of the hard isolation boundary.
`organization_id` is denormalized into every tenant row; a tenant query is IMPOSSIBLE
without scoping by `organization_id` (see [[multitenancy]]). One organization is bound to
one Esupl team (POS/ERP), see [[erp-esupl-integration]].

## Key fields
| Field | Type | Null | Notes |
|---|---|---|---|
| `id` | uuid PK | no | `uuid4`, generated by the application |
| `name` | varchar(512) | no | display name |
| `legal_name` | varchar(512) | yes | legal name |
| `esupl_team_id` | integer | yes | POS binding: organization = one Esupl team |
| `created_at` / `updated_at` | timestamptz | no | `TimestampMixin` |

The tenant's POS access token is **not stored here** — the secret is moved out into
[[integration_credentials]] (scope=org, provider=esupl); the secrets SSOT is a single
table (see [[secret-encryption]]).

## Relations, FK, uniqueness
- `subdivisions` — one-to-many, `cascade="all, delete-orphan"` (deleting the organization
  cascades to delete its subdivisions).
- No FK of its own (this is the root entity).
- Child tenant tables reference `organizations.id` via `OrganizationScopedMixin` with
  **`ondelete="RESTRICT"`** — an organization with operational data cannot be deleted while
  that data exists.
