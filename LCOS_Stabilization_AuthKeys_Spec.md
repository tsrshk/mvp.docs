# LCOS — Спецификация стабилизации: Auth, управление ключами, конфигурация из супер-админки

**Назначение документа.** Описательные требования по *флоу работы* приложения для верификации архитектуры на этапе фактической имплементации. Документ описывает *что* и *почему*, а не конкретный код. Идентификаторы (таблицы, поля, эндпоинты) даны на английском для однозначности.

**Цель пасса стабилизации.** Довести бэк и фронт до состояния, в котором можно end-to-end протестировать:
1. Авторизацию (app-пользователи).
2. Управление ключами к внешним интеграциям (POS/ERP).
3. Управление AI API-ключами.
4. Управление настройками приложения из супер-админки.

Стек-контекст (фиксирован ранее, не меняется): FastAPI + SQLAlchemy 2.0 async + PostgreSQL 16 + Alembic + SQLAdmin; JWT access (15 мин, HttpOnly cookie) + opaque refresh (rotation + reuse detection через `family_id`); organization-as-tenant, single default tenant для Phase 1; фронт React + Vite + TS + RTK Query (FSD).

## 1. Архитектурные инварианты (проверять в первую очередь)

Эти правила — критерий приёмки архитектуры. Нарушение любого = блокер.

| # | Инвариант | Что это исключает |
|---|-----------|-------------------|
| I1 | **Один источник истины на каждое значение конфигурации.** Значение читается либо из `.env`, либо из БД — никогда из обоих. | Дублирование, «тихие» фолбеки, мёртвые ветки чтения. |
| I2 | **Граница `.env` / БД по смыслу:** `.env` = boot & trust-root (неизменяемо в рантайме); БД = операционная конфигурация (меняется супер-админом без редеплоя). | Секреты-в-коде; необходимость редеплоя ради смены ключа. |
| I3 | **Fail-closed везде.** Отсутствие/невалидность ключа или недоступность зависимости (VPN) → жёсткая ошибка. Нет деградации, нет альтернативного источника. | «No fallbacks» на уровне рантайма. |
| I4 | **Секреты в БД шифруются at-rest.** Открытый текст ключей в Postgres запрещён. Master-ключ шифрования — в `.env`. | Утечка секретов при доступе к дампу БД. |
| I5 | **Две независимые плоскости аутентификации.** SQLAdmin (super-admin) и app-пользователи (JWT) не пересекаются: разные бэкенды аутентификации, разные сессии/куки, разные таблицы прав. | Эскалация из app-роли в супер-админа и наоборот. |
| I6 | **Seam есть — реализация одна.** Адаптеры (провайдеры интеграций) имеют интерфейс, но ровно одну реализацию на провайдер. Альтернативы не пишутся до реального триггера. | Мёртвый код «на будущее». |
| I7 | **Один активный секрет на (scope, provider).** Ротация = замена активного, а не накопление неиспользуемых. | Неоднозначность «какой ключ используется». |

## 2. Граница конфигурации: `.env` vs БД

**Принцип (I2):** `.env` содержит только то, что нужно, чтобы *поднять систему и установить корень доверия*. Всё, что оператор может захотеть поменять на живой системе, — в БД под супер-админкой.

| Значение | Источник | Обоснование |
|----------|----------|-------------|
| `DATABASE_URL` | `.env` | Нужен до всего остального. |
| `MASTER_ENCRYPTION_KEY` (шифрование секретов в БД) | `.env` | Корень доверия; им расшифровывается содержимое БД — не может лежать в БД. |
| `JWT_SIGNING_KEY` | `.env` | Корень доверия для подписи токенов. |
| Bootstrap super-admin (login + password/hash) | `.env` | Курица-яйцо: админку нельзя администрировать до её появления. |
| Статическая конфигурация деплоя (порты, gluetun/VPN endpoint, режим окружения) | `.env` | Неизменяемо в рантайме, свойство деплоя. |
| **AI API-ключ (Anthropic)** | **БД** | Операционный секрет, меняется без редеплоя. |
| **POS/ERP-ключи (Esupl и т.п.)** | **БД** | Операционный секрет, per-org. |
| Настройки приложения (флаги, пороги OCR, отображение) | **БД** | Управляются супер-админом. |

**Проверка для агента:** в кодовой базе не должно быть ни одного значения из нижней части таблицы, читаемого из `os.environ` / settings-объекта `.env`. Если такое чтение есть — это нарушение I1 и, вероятно, оставшийся фолбек.

## 3. Целевая модель данных (сверить с фактической схемой БД)

Ниже — концептуальная целевая структура. Агент сверяет существующие таблицы/поля/ограничения с этим и докладывает расхождения. Имена — ориентир, важна семантика и ограничения.

### 3.1 Идентичность и tenancy

- **`users`** — глобальная таблица пользователей приложения. Поля минимум: `id`, `email` (unique), `password_hash`, `is_active`, `created_at`. Не содержит роли напрямую (роль — через membership).
- **`organizations`** — арендатор (tenant). В Phase 1 существует одна запись (default org).
- **`subdivisions`** — подразделения внутри org (`org_id` FK).
- **`memberships`** — связь `user_id` ↔ `org_id` с полем `role`. Право доступа определяется здесь, а не в `users`. Уникальность `(user_id, org_id)`.

### 3.2 Сессии (refresh rotation)

- **`refresh_token_families`** (или эквивалент сессий):
  - `id`, `user_id`, `family_id`, `current_token_hash` (хранить хэш, не сам токен), `issued_at`, `last_used_at`, `expires_at` (sliding idle), `revoked` (bool/reason).
  - **Reuse detection:** предъявление уже ротированного/использованного токена семейства → инвалидация всего `family_id`.
  - Refresh-токены **opaque** (не JWT), хранятся в БД по хэшу.

### 3.3 Операционная конфигурация и секреты

- **`app_settings`** — несекретные настройки приложения, редактируемые супер-админом.
  - Рекомендация: key/value с **валидируемым реестром допустимых ключей** на уровне приложения (никаких orphan-ключей). Поля: `key` (unique, из белого списка), `value` (typed/serialized), `updated_at`, `updated_by`.
  - Причина выбора KV+реестр, а не колонка-на-настройку: избегаем миграции ради каждой настройки и мёртвых колонок (I6), но белый список исключает мусорные записи.

- **`integration_credentials`** (секреты — AI и POS/ERP):
  - `id`
  - `scope` — enum: `platform` | `org` | `subdivision`
  - `org_id` / `subdivision_id` — nullable, заполняется по scope
  - `provider` — enum: `anthropic` | `esupl` | (будущие провайдеры — строкой того же enum)
  - `encrypted_value` — зашифровано master-ключом из `.env` (I4)
  - `is_active` — bool
  - `created_at`, `updated_at`, `rotated_at`, `created_by`
  - **Ограничение (I7):** partial unique index на `(scope, provider, coalesce(org_id,...), coalesce(subdivision_id,...))` где `is_active = true` → максимум один активный секрет на скоуп+провайдер.

**Скоуп по умолчанию (подтвердить):**
- `anthropic` → `scope = platform` (один AI-ключ на деплой).
- `esupl`/POS → `scope = org` (ключ принадлежит заведению).

**Проверка для агента:** нет ли отдельной «таблицы под AI-ключ» и «таблицы под POS-ключ» с дублирующей логикой — это должно быть одной таблицей с `provider`/`scope` (I6). Если разнесено без причины — кандидат на консолидацию.

## 4. Плоскость 1 — Аутентификация app-пользователей

### 4.1 Флоу

1. **Login** — `POST /auth/login` (email + password).
   - Валидация → выдать access JWT (15 мин, HttpOnly cookie) + opaque refresh (HttpOnly cookie, новый `family_id`, sliding idle-окно).
   - Access-payload несёт: `user_id`, активный `org_id` (membership), `role`.
   - Cookie config: `HttpOnly`, `SameSite=Lax`, **без `Secure`** (локальный WiFi PWA-тест). Это осознанное решение для Phase 1 — пометить как требующее пересмотра при выходе за пределы локальной сети.
2. **Access к защищённым эндпоинтам** — проверка подписи и срока access JWT; авторизация по `role` из membership.
3. **Refresh** — `POST /auth/refresh`.
   - Валидировать refresh по хэшу в `refresh_token_families`.
   - Ротация: выдать новый refresh в том же `family_id`, обновить `current_token_hash` и `last_used_at`.
   - **Reuse detection:** если предъявлен старый (уже ротированный) токен семейства → revoke всего `family_id`, форс-релогин.
4. **Logout** — `POST /auth/logout` → revoke текущего семейства, очистить куки.

### 4.2 Fail-closed правила плоскости
- Нет валидного access и нет валидного refresh → 401, без «гостевого» режима.
- Просрочен idle-window refresh → 401, релогин.

### 4.3 Что проверить у агента
- Refresh хранится хэшированным (не в открытом виде).
- Reuse detection реально инвалидирует **всё семейство**, а не один токен.
- Роль берётся из `memberships`, а не захардкожена/из `users`.
- Нет «запасного» пути аутентификации (например, статический токен из `.env`) — это нарушение I3/I5.

## 5. Плоскость 2 — Супер-админка (SQLAdmin)

### 5.1 Флоу
1. **Аутентификация супер-админа** — отдельный auth-backend SQLAdmin, bootstrap-креды из `.env`. Отдельная сессия/кука, не связанная с app JWT (I5).
2. **Область управления супер-админа:**
   - `organizations`, `subdivisions`, `users`, `memberships`.
   - `app_settings` (настройки приложения).
   - `integration_credentials` (AI- и POS-ключи).
3. **Обращение с секретами в формах SQLAdmin (критично):**
   - На запись: форма принимает **plaintext**, модель **шифрует до персиста** (plaintext никогда не сохраняется).
   - На чтение/отображение: значение **маскируется** (например, показать только `provider`, `scope`, `updated_at`, `is_active` и последние 4 символа — но не сам ключ).
   - Ротация: обновление активной записи (или замена активной новой) с обновлением `rotated_at`.

### 5.2 Что проверить у агента
- SQLAdmin **не** использует таблицу `users`/`memberships` приложения для входа (плоскости разделены).
- Формы секретов не логируют и не возвращают plaintext.
- Нет представления/эндпоинта, отдающего расшифрованный секрет наружу (расшифровка только внутри рантайма при вызове провайдера).

## 6. Управление ключами POS/ERP

> Терминология: трактую «POS» как внешний интеграционный провайдер. В текущей модели это Esupl (ERP). Если есть отдельная POS-система, отличная от Esupl — это **просто ещё одна строка `provider`** в `integration_credentials`, механизм тот же. **Подтвердить трактовку.**

### 6.1 Флоу
1. Супер-админ создаёт/ротирует ключ провайдера для конкретной org (`scope = org`, `provider = esupl`, `org_id`).
2. При операции интеграции для org X рантайм запрашивает **активный** секрет `(scope=org, provider=esupl, org_id=X)`.
3. **Fail-closed (I3):** нет активного ключа → операция интеграции завершается ошибкой. Нет чтения из `.env`, нет «пропустить и продолжить».

### 6.2 Что проверить у агента
- Резолвинг ключа детерминированный: ровно один активный на (scope, provider, org). Нет цепочки «попробовать env → потом БД» (это фолбек, нарушение I3).
- Ключ расшифровывается только в точке вызова провайдера, не кэшируется в открытом виде дольше необходимого (см. §8).

---

## 7. Управление AI API-ключом + рантайм-резолвинг

### 7.1 Флоу
1. Супер-админ задаёт/ротирует Anthropic-ключ (`scope = platform`, `provider = anthropic`).
2. Перед AI-вызовом рантайм:
   - **(a)** проверяет доступность VPN (gluetun, fail-closed) — если VPN недоступен, запрос **отклоняется**, не переотправляется в обход;
   - **(b)** запрашивает активный `anthropic`-секрет; если его нет — **жёсткая ошибка**, вызов не выполняется.
3. Обе проверки — до формирования запроса к API.

### 7.2 Что проверить у агента
- Нет пути, где при отсутствии ключа в БД читается ключ из `.env` (частый остаточный фолбек).
- VPN-проверка и наличие ключа — обе fail-closed, порядок явный.
- Единственная реализация `OCRProvider`/AI-провайдера (I6); альтернативных адаптеров нет.

## 8. Кэширование секретов и инвалидация при ротации (требует явного решения)

Проблема: если ключ читается из БД и кэшируется, ротация в админке может «не долетать» до рантайма.

Допустимые варианты (агент выбирает **явно**, не по умолчанию):
- **A. Без кэша** — читать секрет из БД на каждую операцию. Просто, консистентно, лишний запрос к БД.
- **B. Кэш с явной инвалидацией** — кэшировать, но сбрасывать при записи в `integration_credentials` (сигнал/версия/updated_at-проверка).
- **C. Кэш с коротким TTL** — компромисс; ротация «долетает» за TTL.

Требование: выбор задокументировать в коде; **молчаливого** кэша без инвалидации быть не должно (иначе ротация ключа неотличима от бага).

Рекомендация для Phase 1: **A** (без кэша) — минимум скрытого состояния, соответствует принципу «no fallbacks / один источник». Оптимизировать только при реальном триггере нагрузки.

## 9. Чек-лист «мёртвый код / неиспользуемые компоненты»

Агент активно ищет и докладывает:

1. Любое значение из §2-«БД», читаемое также из `.env` (двойной источник → фолбек).
2. Ветки резолвинга ключей вида «env → db» или «db → hardcoded default».
3. Более одной реализации на любой провайдер-seam без активного триггера (I6).
4. Отдельные таблицы под AI-ключ и POS-ключ с дублирующей логикой (кандидат на консолидацию в `integration_credentials`).
5. Роли/enum-значения, не используемые ни одним code-path.
6. Эндпоинты/представления, отдающие расшифрованные секреты.
7. История секретов/аудит-таблицы, если аудит не является требованием Phase 1 (не заводить «на будущее»).
8. Гостевой/анонимный auth-путь или статический сервисный токен в обход JWT.

## 11. Критерии приёмки (сценарии для тестирования)

**Auth:**
- [ ] Login выдаёт access+refresh в HttpOnly куках с корректной конфигурацией.
- [ ] Защищённый эндпоинт доступен с валидным access, отклоняется без него (401).
- [ ] Refresh ротирует токен в рамках `family_id`; `last_used_at` обновляется.
- [ ] Повторное предъявление ротированного refresh инвалидирует всё семейство.
- [ ] Logout ревокает семейство; после него refresh не работает.
- [ ] Роль читается из membership и гейтит доступ.

**Супер-админка:**
- [ ] Вход в SQLAdmin отдельными кредами (`.env`), не app-пользователем.
- [ ] Создание/редактирование `app_settings` из админки отражается в приложении.
- [ ] Секрет вводится plaintext, в БД лежит зашифрованным, в UI маскирован.

**POS/ERP-ключи:**
- [ ] Ключ задаётся из админки на org; операция интеграции его использует.
- [ ] При отсутствии активного ключа операция падает (fail-closed), без чтения из `.env`.

**AI-ключ:**
- [ ] Ключ задаётся из админки (platform); AI-вызов его использует.
- [ ] При VPN down — вызов отклоняется, не переотправляется.
- [ ] При отсутствии ключа в БД — жёсткая ошибка, без фолбека на `.env`.

**Инварианты:**
- [ ] Grep по кодовой базе не находит ни одного «БД-значения», читаемого из `.env`.
- [ ] На каждый provider-seam ровно одна реализация.
