# ADR-022 · Диплинки и tenant-скоупинг

**Status:** accepted · **Date:** 2026-07-16

## Контекст
Редизайн поставщиков (таблица + страница деталей с табами) и деталь накладной вводят диплинки по всему фронтенду: URL должен адресовать конкретную табу конкретного поставщика/накладной, переживать reload, back/forward и прямой заход по ссылке. Одновременно приложение мультитенантно ([[ADR-008]]: organization-as-tenant), активный контекст (подразделение) переключается через `POST /auth/switch-context`, а идентичность поставщика — локальная ([[ADR-021]]: `SupplierOut.id` — локальный SSOT, `external_id` — только POS-matching). Нужно зафиксировать, что именно кодируется в URL, а что — нет, чтобы диплинки не протекли за границу изоляции тенанта и не завязались на эфемерный POS-id.

## Решение
1. **URL — источник истины для идентичности сущности и активной табы.** `/suppliers/:supplierId/:tab` и `/invoices/:invoiceId` полностью определяют, что показано. Табы поставщика — **сегменты пути через вложенные маршруты** (не `?tab=`): участвуют в крошках, заголовке, `NavLink isActive` и кнопке «назад». Невалидный слаг табы → `<Navigate to="info" replace>`.
2. **Tenant в URL не кодируется.** Бэкенд — граница изоляции: чужой/несуществующий id → 404/422 → экран «не найдено в текущем контексте». При `switch-context` с детального маршрута — redirect на родительский список (`replace`), т.к. локальный id не существует в новом скоупе. Списки и прочие маршруты остаются на месте.
3. **id в URL = локальный `SupplierOut.id`,** а не `external_id`. `external_id` фигурирует только в POS-match/prepare ([[ADR-021]], ARCH §8), в URL не попадает.
4. **Wizard `/invoices/new` остаётся вне URL-состояния.** Шаг мастера и фото живут в Redux — фото не переживают reload, диплинк на шаг мастера бессмыслен (осознанное решение, согласуется с photo-first [[ADR-013]]). Деталь накладной read-only; гидрация воркбенча по id отложена.
5. **SSOT путей — `shared/config/routes.ts`** (`ROUTES` билдеры + `ROUTE_PATTERNS` для `matchPath`). Все `navigate`/`<Link>`/паттерны маршрутов резолвятся через него — форма URL определена в одном месте.
6. **Метаданные маршрутов — через `handle` + `useMatches()`** (заголовок/крошки/футер), а не сопоставлением по «голому» pathname (параметрные URL молча ломают словари точного совпадения). Вторичное состояние (`?hidden=`, `?upload=`, `?sku=`) — в searchParams с сеттерами `replace: true`.

## Последствия
- Диплинк-QA обязателен по матрице: reload / прямой заход разлогиненным (→ login → возврат на intended URL) / back-forward / switch-context / несуществующий и нечисловой id / выключенный `module_suppliers_enabled` (404).
- `ContextSwitcher` переведён с fire-and-forget на awaited mutation с `.unwrap()`; после успеха — `matchPath` по `ROUTE_PATTERNS` деталей и redirect на список. Существующая инвалидация тегов (`Me`/`Invoice`/`Supplier`/`Ingredient`) сохранена.
- `/invoices/:invoiceId` матчит и `/invoices/new`; при редиректе после switch-context маршрут создания исключается явно.
- Intended-URL после логина санируется (`sanitizeInternalPath` — только внутренние пути, защита от open-redirect).
