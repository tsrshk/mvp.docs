# DEC-0011 ┬╖ POS as the source of truth for ingredient identity; mapping on a durable POS ID; two-phase authority

**Status:** Accepted ┬╖ **Date:** 2026-07-08 ┬╖ **Codified in:** [[ADR-018]]
**Depends on:** VER-021 (stability of `pos_ingredient_id`), ALIGN-014 (`ERPProvider` READ seam)
**Related:** DEC-0012 ([[ADR-019]], `source_key` key rule), [[DEC-0013]]
**Blocks:** F5тАУF10 (in the ingredient sync part)

## Context
The POS (Esupl) holds the ingredient catalog. LCOS recognizes an invoice and matches its lines to POS ingredients (SKU matching). We must decide: who owns the ingredient data, where it is stored, how to avoid desync тАФ while the invoice commit is the responsible stage and must obey fail-closed.

The decision is driven by separating two entities:
- **Ingredient master data** (`name`, `unit`, `category`) тАФ a foreign asset. The owner is the POS. LCOS is never authoritative over it.
- **Mapping** (`source_key` тЖТ ingredient identity) тАФ an LCOS asset. An accumulating moat. LCOS is the sole source of truth.

## Decision
1. **POS is the sole SoT for ingredient identity and attributes.** LCOS does not create or edit ingredient master data as authoritative.
2. **LCOS owns the mapping.** `sku_mapping` is anchored to a durable POS identifier (`pos_ingredient_id`), NOT to the name and NOT to the local cache's surrogate PK.
3. **The local `ingredient_cache` is non-authoritative.** It serves only the draft matching phase and display. It can be fully dropped and rebuilt without losing mappings.
4. **Two-phase authority:**
   - **Phase 1 тАФ Matching (draft; tolerant, cheap):** resolution via `ingredient_cache` in the order `subdivision mapping тЖТ org mapping тЖТ fuzzy тЖТ AI`. The result is a hypothesis, not a fact.
   - **Phase 2 тАФ Invoice commit (responsible; fail-closed):** before writing the receipt, each resolved `pos_ingredient_id` is validated with a live query against the POS. POS unavailability / a missing ID / a `unit` discrepancy тЖТ the commit is blocked, the line goes to review. Authority at this stage is NOT taken from the cache.

**Why the cache does not violate "no fallbacks":** a fallback = substituting a trusted source with a less trusted one in order to complete a responsible action anyway. Here the responsible action (commit) remains fail-closed and always goes to the live POS; the cache serves only the optional draft, where it is explicitly marked as a hypothesis.

## Normative rules (for the agent)

**MUST**
- `sku_mapping.pos_ingredient_id` MUST reference a durable POS ID.
- The commit path MUST perform live validation of each `pos_ingredient_id` against the POS before writing the receipt.
- A `unit` discrepancy (mapping-time vs live POS) MUST block the commit and trigger re-validation of the corresponding mapping.
- `ingredient_cache` MUST be fully rebuildable (drop + rebuild) without changing `sku_mapping`.
- Mapping resolution MUST honor the priority: `subdivision` before `org`.

**MUST NOT**
- `sku_mapping` MUST NOT have an FK to `ingredient_cache.id` (or any surrogate PK of the cache).
- The commit path MUST NOT read authority (existence / `unit` of an ingredient) from `ingredient_cache`.
- A `unit` discrepancy MUST NOT silently update the cache and continue the commit.
- LCOS MUST NOT store ingredient master data as an authoritative source.
- This model MUST NOT introduce an embedding dependency (the unused `pgvector` column must not be pulled into matching; its fate is a separate dead-code cleanup task тЖТ backlog **DEC-02**, `invoice_lines.sku_embedding` is dead).

## Data model (sketch; finalized in [[architecture]])

```
ingredient_cache            -- NOT authoritative: acceleration + display only
  id              PK (surrogate)      -- NEVER a reference for mapping
  scope_type      enum(org|subdivision)
  scope_id        FK(org_id|subdivision_id)
  pos_ingredient_id           -- durable POS ID (see VER-021)
  name, unit, category        -- snapshot from POS
  pos_version | content_hash  -- change detection on refresh
  is_active       bool                -- soft-delete if POS removed it
  synced_at       timestamptz
  UNIQUE(scope_type, scope_id, pos_ingredient_id)

sku_mapping                 -- authoritative: LCOS asset (moat)
  id              PK
  scope_type      enum(org|subdivision)
  scope_id        FK
  source_key      text                -- see DEC-0012 (normalization/key rule)
  supplier_external_id        -- added by DEC-0012 / ADR-019 (composite key)
  pos_ingredient_id           -- durable POS ID; NOT an FK to ingredient_cache.id
  method          enum(manual|fuzzy|ai)
  confidence      numeric
  confirmed_by, confirmed_at
  created_at, updated_at
  UNIQUE(scope_type, scope_id, supplier_external_id, source_key)
```

**Linchpin:** `sku_mapping.pos_ingredient_id` holds onto a durable POS ID. `ingredient_cache` can be dropped entirely тАФ mappings will survive it.

## Invariants тЖТ merge-blocking tests
- `test_cache_rebuild_preserves_mappings` тАФ a drop + rebuild of `ingredient_cache` leaves `sku_mapping` intact and resolvable.
- `test_commit_blocked_when_pos_unavailable` тАФ POS unavailable at commit тЖТ the receipt is NOT written, lines go to review.
- `test_commit_blocked_when_pos_id_missing` тАФ `pos_ingredient_id` no longer exists in the POS тЖТ commit blocked.
- `test_commit_blocked_on_unit_mismatch` тАФ `unit` in the live POS тЙа `unit` at mapping time тЖТ commit blocked + mapping re-validation.
- `test_matching_resolves_subdivision_before_org` тАФ when both mappings exist, subdivision wins.
- `test_mapping_has_no_fk_to_cache` тАФ a schema test: `sku_mapping` has no FK to `ingredient_cache`.

## Sync strategy
For Customer Zero (a coffee-shop catalog of тЙИ hundreds of items): lazy TTL-refresh + a forced refresh when opening an invoice + re-validation at commit (which incidentally updates the affected cache rows). Delta-sync / webhooks тАФ **DEFER-016** (until a volume/frequency trigger). All syncing is behind the `ERPProvider` seam, one implementation (Esupl) until a real trigger for an alternative ([[ADR-009]]).

## Amendment 2026-07-08 тАФ two-context resolution model (implementation)
Out of the implementation a two-context model crystallized (it replaces the old "Phase 1 matching order" as a single list; see [[architecture]] тЖТ "SKU identity & the two-context resolver"):

- **Draft resolution (`prepare()`, tolerant):** builds the Esupl payload from the local catalog (numeric FKs). Hints (fuzzy layer-1, LLM `suggest-matches`, exact `ingredient_cache`) live ONLY here. `prepare()` does NOT touch `pos_ingredient_id`.
- **Commit resolution (`submit()` тЖТ `_resolve_commit_identity` тЖТ Phase 2, fail-closed):** durable `pos_ingredient_id` via `normalize_source_key(line.description)` from `sku_mapping`, priority subdivision тЖТ org, ONLY confirmed identity (`method=manual` OR `confirmed_by`). Cache/fuzzy/AI do NOT participate on the commit path. Then live validation; None/mismatch/unavailability тЖТ block+review.

**T2 (`esupl_item_id` vs `pos_ingredient_id`):** one Esupl entity in two representations тАФ `esupl_item_id` (int, a catalog copy for the payload) and the durable `pos_ingredient_id` (str, the identity anchor in `sku_mapping`/on the line). `pos_ingredient_id == str(esupl ingredient id)`.

**T5/VER-022:** `ingredient_cache` is draft-only, scope-aware; under [[DEC-0013]](A) it is NOT a commit tier тЖТ the former scope asymmetry "cache is org-only" is eliminated (commit authority comes only from `sku_mapping`).

**Open dependency:** VER-021 (durability) тАФ still a GATE, empirically NOT confirmed (see `scripts/ver021_durability_probe.py`).

## Consequences
- (+) The moat (`sku_mapping`) is resilient to catalog staleness: it holds onto a durable ID, not attributes.
- (+) Commit is protected by fail-closed; desync cannot lead to a wrong receipt write.
- (тИТ) Requires the `ERPProvider` READ seam (`list_ingredients`, `get_ingredient`) тАФ currently the integration is write-point only. тЖТ **ALIGN-014**.
- (тИТ) Tied to the stability of `pos_ingredient_id` in Esupl. If the ID regenerates on edit/merge тАФ the model reopens onto an alternative anchor. тЖТ **VER-021** (P0 linchpin).
