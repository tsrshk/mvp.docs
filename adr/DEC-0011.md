# DEC-0011 · POS как источник истины для идентичности ингредиента; mapping на durable POS ID; two-phase authority

**Status:** Accepted · **Date:** 2026-07-08 · **Codified in:** [[ADR-018]]
**Depends on:** VER-021 (стабильность `pos_ingredient_id`), ALIGN-014 (`ERPProvider` READ seam)
**Related:** DEC-0012 ([[ADR-019]], правило ключа `source_key`), [[DEC-0013]]
**Blocks:** F5–F10 (в части синхронизации ингредиентов)

## Контекст
POS (Esupl) держит каталог ингредиентов. LCOS распознаёт накладную и сопоставляет её строки с ингредиентами POS (SKU matching). Нужно решить: кто владеет данными ингредиента, где они хранятся, как избежать рассинхрона — при этом commit накладной является ответственной стадией и обязан подчиняться fail-closed.

Решение продиктовано разделением двух сущностей:
- **Master data ингредиента** (`name`, `unit`, `category`) — чужой актив. Владелец — POS. LCOS никогда не авторитетен над ними.
- **Mapping** (`source_key` → идентичность ингредиента) — актив LCOS. Накапливающийся moat. LCOS — единственный источник истины.

## Решение
1. **POS — единственный SoT идентичности и атрибутов ингредиента.** LCOS не создаёт и не редактирует master data ингредиента как авторитетную.
2. **LCOS владеет маппингом.** `sku_mapping` привязан к durable-идентификатору POS (`pos_ingredient_id`), НЕ к имени и НЕ к суррогатному PK локального кэша.
3. **Локальный `ingredient_cache` неавторитетен.** Он обслуживает только фазу draft-сопоставления и отображение. Его можно полностью сбросить и пересобрать без потери маппингов.
4. **Two-phase authority:**
   - **Phase 1 — Matching (draft; толерантно, дёшево):** резолюция через `ingredient_cache` в порядке `subdivision mapping → org mapping → fuzzy → AI`. Результат — гипотеза, не факт.
   - **Phase 2 — Invoice commit (ответственно; fail-closed):** перед записью прихода каждый разрешённый `pos_ingredient_id` валидируется живым запросом к POS. Недоступность POS / отсутствующий ID / расхождение `unit` → commit блокируется, строка уходит в review. Авторитет на этой стадии НЕ берётся из кэша.

**Почему кэш не нарушает «no fallbacks»:** fallback = подмена доверенного источника менее доверенным ради всё же завершить ответственное действие. Здесь ответственное действие (commit) остаётся fail-closed и всегда идёт в живой POS; кэш обслуживает только опциональный draft, где он явно помечен как гипотеза.

## Нормативные правила (для агента)

**MUST**
- `sku_mapping.pos_ingredient_id` MUST ссылаться на durable POS ID.
- Commit-путь MUST выполнять живую валидацию каждого `pos_ingredient_id` против POS перед записью прихода.
- Расхождение `unit` MUST блокировать commit (строка → review). См. *Amendment 2026-07-20*: реализовано как сверка живого POS-`unit` с `unit` строки накладной (OCR), а не «на момент mapping» — у `sku_mapping` нет колонки `unit`; отдельной ре-валидации маппинга нет (исход — status=rejected).
- `ingredient_cache` MUST быть полностью пересобираемым (drop + rebuild) без изменения `sku_mapping`.
- Резолюция маппинга MUST соблюдать приоритет: `subdivision` перед `org`.

**MUST NOT**
- `sku_mapping` MUST NOT иметь FK на `ingredient_cache.id` (или любой суррогатный PK кэша).
- Commit-путь MUST NOT читать авторитет (существование / `unit` ингредиента) из `ingredient_cache`.
- Расхождение `unit` MUST NOT молча обновлять кэш и продолжать commit.
- LCOS MUST NOT хранить master data ингредиента как авторитетный источник.
- Эта модель MUST NOT вводить зависимость от embedding (неиспользуемая колонка `pgvector` не должна затягиваться в matching; её судьба — отдельная задача очистки мёртвого кода → backlog **DEC-02**, `invoice_lines.sku_embedding` мёртв).

## Модель данных (набросок; финализируется в [[architecture]])

```
ingredient_cache            -- NOT authoritative: acceleration + display only
  id              PK (surrogate)      -- NEVER a reference for mapping
  scope_type      enum(org|subdivision)
  scope_id        FK(org_id|subdivision_id)
  pos_ingredient_id           -- durable POS ID (see VER-021)
  name, unit, category        -- snapshot from POS
  pos_version | content_hash  -- change detection on refresh
  is_active       bool                -- soft-delete if POS removed it
  synced_at       timestamptz
  UNIQUE(scope_type, scope_id, pos_ingredient_id)

sku_mapping                 -- authoritative: LCOS asset (moat)
  id              PK
  scope_type      enum(org|subdivision)
  scope_id        FK
  source_key      text                -- see DEC-0012 (normalization/key rule)
  supplier_external_id        -- added by DEC-0012 / ADR-019 (composite key)
  pos_ingredient_id           -- durable POS ID; NOT an FK to ingredient_cache.id
  method          enum(manual|fuzzy|ai)
  confidence      numeric
  confirmed_by, confirmed_at
  created_at, updated_at
  UNIQUE(scope_type, scope_id, supplier_external_id, source_key)
```

**Linchpin:** `sku_mapping.pos_ingredient_id` держится за durable POS ID. `ingredient_cache` можно полностью сбросить — маппинги его переживут.

## Инварианты → merge-blocking тесты
- `test_cache_rebuild_preserves_mappings` — drop + rebuild `ingredient_cache` оставляет `sku_mapping` целым и резолвимым.
- `test_commit_blocked_when_pos_unavailable` — POS недоступен при commit → приход НЕ записывается, строки уходят в review.
- `test_commit_blocked_when_pos_id_missing` — `pos_ingredient_id` больше не существует в POS → commit заблокирован.
- `test_commit_blocked_on_unit_mismatch` — `unit` в живом POS ≠ `unit` строки накладной → commit заблокирован (строка → review, status=rejected). См. *Amendment 2026-07-20*: сверка идёт против unit строки (OCR), у `sku_mapping` нет колонки `unit`; отдельной ре-валидации маппинга нет.
- `test_matching_resolves_subdivision_before_org` — когда оба маппинга существуют, побеждает subdivision.
- `test_mapping_has_no_fk_to_cache` — schema-тест: у `sku_mapping` нет FK на `ingredient_cache`.

## Стратегия синхронизации
Для Customer Zero (каталог кофейни в ≈ сотни позиций): lazy TTL-refresh + принудительный refresh при открытии накладной + ре-валидация при commit (которая заодно обновляет затронутые строки кэша). Delta-sync / webhooks — **DEFER-016** (до триггера по объёму/частоте). Вся синхронизация за seam `ERPProvider`, одна реализация (Esupl) до реального триггера для альтернативы ([[ADR-009]]).

## Amendment 2026-07-08 — two-context модель резолюции (реализация)
Из реализации выкристаллизовалась two-context модель (она заменяет старый «Phase 1 matching order» как единый список; см. [[architecture]] → «SKU identity & the two-context resolver»):

- **Draft resolution (`prepare()`, толерантно):** строит Esupl payload из локального каталога (числовые FK). Хинты (fuzzy layer-1, LLM `suggest-matches`, exact `ingredient_cache`) живут ТОЛЬКО здесь. `prepare()` НЕ трогает `pos_ingredient_id`.
- **Commit resolution (`submit()` → `_resolve_commit_identities` → Phase 2, fail-closed):** durable `pos_ingredient_id` через `normalize_source_key(line.description)` из `sku_mapping`, приоритет subdivision → org, ТОЛЬКО подтверждённая идентичность (`method=manual` OR `confirmed_by`). Cache/fuzzy/AI НЕ участвуют на commit-пути. Затем живая валидация; None/mismatch/недоступность → block+review.

**T2 (`esupl_item_id` vs `pos_ingredient_id`):** одна сущность Esupl в двух представлениях — `esupl_item_id` (int, копия каталога для payload) и durable `pos_ingredient_id` (str, якорь идентичности в `sku_mapping`/на строке). `pos_ingredient_id == str(esupl ingredient id)`.

**T5/VER-022:** `ingredient_cache` только draft, scope-aware; при [[DEC-0013]](A) он НЕ commit-tier → прежняя асимметрия scope «cache is org-only» устранена (авторитет commit исходит только из `sku_mapping`).

## Amendment 2026-07-20 — реализация unit-mismatch (drift-фикс)

Нормативный MUST выше описывал сверку `unit` «на момент mapping vs живой POS». По факту:
- `sku_mapping` НЕ несёт колонку `unit`, поэтому «unit на момент mapping» не хранится и такая сверка невозможна.
- `validate_ingredient_on_commit` сверяет живой POS-`unit` с `unit` СТРОКИ накладной (OCR), толерантно к пустым (блок только когда обе стороны заданы и различаются).
- При расхождении commit блокируется (status=rejected, строка → review); отдельной «ре-валидации маппинга» нет — это осознанное упрощение (строгая mapping-time-unit форма отложена, потребовала бы колонки `unit` в `sku_mapping`).

Тест `test_commit_blocked_on_unit_mismatch` отражает реализованное поведение (сверка против unit строки), не строгую mapping-time форму.

**Открытая зависимость:** VER-021 (durability) — всё ещё GATE, эмпирически НЕ подтверждён (см. `scripts/ver021_durability_probe.py`).

## Последствия
- (+) Moat (`sku_mapping`) устойчив к устареванию каталога: он держится за durable ID, а не за атрибуты.
- (+) Commit защищён fail-closed; рассинхрон не может привести к неверной записи прихода.
- (−) Требует seam `ERPProvider` READ (`list_ingredients`, `get_ingredient`) — сейчас интеграция только write-point. → **ALIGN-014**.
- (−) Завязан на стабильность `pos_ingredient_id` в Esupl. Если ID регенерируется при edit/merge — модель переоткрывается на альтернативный якорь. → **VER-021** (P0 linchpin).
