# ADR-018 · SKU-identity commit-gate: POS = SoT, mapping on a durable id, variant A (block until confirmation)

**Status:** accepted · **Date:** 2026-07-08 (кодификация [[DEC-0011]] + [[DEC-0013]]); вето варианта C — 2026-07-09

> Этот ADR **кодифицирует** два детальных решения. Полные записи: [[DEC-0011]] (POS как SoT + durable id + two-phase authority) и [[DEC-0013]] (commit требует подтверждённой идентичности — вариант A).

## Контекст
Кто владеет идентичностью ингредиента и как коммитить строку с exact-cache-match, но без подтверждённого `sku_mapping` — авто или блокировать.

## Решение
Кодифицирует **[[DEC-0011]]**: POS — единственный SoT идентичности/атрибутов ингредиента; `sku_mapping` привязан к durable `pos_ingredient_id`, **НЕ** к cache/суррогатному PK; two-context resolution — draft толерантно из локального каталога, commit fail-closed с живой валидацией против POS.

Плюс **[[DEC-0013]] вариант A**: commit разрешён ТОЛЬКО для commit-eligible маппинга (`method=manual` **ИЛИ** `confirmed_by IS NOT NULL`); cache exact / fuzzy / AI **НЕ** автокоммитятся и **НЕ** автосоздают маппинг. Moat (`sku_mapping`) растёт только через явное подтверждение человеком.

## Вето (2026-07-09, Ivan)
Предложение `TZ__STABILIZATION_2026-07-09` (D1 / S6) возродить **вариант C** — где exact-cache-match автосоздаёт `sku_mapping` (`method='cache_exact'`, `confirmed_by='system'`) и коммитит — **отклонено**. Причина: вариант C стоит ниже DEC-0013 по авторитету, откатывает инвариант fail-closed и ломает merge_gate тест `test_exact_cache_match_does_not_commit_and_creates_no_mapping`. Единственная точка интеграции резолюции — только `sku_mapping`; cache отсутствует на commit-пути.

## Последствия
- Каждый новый элемент требует единоразового ручного подтверждения (это и есть «review»).
- Moat маппингов переживает drop+rebuild кэша.
- VER-022 (асимметрия scope кэша) закрыт — cache вне commit-пути.
- **Открыто: VER-021** (durability `pos_ingredient_id`) — GATE, требует WRITE в песочницу Esupl → owner-run, не закрывается в read-only.
