# ADR-016 · Source of stock and consumption

**Status:** proposed (to be clarified in the browser during implementation; no live token available up front) · **Date:** 2026-07-03

## Context
Order planning (Phase 1, Э4b) requires ingredient stock levels and a consumption estimate. Whether Esupl returns remains/sales per warehouse has not been verified live (in the mirror `api/esupl/warehouse.md` the paths exist, but there are no response examples), and no read-only token is available up front — verification is possible only through a working application after the token has been saved in settings.

## Decision
At the start of Э3 — **variant C** (manual stock entry, `source='manual'`) as a guaranteed-working path, so the planner does not wait on a live check. In parallel, **A** (remains from Esupl, `source='esupl'`) is implemented and enabled after the contract is confirmed in the browser. **B** (receipts − consumption as a proxy) is an optional add-on in F4.4, not a blocker.

That is, the question "does Esupl return remains" is converted from a blocker into a gradual enablement.

## Consequences
F3.1 builds both `get_stock` (A) and manual entry (C); F3.3 always allows manual adjustment; the consumption path in F4.4 is optional; the Э3 kill trigger (>50% discrepancy on a live check) leaves the system on C without loss of function.
