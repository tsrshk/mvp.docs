# ADR-019 · DEC-0012: composite key for `sku_mapping` (supplier in the key)

**Status:** accepted · **Date:** 2026-07-09 · **Ratifies:** DEC-0012 (previously deferred)

## Context
The FE learning-loop learns mappings **per-supplier** (`provider::org::supplier::name`), whereas the backend `sku_mapping.source_key` was supplier-agnostic with `UNIQUE(scope_type, scope_id, source_key)`. When migrating the learning-loop into the backend (moat, [[DEC-0013]]), two different suppliers with the same raw line name ("Milk") map to DIFFERENT POS SKUs → collapse into one key → overwrite → wrong `pos_ingredient_id` at commit.

## Decision
The `sku_mapping` key becomes **composite**: `supplier_external_id` is added (durable Esupl supplier id, `''` = supplier-agnostic/legacy), `UNIQUE(scope_type, scope_id, supplier_external_id, source_key)`. Commit resolution (`_resolve_commit_identities`) filters by the invoice's `supplier_external_id` (at commit it is no longer None — `prep.ready` requires the supplier to be resolved). The subdivision → org priority is preserved.

## Consequences
- The learning-loop migrates into the backend without collisions between suppliers.
- Migration `0008_sku_mapping_supplier_key`.
- Merge_gate tests updated (mappings carry `supplier_external_id`), an anti-collision test added.
- Key normalization is owned by the backend (`normalize_source_key`) — the FE sends the raw `rawName` and reads mappings from the backend (eliminating the dual SSOT with the FE `canonicalText`).
