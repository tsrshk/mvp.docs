# ADR-019 · DEC-0012: composite key for `sku_mapping` (supplier in the key)

**Status:** accepted · **Date:** 2026-07-09 · **Ratifies:** DEC-0012 (ранее отложенное)

## Контекст
FE learning-loop обучает маппинги **по-поставщикам** (`provider::org::supplier::name`), тогда как бэкендовый `sku_mapping.source_key` был supplier-agnostic с `UNIQUE(scope_type, scope_id, source_key)`. При миграции learning-loop в бэкенд (moat, [[DEC-0013]]) два разных поставщика с одинаковым сырым именем строки («Milk») мапятся на РАЗНЫЕ POS SKU → схлопывание в один ключ → перезапись → неверный `pos_ingredient_id` при commit.

## Решение
Ключ `sku_mapping` становится **composite**: добавляется `supplier_external_id` (durable Esupl supplier id, `''` = supplier-agnostic/legacy), `UNIQUE(scope_type, scope_id, supplier_external_id, source_key)`. Commit-резолюция (`_resolve_commit_identities`) фильтрует по `supplier_external_id` накладной (на commit он уже не None — `prep.ready` требует разрешённого поставщика). Приоритет subdivision → org сохраняется.

## Последствия
- Learning-loop мигрирует в бэкенд без коллизий между поставщиками.
- Миграция `0008_sku_mapping_supplier_key`.
- Merge_gate тесты обновлены (маппинги несут `supplier_external_id`), добавлен анти-коллизионный тест.
- Нормализация ключа принадлежит бэкенду (`normalize_source_key`) — FE отправляет сырой `rawName` и читает маппинги из бэкенда (устраняя двойной SSOT с FE `canonicalText`).
