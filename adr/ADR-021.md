# ADR-021 · Локальный SSOT поставщика; POS только для SKU-связей; единый ряд цен supplier_prices с source-дискриминатором

**Status:** accepted · **Date:** 2026-07-13

## Контекст
Между документацией и кодом накопилось расхождение по поводу того, кто является источником истины для справочника поставщиков:
- `entities/suppliers.md` описывает `suppliers` как «зеркало Esupl following + локальную карточку».
- Код же трактует справочник иначе: `GET /suppliers` и `/suppliers/search` — живой pass-through из Esupl following (`routes/suppliers.py:42-73`) по решению [[DEC-0011]] «POS = SSOT», а локальная таблица `suppliers` помечена в коде как «больше НЕ источник справочника».

Одновременно владелец зафиксировал требование: из POS берутся ТОЛЬКО связи для SKU, а сам поставщик — локальная сущность. Появление загружаемых прайс-листов и ассортимента (фичи F72-F75, эпик [[LCOS-E4-suppliers]]) усиливает это требование: прайсы, версии, freshness и проекция цен привязываются к локальному поставщику, поэтому владельцем данных о поставщике должна быть локальная система, а не эфемерный pass-through из POS.

## Решение
1. Справочник и карточка поставщика — **локальный SSOT**. `GET /suppliers` (list) и `/suppliers/search` читают локальную таблицу `suppliers`. Esupl following используется ТОЛЬКО для matching и для durable `supplier_external_id` в `sku_mapping`. `POST /suppliers/sync` остаётся как обогащение локальной карточки данными из POS (upsert), а не как источник справочника.
2. **Частичный реверс [[DEC-0011]]**: принцип «POS = SSOT» СОХРАНЯЕТСЯ для каталога ИНГРЕДИЕНТОВ, но НЕ применяется к справочнику ПОСТАВЩИКОВ. Реверсируется ровно эта часть DEC-0011 и только для поставщиков.
3. **Единый временной ряд цен** `supplier_prices` (фича [[LCOS-F20-price-history]]) расширяется дискриминатором `source ∈ {invoice, manual, price_list_upload}`. Разрешённые позиции прайса ([[price_list_line]]) проецируются в тот же ряд (`source=price_list_upload`), идемпотентно по `source_price_list_line_id`. Так сигнал изменения цены ([[LCOS-F21-price-change-signal]]) и сравнение поставщиков ([[LCOS-F75-supplier-price-analytics]]; legacy PHASE_F4_SUPPLIER_COMPARE) работают поверх всех источников однородно.

## Последствия
- `GET /suppliers` и `/suppliers/search` переводятся с pass-through на локальное чтение из `suppliers` — это миграция поведения, требует обновления и покрытия тестом.
- `supplier_external_id` становится опциональной durable-ссылкой на POS (для matching и `sku_mapping`), а не обязательным ключом идентичности поставщика.
- Локальная карточка (`POST /suppliers` create_card, `PATCH /suppliers/{id}` update_card, `GET /suppliers/{id}` по int id, `POST /suppliers/sync`, `POST /suppliers/match`) остаётся и становится каноническим владельцем данных.
- Аналитика и сигналы ([[LCOS-F21-price-change-signal]], [[LCOS-F75-supplier-price-analytics]]) работают поверх единого `supplier_prices` без ветвления по источнику.
- `entities/suppliers.md` приводится в соответствие: локальный SSOT + опциональная POS-ссылка.

## Отложено
- Запись/создание поставщика в POS (обратный sync) — остаётся за флагом `ERP_WRITE_ENABLED` (по умолчанию off). Требует отдельного решения о разрешении конфликтов и идемпотентности обратной записи.
