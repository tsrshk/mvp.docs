# ADR-023 · RBAC 3-ролевая модель + org-level membership + user-management API

**Status:** accepted · **Date:** 2026-07-21

## Контекст
Phase 1 явно объявляла RBAC-матрицу non-goal: было два уровня — `superadmin` (флаг
`users.is_superadmin`) и `admin` (единственное значение enum `Role`, на уровне
подразделения через [[memberships]]). Управление организациями/подразделениями/юзерами
жило только в плоскости SQLAdmin operator ([[ADR-007]]), без орг-скоупинга. Владелец
запросил полноценное управление структурой и пользователями из приложения: CRUD орг,
подразделений, юзеров; назначение ролей; одноразовые пароли; 3 роли с иерархией
«сверху вниз». Это осознанно расширяет прежний non-goal.

## Решение
1. **Три роли.**
   - `superadmin` — остаётся **платформенным флагом** `users.is_superadmin` (не значение
     enum, не орг-скоупная роль). Режим Бога над всеми организациями. Назначать
     superadmin может только другой superadmin.
   - `admin` — значение enum `Role`. Управляет своей организацией и всеми её
     подразделениями (сверху вниз): CRUD подразделений/юзеров, назначение ролей
     admin/manager вниз, настройки орг. **НЕ** создаёт новые организации и не выдаёт
     superadmin.
   - `manager` (**новое** значение enum `Role`) — «Управляющий»: полный доступ к
     прикладным фичам (накладные, поставки, поставщики, каталог, заявки), но БЕЗ
     управления сотрудниками и настройками организации.
2. **Org-level membership.** `memberships` получает `organization_id` (NOT NULL) и
   NULLable `subdivision_id`: `subdivision_id IS NULL` ⇒ роль действует на всю
   организацию. Так admin/manager назначаются на всю орг (в т.ч. орг без
   подразделений) либо на конкретное подразделение. Инвариант «subdivision принадлежит
   organization_id» проверяется сервисом. Partial-unique `(user, org) WHERE
   subdivision_id IS NULL`; прежний unique `(user, subdivision)` сохранён. Миграция
   `0024` (бэкфилл org_id из подразделения). `refresh_sessions.active_organization_id`
   хранит org-контекст явно (org-admin может не иметь активного подразделения).
3. **Единый SSOT прав — `app/auth/rbac.py`.** Предикаты по клеймам (`has_app_access`,
   `is_org_manager_role`) + async-авторизация против реальных membership
   (`assert_can_manage_org` = superadmin ИЛИ любое admin-членство в целевой орг). Прежний
   разнобой (ad-hoc `_authorize` только в pos-config) заменён общими хелперами.
   Прикладные write-роуты гейтятся `require_app_write` (admin|manager|superadmin);
   управление — `assert_can_manage_org`/`require_superadmin`.
4. **Роли из API.** `GET /roles` отдаёт назначаемые роли с метаданными (scope,
   platform, app_access, can_manage); superadmin дополнительно видит платформенную
   роль. Фронт не хардкодит роли (задел под расширение).
5. **Управление через REST (app-plane, JWT).** `organizations` (+ subdivisions) и
   `users` (+ memberships, reset-password) роутеры. Создание/удаление орг — только
   superadmin; остальное — admin своей орг-поддеревом или superadmin. Это **не отменяет**
   плоскость SQLAdmin operator ([[ADR-007]]) — две плоскости по-прежнему не смешиваются;
   REST-управление идёт по app-JWT и RBAC.
6. **Одноразовый пароль.** Пароль, заданный админом при создании/сбросе, помечает
   `must_change_password=True`; пользователь входит по нему и обязан сменить (существующий
   `change-password` флоу). Сгенерированный сервером пароль показывается администратору
   один раз. Reset отзывает все сессии пользователя.
7. **Политика паролей** (конфиг-driven, сервер — авторитет): длина ≥ 8, хотя бы одна
   буква и одна цифра, без крайних пробелов, новый ≠ текущему. Логин — email
   (структурная валидация; уникальность/регистронезависимость — БД).

## Последствия
- Email-рассылки пока нет: флоу спроектирован так, что «сброс пароля письмом»
  подключается позже без изменения контракта (одноразовый пароль → форс смены).
- `manager` не имеет доступа к роутам управления (403); чтение прикладных данных и
  запись прикладных фич — разрешены.
- Роль в JWT (`role`) отражает действующую роль активного контекста; управление же
  авторизуется по реальным membership в БД (admin может администрировать орг, отличную
  от активной). 404-для-superadmin в `switch-context` сохранён.
- FE: «глазик» на паролях, email-логин, экран смены пароля дожидается свежего
  `/auth/me` перед навигацией (устранена гонка stale-кэша AuthGuard); раздел
  «Управление» виден только admin/superadmin (гейт `canManage`, сервер — авторитет).
- Тесты: BE pytest (RBAC-матрица, скоуп, one-time-password, `/roles`), FE vitest
  (политика/`can()`), Playwright e2e (создание юзера → OTP → форс-смена, глазик).
- Документы ролей ([[superadmin]]/[[admin]]/[[manager]]) обновлены; прежний «RBAC —
  non-goal» снят этим ADR.
