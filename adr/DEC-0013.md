# DEC-0013 · Commit requires CONFIRMED SKU identity (variant A)

**Status:** Accepted · **Date:** 2026-07-08 · **Codified in:** [[ADR-018]]
**Decided by:** Ivan (veto window of TZ `TZ__DEC-0011_merge-gate_2026-07-08.md`)
**Depends on:** [[DEC-0011]] (POS as SoT of identity; two-phase authority)
**Relates to:** VER-021 (durability, GATE, open), VER-022 (cache scope, closed by this decision)

## Context
A line without a confirmed `sku_mapping` but with an exact match in `ingredient_cache` (Tier 3) — commit it automatically or not? The TZ proposed **variant C** by default (auto-resolve exact-match + auto-create mapping). Open question §3.4 of the patch.

## Decision — variant A (block until manual confirmation)
**Commit is allowed ONLY for a confirmed identity.** A line without a commit-eligible `sku_mapping` (`method=manual` **OR** `confirmed_by IS NOT NULL`) at commit → **block + review**, not a write. An exact `ingredient_cache` match and fuzzy/AI are draft-phase hints; they are **not** auto-committed and do **not** auto-create a mapping. The moat grows through explicit human confirmation (creating a manual mapping or confirming a hint → `confirmed_by`).

This is stricter than the patch's recommendation (§3.4) and stricter than variant C: the most fail-closed reading of the principle "the invoice is the responsible stage; the AI shows, the human decides."

### Variants considered
| Variant | UX friction | Moat | Fail-closed | Choice |
|---|---|---|---|---|
| A. Block until manual confirmation | high (on every new item) | yes (via confirmation) | yes | **✅ accepted** |
| B. Auto-resolve exact without mapping | none | no | yes | rejected |
| C. Auto-resolve exact + auto-mapping | none | yes | yes | rejected (was the TZ default) |

> **Veto of variant C reaffirmed 2026-07-09** ([[ADR-018]]): the `TZ__STABILIZATION_2026-07-09` (D1/S6) attempt to revive C was rejected — it breaks the merge_gate test `test_exact_cache_match_does_not_commit_and_creates_no_mapping`.

## Normative rules

**MUST**
- Commit resolution: `normalize_source_key(line.description)` → `sku_mapping`, priority subdivision → org.
- Commit-eligible = `method == manual` OR `confirmed_by IS NOT NULL`.
- No commit-eligible mapping → block+review (status `rejected`), the receipt is NOT written, validation is NOT skipped.

**MUST NOT**
- Cache exact-match / fuzzy / AI MUST NOT auto-commit and MUST NOT auto-create `sku_mapping`.
- The commit path MUST NOT read `ingredient_cache` as a source of authority (consistent with [[DEC-0011]]).

## Consequences
- (+) No unconfirmed guess reaches the POS. No rollback needed — garbage is not written.
- (+) VER-022 closed: the cache is off the commit path → the scope asymmetry "cache is org-only" is impossible.
- (−) Every new item requires a one-time confirmation (expected friction; this is the "review").
- Implementation: `invoice_service._resolve_commit_identity`; tests `test_dec0013_commit_gate.py` (`test_exact_cache_match_does_not_commit_and_creates_no_mapping`, `test_fuzzy_mapping_without_confirmation_blocks_commit`, `test_confirmed_mapping_commits`) — merge_gate.
