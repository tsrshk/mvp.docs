# ADR-020 · Moat accumulation channel: client-side `POST /ingredients/mappings` in `onSend`; persist-then-commit independent of the invoice's fate; FE `save()` removed

**Status:** accepted · **Date:** 2026-07-09 (fix of DISCREPANCY Bucket 1 per `TZ__VERIFY_APP_OVERVIEW`)

> The newest ADR in the registry.

## Context
The APP_OVERVIEW verification (V1/V2) surfaced two things:
- (a) The decision about the **channel** for moat accumulation and about persist surviving a reject lived only in the descriptive APP_OVERVIEW (trust tier 3) with a **dangling reference** to [[ADR-013]], which is actually about photo-first supplier resolution — there was no normative record.
- (b) The FE `IngredientSKUFactory.save()` was presented (in past specs) as the mapping accumulation path, but in the code it is a self-documented **no-op with zero callers** — dead code; the real write happens only via `POST /ingredients/mappings`.

## Decision
1. **The single runtime channel** for moat accumulation is the client-side `POST /ingredients/mappings`, which the FE calls in the `onSend` handler (`widgets/invoice-workbench`, via `entities/invoice/lib/backendMappings.persistLineMapping`) **before** the `sendInvoice` mutation (`method='manual'`, `confirmed_by` = the authenticated user). The BE submit endpoint does **not write** `sku_mapping` — it only reads it during commit resolution (`_resolve_commit_identities`).
2. **Persist happens BEFORE submit and in a separate transaction** → a confirmed mapping **survives an invoice reject** by design. Rationale: commit fail-closed resolves identity only from `sku_mapping`; if persist depended on the invoice's fate (or ran after a throwing send), a fail-closed reject of the first invoice would block the moat's initialization.
3. FE `IngredientSKUFactory.save()` is **removed** as dead (along with the `save` member of the `SKUFactory` interface). `SupplierSKUFactory.save()` is out of scope for this decision.

## Consequences
Persist-independence is locked by tests — T1 (BE: submit→reject→a `sku_mapping` row is present) and T2 (FE: `onSend` calls `persistLineMapping` and awaits it **before** `sendInvoice`). APP_OVERVIEW §8 references this ADR; the dangling reference to `ADR-013` from §8 has been removed. The moat write channel is single (SSOT); the FE factory does not participate in persistence.
