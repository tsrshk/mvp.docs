# Можем ли мы распознавать фото накладных БЕЗ ИИ?
### Архитектурная памятка по вариантам (анализ, НЕ план разработки)

---

## 1. Краткий ответ (TL;DR)

Полностью отказаться от ИИ на нашем основном сценарии — **фото бумажной ТТН (кириллица, печать вперемешку с печатями и рукописными полями, снято под углом и при разном свете) — реалистично НЕЛЬЗЯ** без падения качества ниже порога «исправлять выгоднее, чем набивать руками». Главная и единственная по-настоящему «жёсткая» зависимость от ИИ — это **зрительное распознавание + понимание структуры таблицы** (какая ячейка к какой колонке относится); классические OCR-движки в браузере на кривом телефонном фото кириллицы это делают плохо и шифруют цифры в qty/price/sum. **При этом МИНИМИЗИРОВАТЬ ИИ — реалистично, и частично уже сделано:** сопоставление SKU и резолв поставщика уже работают без генеративного ИИ (локальный fuzzy), арифметика и нормализация чисел детерминированы. Дополнительно можно убрать ИИ из ещё нескольких мест (штрихкод даёт серию/номер ТТН точно, текстовый LLM дешевле зрительного) и оставить дорогой зрительный вызов только как fallback по низкой уверенности. **Вывод: цель — не «без ИИ», а «ИИ только там, где без него никак, и самой дешёвой моделью».**

---

## 2. Где сейчас используется ИИ

Во всём кодовом базисе ровно **два генеративных call-site**, оба идут через `shared/llm/vendors` (Gemini / Claude), temperature 0; у обоих есть mock-провайдеры:

| # | Вызов | Что делает | Насколько критичен |
|---|---|---|---|
| **Вызов 1** | **Зрительный OCR** `OcrProvider.extractInvoice()` | До ~3 страниц-картинок → структурированный `OcrResult`: header `{supplierName, invoiceNumber, waybillSeries, waybillNumber, date, currency, total}` + `lines[]` `{rawName, qty, unit, unitPrice, sum, confidence, bbox}` | **Единственная жёсткая зависимость.** Это и есть основная ценность |
| **Вызов 2** | **Семантический матч SKU** `MatchProvider.suggestMatches()` (слой 2) | Только остаток нераспознанных строк + сокращённый список кандидатов → `{lineId, skuId, confidence, reason}`, применяется при confidence ≥ 0.6 как неподтверждённые подсказки | **Уже опционален.** Чистый «апгрейд» поверх локального слоя |

### Что УЖЕ без ИИ или уже гибрид (важно подчеркнуть)

- **Матч SKU, слой 1** — локальный fuzzy (Jaccard token-set + trigram Dice + смещение по поставщику), бесплатно/мгновенно/оффлайн. ИИ эскалируется только на остаток. (`shared/match/fuzzy.ts`)
- **Mock-провайдер матча** — это просто `rankSkus`, доказывает, что весь flow подсказок работает при нуле LLM.
- **Резолв поставщика** — полностью без ИИ, `bestNameMatch` по тому же fuzzy-скореру после OCR.
- **Арифметическая проверка строки** (qty×price vs sum) — чистая детерминированная арифметика (`selectors.ts`).
- **Пре-фильтр кандидатов для ИИ** — каталог сужается `rankSkus` до того, как модель его увидит (промпт меньше).
- **Препроцессинг изображения** — EXIF-ориентация + даунскейл до 1568px + один JPEG-проход, без ИИ.
- **Весь детерминированный пост-процессинг** за границей провайдера: `parse.ts` (нормализация чисел `"1 350,50"`→`1350.5`, перенос 7-значного номера в waybill, отброс строк ИТОГО/НДС/ВСЕГО), `rules.ts` (форматы серии/номера ТТН), подсветка низкой уверенности.

**Ключевой архитектурный факт:** любой альтернативный распознаватель, умеющий выдать форму `OcrResult`, переиспользует весь этот конвейер БЕЗ изменений — он сидит *за* границей провайдера (`OcrProvider`), а не внутри неё.

**Честная оговорка про страховку уверенности.** Существующая сеть LOW_CONFIDENCE настолько хороша, насколько честны self-reported оценки провайдера. Порог `LOW_CONFIDENCE = 0.9`, а `parse.ts` при отсутствии confidence **подставляет дефолт 0.9** — то есть ровно порог; строгое сравнение `<` означает, что строка без явной оценки трактуется как *не* low-confidence. Иными словами, молчаливо-самоуверенный вывод уже сегодня может проскочить мимо подсветки. Это важно: сеть ловит ошибки только тогда, когда движок сам сообщает низкий скор.

## 3. Что принципиально требует «интеллекта», а что — чистая механика

### Жёстко зависит от зрительного интеллекта (заменителя на правилах в браузере нет)

- **Чтение названий/чисел с перекошенных, заштампованных, рукописных, неравномерно освещённых фото** пред-печатных кириллических бланков ТТН.
- **Понимание структуры таблицы** — привязка каждой напечатанной ячейки к её логической колонке (name / qty / unit / unitPrice / sum) на нерегулярных, частично рукописных таблицах. **Это и есть ядро ценности.** Именно здесь классический OCR ломается: на кривом фото колонки «съезжают», и целая колонка чисел уходит не в то поле.
- **Различение номера ТТН и номера счёта на визуальном уровне** (тот самый «7-значный номер у штрихкода»). *Механическая страховка ниже по конвейеру есть (перенос в `parse.ts`), но первичное «который из них который» — визуальное.*
- **bbox-локализация** для оверлея (best-effort, никогда не доверяем — `parse.ts` дропает негодные).
- **Семантический матч SKU** аббревиатур/синонимов/бренд-vs-дженерик/транслита — но это уже за-гейчено, fuzzy-слой покрывает буквальные случаи.

### Механика / правила — уже делается БЕЗ ИИ или может делаться

- Нормализация чисел `"1 350,50"`→`1350.5` (`parse.ts`).
- Отброс ИТОГО / НДС / ВСЕГО (`rules.ts` + `parse.ts`).
- Перенос чисто-7-значного номера в `waybillNumber`.
- Валидация формата серии (2 кир. буквы) / номера (7 цифр) ТТН.
- Резолв поставщика, SKU-слой-1, арифметика, препроцессинг.
- **Штрихкод Code-128 на стандартном бланке ТТН РБ кодирует серию+номер ТТН** — это можно взять *точно*, а не угадывать OCR-ом. Чистый выигрыш (на других/электронных формах штрихкода может не быть).

> **Суть в одной фразе:** механику мы и так умеем без ИИ; неустранимое — это «прочитать кривое фото» + «понять, какая ячейка в какой колонке».

## 4. Варианты решения

### Вариант 1 — Полностью локальный: Tesseract.js / PaddleOCR-wasm + OpenCV.js, ноль LLM

- **Суть.** OpenCV.js — deskew/бинаризация/кроп + детекция линий таблицы (Hough/морфология); WASM-OCR (Tesseract `rus` или сильнее — PaddleOCR-ONNX) читает текст по ячейкам; наш код строит колонки→поля, режет строки, дропает ИТОГО/НДС, достаёт серию/номер. Движок даёт только текст+боксы.
- **Сколько ИИ убирает.** Весь генеративный ИИ из OCR (остаётся классический ML внутри движка, без multimodal-reasoning). Вся семантика («это цена, а это сумма», «7 цифр у штрихкода = номер ТТН», «ИТОГО — не товар») переезжает в крупный слой геометрических эвристик, которые мы строим и вечно поддерживаем.
- **Качество (оценки приблизительные, НЕ замерены на наших ТТН).** Чистый электронный путь (скрин/PDF/распечатка, ровная разлинованная таблица): по общему опыту приемлемо (Tesseract порядка 95–96% на символ, PaddleOCR обычно лучше). Но 25-символьное название при 96%/символ — лишь ~36% шанс быть полностью верным (арифметика верна), и одна неверная цифра молча портит qty/price/sum. **Фото бумажной ТТН под углом: ориентировочно ~75–80% на символ out-of-box, только после сильного препроцессинга ~90%+; путаница 0/О, 8/В, 1/I/l, 5/Б рушит арифметику. Рукопись/штампы — фактически непригодно (нет handwriting-модели).** Цифры выше — анекдотические диапазоны из литературы, а не бенчмарк на нашем наборе документов. Восстановление таблицы по линиям OpenCV надёжно только на полностью разлинованных малоскошенных таблицах.
- **Плюсы.** Полностью оффлайн и бесплатно за страницу; browser-direct без бэкенда/CORS/ключей; детерминизм и приватность (фото не покидает устройство); силён на чистом электронном пути; ложится за `OcrProvider`.
- **Минусы.** Проваливает рукопись/штампы и кривые фото — наш основной вход; огромный хрупкий слой структурных эвристик на нас навсегда; нет вменяемой per-line confidence (ослабляет существующую страховку LOW_CONFIDENCE — а она и так слаба, см. §2); тяжёлый первый загруз (десятки МБ моделей) и медленный/батарея/память инференс на телефонах (секунды–десятки секунд/стр.); threads требуют COOP/COEP cross-origin isolation, которых у статик-хоста может не быть.
- **Оффлайн.** Полностью (сильнейшая сторона).
- **Стоимость.** $0 за страницу; вместо денег — крупный one-time инжиниринг + поддержка + десятки МБ трафика на первом запуске.
- **Архитектура.** Новый класс за `OcrProvider` + один case в `factory.ts`, эмитит в `parseOcrResult`; но `rules.ts`/`parse.ts` придётся радикально расширить (вся структура).
- **Вердикт.** Жизнеспособно **как fallback** (лучше PaddleOCR-wasm, не Tesseract) — оффлайн/без-ключа провайдер и для чистого электронного пути. **Не годится как единственная замена** на фото кириллических ТТН.

### Вариант 1b — VLM на устройстве (WebGPU / transformers.js, малая квантованная зрительная модель)

- **Суть.** Запустить небольшую квантованную мультимодальную модель локально через WebGPU (transformers.js или аналог) — оффлайн-аналог зрительного LLM: лучше Tesseract по скосу/структуре, без ключа и без облака.
- **Сколько ИИ убирает.** Технически ИИ остаётся (это VLM), но уходит облако/ключ/стоимость за страницу; концептуально это «зрительный LLM, но локальный».
- **Качество.** Потенциально между Tesseract и облачным зрительным LLM, но на сегодня модели достаточного качества для кириллических рукописно-штампованных ТТН слишком тяжелы для телефона (объём весов, VRAM/память, скорость через WebGPU на мобильных, незрелость рантайма). Замеров на нашем кейсе нет.
- **Вердикт.** **Отклоняем сейчас — по весу и зрелости, не по идее.** Назвать явно, чтобы вопрос не всплыл на ревью; пересмотреть, когда мобильный WebGPU-инференс мультимодалок созреет.

### Вариант 2 — Специализированные облачные document-AI OCR (НЕ chat-LLM): Google Cloud Vision / Azure Document Intelligence / AWS Textract

- **Суть.** Специализированные ML-сервисы OCR (не чат-модели). Для нашего кейса они расходятся резко.
- **Сколько ИИ убирает.** Снижает зависимость от генеративного зрительного LLM на стадии OCR. Но: **Google Vision** = чистый OCR (текст+боксы, отличная кириллица), **без полей счёта и без структуры таблицы** → всё равно нужен LLM/тяжёлый парсер ниже, т.е. ИИ толком не убирается. **Azure prebuilt-invoice** = единственный, чья форма вывода (типизированные KV-поля + line items + таблицы) совпадает с нашей схемой; по заявлениям Microsoft в 2025 добавлена поддержка русского — *но это утверждение нужно проверить по докам перед опорой на него* — таксономия западная (VendorName/InvoiceTotal), нет серии/номера ТТН и белорусской семантики колонок → ремап + всё равно `rules.ts`. **AWS Textract дисквалифицирован — нет кириллицы вообще.**
- **Качество.** Google: великолепный сырой OCR кириллицы даже на кривых фото, но end-to-end упирается во вторую стадию структурирования. Azure: хорошо на чистых электронных накладных, слабо и непредсказуемо на бумажном бланке ТТН со штампами/рукописью/углами (особенно серия 2 буквы + 7 цифр у штрихкода). Рукопись — слабо у всех. **Ни один не превосходит наш текущий зрительный LLM на реальной задаче «фото → корректный JSON ТТН за один проход».**
- **Плюсы.** Дёшево и предсказуемо за страницу (Google/Azure-Read порядка $1.5/1k стр.); боксы Google могут усилить наши bbox/confidence; Azure — единственное совпадение формы вывода.
- **Минусы.** AWS — нет кириллицы. **Google browser-direct возможен только с НЕограниченным ключом:** Cloud Vision API **отклоняет ключи с HTTP-referrer-ограничением** («API keys with referer restrictions cannot be used with this API»), поэтому из браузера его можно дёрнуть лишь незалоченным ключом — а это значит, что любой утёкший ключ используется откуда угодно и тарифицируется на нас, без referrer-замка. С нашим требованием безопасности это плохой профиль. **Azure НЕ browser-direct** (нет CORS/PNA-заголовков, ключ полностью открыт без referrer-lockdown, async submit-then-poll — фактически нужен бэкенд, которого у нас нет). AWS требует SigV4 с секретными кредами → нельзя клиентски. **Никто не работает оффлайн.** Google не даёт поля/структуру → ИИ, который мы хотим убрать, возвращается ниже по стеку. Prebuilt-модели обучены на западных счетах. Ещё один вендор/ключ для пользователя.
- **Оффлайн.** Нет (облако, конфликт с ценностью PWA).
- **Стоимость.** Копейки/мес на нашем объёме; не решающий фактор.
- **Архитектура.** Google ляжет за `OcrProvider` как сырой OCR-фидер (но требует дострукт. слой). Azure практически требует бэкенд.
- **Вердикт.** Как замена всей тройки — **не годится.** Google Vision — слабый fallback: отличная кириллица и дёшево, но (а) browser-direct только незалоченным ключом — плохой профиль безопасности, (б) не уменьшает ИИ (нужен структуризатор ниже). Azure — лучшее совпадение формы, но не browser-direct (нужен бэкенд). AWS — не годится.

### Вариант 3 — OCR → маленький ТЕКСТОВЫЙ LLM (пиксели→текст классикой, текст→JSON дешёвой текст-моделью)

- **Суть.** (1) Tesseract.js (кириллица) в браузере → сырой текст + боксы слов; (2) отправляем **только ТЕКСТ** (без картинок) в дешёвую текст-модель (напр. Gemini 2.5 Flash-Lite) → структурируем в `OcrResult`.
- **Сколько ИИ убирает.** Понижает ИИ с зрительного до дешёвого текстового для структурирования таблицы. Зрительный LLM остаётся как gated-fallback.
- **Качество.** Электронные накладные: сильно, сравнимо с зрительным LLM (иногда лучше на длинных таблицах — нет бюджета токенов на картинку). **Фото бумажной ТТН: заметно хуже** — у Tesseract нет коррекции перспективы/кривого текста, рукопись/скос рушатся, привязка колонок хрупкая; текст-LLM не восстановит символы, которых OCR не увидел (garbage-in/garbage-out). Нет доверенной per-cell confidence.
- **Плюсы.** **Существенно дешевле** (текст-токены вместо image-токенов — картинки сейчас основной объём; + дешёвый tier); быстрее, меньше запросы (хорошо для мобильного/слабой сети); приватнее (пиксели не уходят на OCR-стадии); зеркалит проверенный гибрид `shared/match`.
- **Минусы.** Регрессия качества на самом частом и тяжёлом входе (фото бумажной ТТН); теряется доверенная per-line confidence/bbox; двухстадийный конвейер = больше движущихся частей, deskew становится обязательным; первый загруз ~5МБ WASM + кириллическая traineddata + CPU-тяжёлый проход на телефоне (нужен Web Worker + кэш).
- **Оффлайн.** Частично — OCR оффлайн (даёт сырой текст), структурирование текст-LLM требует сети (или тяжёлый on-device WebLLM). Строго лучше, чем сегодня (сегодня OCR полностью онлайн).
- **Стоимость.** Дешевле текущего по двум осям (Tesseract бесплатен; убираем image-токены + дешёвый tier).
- **Архитектура.** Почти нулевой churn: новый `TesseractTextOcrProvider` за `OcrProvider`, переиспользует `parse.ts`/`rules.ts`/`shared/llm` текстовый транспорт; call-site, UI, `rules.ts` не трогаем.
- **Вердикт.** **Fallback** — отлично для чистых электронных накладных; на грязных фото оставить зрительный LLM.

### Вариант 4 — Гибрид с эскалацией по уверенности (минимизация, а не удаление)

- **Суть.** Тот же паттерн, что уже работает в `shared/match`: сначала бесплатный/дешёвый локальный слой (Вариант 1/3 + штрихкод + якоря), а дорогой **зрительный LLM срабатывает только на остатке** — на страницах/строках, которые локальный слой не прошёл с уверенностью.
- **Сколько ИИ убирает.** На чистых документах, прошедших гейт уверенно — ИИ из OCR убран полностью; на тяжёлых фото зрительный LLM всё ещё работает. Blended-стоимость падает — дорогой вызов на меньшинстве страниц.
- **Качество.** Лучшее из обоих: чистый путь — локально/дёшево; тяжёлый путь — текущее качество зрительного LLM сохраняется.
- **⚠ Главный риск — и он недооценён.** Вся экономия Варианта 4 держится на надёжности гейта «локальный слой справился?». Проблема в том, что **самый частый режим отказа OCR — это сдвиг целой колонки**, который остаётся *внутренне самосогласованным*: если qty, price и sum съехали синхронно, проверка `qty×price==sum` и `Σstrok==total` проходит на неверных данных. То есть наш главный детерминированный сигнал гейта **не ловит именно тот класс ошибок, который чаще всего и случается.** Без дополнительного независимого сигнала уверенности (например, per-cell confidence от движка, которой у Tesseract нет) гейт будет молча пропускать неверные-но-арифметически-согласованные таблицы мимо fallback. Это решает, действительно ли Вариант 4 дешевле — или просто тихо хуже. До прототипа с честной валидацией гейта на реальных ТТН это открытый вопрос, а не данность.
- **Плюсы.** Не оголяет качество на тяжёлых случаях; снижает стоимость и зависимость от облака; знакомый низкорисковый паттерн (уже в коде); композируется послойно.
- **Минусы.** Логика гейтинга = инжиниринг и тесты; **нужен честный сигнал успеха, которого у локального OCR может не быть** (см. риск выше); всё ещё держим зрительный LLM (значит, полностью от ИИ не уходим — по дизайну).
- **Оффлайн.** Частично — локальный слой оффлайн, fallback онлайн.
- **Стоимость.** Потенциально ниже текущей (дорогой вызов реже) — *при условии, что гейт надёжен*; иначе экономия мнимая.
- **Архитектура.** Ложится идеально: гейт между провайдерами, форма `OcrResult` неизменна; повторяет уже принятый в `shared/match` паттерн.
- **Вердикт.** **Это и есть «минимизировать ИИ» в чистом виде** — рекомендуемая целевая форма, **но её жизнеспособность напрямую упирается в надёжность гейта** (риск выше). Прототип гейта на реальных ТТН — обязательное условие, прежде чем считать экономию реальной.

---

### Вариант 5 — Шаблоны/якоря + штрихкод для фиксированных полей (точечное удаление ИИ)

- **Суть.** Не трогая таблицу товаров: (а) декодировать штрихкод Code-128 → серия+номер ТТН **точно** (BarcodeDetector / ZXing-WASM); (б) тянуть фиксированные поля шапки по позиции из боксов слов OCR (anchor/template).
- **Сколько ИИ убирает.** Полностью убирает ИИ из серии/номера ТТН (штрихкод + `rules.ts`, чистая механика) и из части полей шапки. Резерв ИИ — только переменная таблица товаров.
- **Качество.** **Штрихкод строго ТОЧНЕЕ текущего** (точное значение, не OCR-догадка) всегда, когда есть читаемый Code-128. Якоря по позиции надёжны для фиксированной разметки бланка.
- **Плюсы.** Чистый выигрыш без минусов по качеству на этих полях; убирает ИИ из единственного поля со строгим форматом, который `rules.ts` и так валидирует; можно внедрять инкрементально первым шагом.
- **Минусы.** **Нативный BarcodeDetector есть только в Chromium — его НЕТ в Firefox и во ВСЕХ браузерах iOS (там всё WebKit).** Для mobile-first PWA, где заметная доля пользователей на iPhone, это значит, что **ZXing-WASM ponyfill — не крайний случай, а основной путь на iOS**, т.е. мы фактически всё равно тащим WASM-payload (это корректирует ярлык «$0 / низкие усилия» — усилия не нулевые); многие реальные фото не имеют достаточно резкого штрихкода; покрывает только фиксированные поля, не таблицу товаров.
- **Оффлайн.** Полностью (штрихкод + якоря локальны).
- **Стоимость.** $0 по облаку; цена — WASM-payload ZXing на iOS и невысокий one-time инжиниринг.
- **Архитектура.** Ложится в `rules.ts`/`parse.ts` + препроцессинг как отдельный слой перед/рядом с провайдером; форма `OcrResult` неизменна.
- **Вердикт.** **Чистый выигрыш по качеству фиксированных полей — делать первым** (внутри Варианта 4 как самый дешёвый слой), помня, что на iOS это поставка ZXing-WASM, а не «бесплатный нативный API».

---

### Почему не рассматриваем клиентский прокси / serverless-функцию

Несколько вариантов (Azure, AWS, защита облачного ключа) «чинятся» крошечной edge-функцией (Cloudflare Worker и т.п. ~20 строк): она держит секрет, добавляет CORS, делает submit-then-poll. **Мы сознательно выносим это за рамки** из-за исходного ограничения продукта: **ноль серверной инфраструктуры, чистый статик-хост, ключи у пользователя.** Появление даже одной edge-функции меняет операционную модель (деплой, секреты, биллинг, поддержка), что выходит за рамки текущей памятки. Если это ограничение будет снято — Azure и защищённый облачный ключ возвращаются в игру и заметно меняют расклад §4-V2.

## 5. Рекомендация

> **Статус: только анализ. К разработке НЕ утверждено.**

Сопоставляя с реальностью этого приложения (фото кириллических бумажных ТТН + чистые электронные; браузер без бэкенда; user-ключи; оффлайн как ценность), ранжирование «что пробовать первым по соотношению выигрыш/риск/стоимость»:

1. **Вариант 5 (штрихкод + якоря) — первым.** Чистый выигрыш по качеству, ноль регрессии, оффлайн, точнее текущего на серии/номере ТТН. Самое дешёвое и безрисковое — с поправкой, что на iOS придётся поставлять ZXing-WASM (нативного BarcodeDetector там нет), так что усилия не строго нулевые.
2. **Вариант 4 (гибрид с эскалацией по уверенности) — целевая архитектура, но с открытым риском.** Это прямой ответ на вопрос владельца «минимизировать ИИ»: дешёвый/локальный слой первым, дорогой зрительный LLM только на остатке. Повторяет уже работающий в `shared/match` паттерн → низкий архитектурный риск. **Но реальная экономия не доказана, пока не проверена надёжность гейта:** колонко-сдвиг проходит арифметическую проверку и пропускает fallback (см. §4-V4). Обязателен прототип гейта на реальных ТТН до любых выводов об экономии.
3. **Вариант 3 (OCR → текст-LLM) — как дешёвый слой внутри Варианта 4**, особенно для чистого электронного пути; заметно дешевле текущего (убираем image-токены).
4. **Вариант 1 (полностью локальный, PaddleOCR-wasm) — как оффлайн/без-ключа fallback** и для электронных накладных; **не как единственная замена.**
5. **Вариант 2 (облачный document-AI):** держать текущий зрительный LLM основным; Google Vision рассмотреть позже только если захотим более дешёвый текстовый OCR-слой — и приняв два минуса: структуризатор придётся вернуть, а browser-direct возможен лишь незалоченным ключом (плохой профиль безопасности). Azure/AWS — отложить (бэкенд/нет кириллицы).
6. **Вариант 1b (VLM на устройстве)** — отложить по весу/зрелости мобильного WebGPU, пересмотреть позже.

**Чего НЕ делать:** не выпиливать зрительный LLM полностью под лозунгом «без ИИ» — на фото бумажных ТТН это уронит качество ниже порога «исправлять выгоднее, чем набивать», то есть подорвёт саму UX-предпосылку продукта.

**Что важно сохранить при любом варианте:** существующие страховки (per-line confidence LOW_CONFIDENCE — с учётом её ограничений из §2, ручное добавление/правка/удаление строк, арифметическая проверка, 2-шаговый flow) — именно они позволяют OCR быть «достаточно хорошим», а не идеальным.
