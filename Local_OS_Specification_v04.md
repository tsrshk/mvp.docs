## 1. Контекст проекта

### 1.1 Кто строит

- **Founder:** физик-инженер, frontend-разработчик. Работает full-time. ~10–15 ч/нед на проект вечерами.
- **Customer Zero / партнёр:** жена. Управляет нашей кофейней. Закупки, меню, поставщики, сотрудники — всё на ней. Боль реальная, ежедневная.
- **Команда (потенциально на Phase 2):** Java BE на парт-тайм (10 ч/нед), manual QA на парт-тайм. Подключаются когда продукт выходит на рынок.

### 1.2 Жизненный фрейм

- Двое детей
- Стройка дома (есть бригада, founder занимается только проектированием и тех. контролем)
- Кофейня (управляет жена, founder только юр-директор)
- Основная работа full-time
- Финансовая подушка >6 мес расходов
- Стартовый капитал для Phase 1: ≤$1500

### 1.3 Цели проекта

1. **Главная цель (Phase 1):** облегчить жизнь жене. Снять с неё ежедневную рутину анализа и закупок.
2. **Бизнес-цель (Phase 2):** $1M годовой прибыли через 5 лет от того же продукта, проданного 200–500 кофейням / пекарням в СНГ.

Цели в этом порядке — не наоборот. Если первая не достигнута, ко второй не переходим.

## 2. Главный принцип

> **«Сначала помочь жене. Потом — деньги.»**

### 2.1 Что это означает практически

- Phase 1 (стадии S0–S2): продукт для одного человека — жены. Бесплатно. Цель — измеримая ценность для неё.
- Phase 2 (стадии S3–S5+): тот же продукт для рынка. Платно. Цель — $1M прибыли.
- **Wife-Gate** — критерий перехода между фазами: жена сама говорит «без этого хуже» и пользуется ежедневно 4+ недель.

### 2.2 Почему такой порядок

| Что | Классический стартап | Наш подход |
|---|---|---|
| Валидация боли | 15 customer interviews с незнакомцами | 1 разговор с женой (Customer Zero) |
| Стоимость провала | $5–15K + 6+ месяцев | $500–1500 + 3 месяца |
| Скорость PMF | 3–12 месяцев | Ежедневный фидбек |
| Founder motivation | Деньги, ego — выгорает | Близкий человек — устойчиво |

## 3. Функциональные требования

Все требования собраны от Customer Zero (жены). Источник — её повседневная рутина.

### 3.1 Блок A: OCR накладных

**Контекст:** жена получает 20–40 накладных в месяц от поставщиков. Бумажные и электронные. Сейчас вручную переносит позиции в Esupl. Это **самая времязатратная рутина**.

#### A.1 Функциональные требования

- Жена фотографирует накладную → отправляет в наш Telegram-бот
- Система извлекает структурированные данные:
  - Поставщик (название, реквизиты если есть)
  - Дата накладной
  - Список позиций: название, количество, единица измерения, цена за единицу, сумма
  - Итоговая сумма
- Жена видит результат в боте, может поправить ошибки (если есть)
- После подтверждения данные:
  - Сохраняются в нашу БД (история накладных)
  - Автоматически добавляются в Esupl через API (приход на склад)

#### A.2 Технические требования

- **OCR-решение:** будет выбрано после эксперимента. Кандидаты: **Claude Vision** (рекомендуется) и **Azure Document Intelligence** (текущий выбор founder'а)
- Формат входа: JPG, PNG, PDF
- Качество распознавания: ≥95% на стандартных накладных
- Время обработки: ≤30 сек на накладную
- Поддержка кириллицы (русский, белорусский)

#### A.3 Open questions

- [ ] Бенчмарк-тест: 5 реальных накладных через Claude vs Azure → выбор решения
- [ ] Формат накладных: бумажные фото / PDF / Excel?
- [ ] Стандартные формы (ТТН, акт) или разнообразные?
- [ ] Сколько занимает ручной ввод сейчас? (для расчёта ROI)

### 3.2 Блок B: Маппинг позиций накладная ↔ Esupl

**Контекст:** позиция в накладной называется иначе, чем в Esupl. У разных поставщиков одна и та же позиция называется по-разному.

#### B.1 Пример проблемы

```
Поставщик "Кофеман":     "Зерно эфиопия Иргачеф 1кг"
Поставщик "BravaCoffee": "Зёрна 1kg Ethiopia Yirgacheffe"  
Поставщик "Roca":        "Эфиоп Йиргачефф зерно 1000г"
В Esupl номенклатура:    "Зерно эфиопия 1кг"  ← каноническое название жены
```

Все три позиции должны быть распознаны как одна и та же.

#### B.2 Архитектурное решение

1. **Внутренний справочник SKU:** хранит каноническое название (из Esupl) + множество известных вариантов написания от разных поставщиков
2. **Learning loop:** при первой встрече новой позиции — AI предлагает кандидата из справочника, жена подтверждает/корректирует
3. **Auto-mapping:** в следующий раз эта позиция от этого поставщика мапится автоматически
4. **Fuzzy matching + embeddings:** для предложения кандидатов на маппинг

#### B.3 Функциональные требования

- При импорте накладной для каждой позиции:
  - Поиск в справочнике маппингов «{поставщик} + {название из накладной}» → найдена связь? → автомат
  - Если нет — AI предлагает top-3 кандидата из Esupl SKU
  - Жена выбирает или вводит новый SKU
  - Запись связи в справочник
- Управление справочником через UI/бот:
  - Просмотр всех маппингов
  - Удаление/корректировка
  - Добавление новых SKU

#### B.4 Технические требования

- Vector embeddings для семантического поиска (Claude embeddings или OpenAI)
- Levenshtein/fuzzy matching как fallback
- Хранение: PostgreSQL таблица `sku_mappings(supplier_id, raw_name, canonical_sku_id)`

#### B.5 Open questions

- [ ] Сколько уникальных SKU в Esupl у жены? **Известно: 50–150**
- [ ] Сколько активных поставщиков? **Известно: 7–12**
- [ ] Готова ли жена потратить 1–2 часа на onboarding для начального маппинга?
- [ ] Что делать если поставщик прислал позицию которой нет в Esupl? (новый сорт, расходник и т.п.)

### 3.3 Блок C: Сравнение поставщиков

**Контекст:** жена хочет видеть «у кого лучше заказать». При этом цены у каждого клиента **персональные** (свои скидки, условия).

#### C.1 Ключевое решение: персональная база цен

- База поставщиков и цен **не агрегируется между клиентами**
- Каждый пользователь ведёт **свою** базу: поставщики, их предложения, цены с учётом персональных скидок
- На MVP — никакой межклиентской агрегации (агрегация анонимная — на Phase 2+ когда будет N клиентов)

#### C.2 Функциональные требования

- Жена ведёт справочник своих поставщиков:
  - Контакты, условия доставки, минимальная партия
  - Прайс-лист (вводится вручную, можно обновлять)
  - История цен (как изменялись с течением времени)
- Когда система получает накладную — автоматически проверяет: текущая цена ≠ цена в справочнике? → подсказка жене
- Система проактивно показывает сравнение:
  - «Зерно эфиопии: текущий поставщик A — 1350 ₽/кг. У B в твоей базе — 1400 ₽/кг. У C — 1380 ₽/кг с мин. партией 5кг»
  - «Поставщик A поднял цены на 8% за квартал — рассмотреть альтернативы?»

#### C.3 Источники данных о ценах поставщиков

- **Phase 1:** жена вводит вручную, актуализирует периодически
- **Phase 2+:** парсинг прайс-листов (Excel, PDF), интеграция с поставщиками
- **Phase 3+ (если оправдано):** анонимная агрегация цен между клиентами для бенчмарков

#### C.4 Open questions

- [ ] В каком формате жена получает прайсы поставщиков? (Excel, PDF, мессенджер)
- [ ] Готова ли вручную поддерживать базу прайс-листов?
- [ ] Хочется ли отслеживать историю цен (графики, тренды)?
- [ ] Какие пары поставщик/SKU самые важные для сравнения? (приоритеты)

### 3.4 Блок D: Анализ конкурентов

**Контекст:** жена хочет понимать позиционирование своей кофейни относительно ближайших конкурентов.

#### D.1 Источники данных — что на MVP

- **Меню конкурентов через фото.** Жена/founder раз в квартал делает обход 5–10 ближайших кофеен, фотографирует меню → OCR → база цен и ассортимента
- **Google Maps reviews:** через Google My Business API (если кофейня клиент — OAuth) + парсинг публичных отзывов конкурентов
- **AI-анализ отзывов:** sentiment, тренды жалоб, упоминания конкретных позиций

#### D.2 Источники данных — НЕ на MVP

- **Соцсети (Instagram, ВК, TikTok):** парсинг сложен юридически, неустойчивые API, много noise. На Phase 2+, если оправдано.

#### D.3 Функциональные требования (MVP scope)

- Жена видит карту конкурентов с фильтрами (тип, цена, рейтинг)
- Сравнение цен топ-позиций своей кофейни vs средняя по конкурентам района
- Еженедельная сводка по отзывам: что хвалят/ругают у нас и у соседей
- Тренды: «упоминания матчи в отзывах конкурентов выросли на 60% за месяц»

#### D.4 Open questions

- [ ] Сколько конкурентов реально интересует жену? (5? 10? 20?)
- [ ] Как часто делать обход с фото меню? (раз в месяц? в квартал?)
- [ ] Какие метрики конкурентов самые важные? (цена / ассортимент / рейтинг / тренды)
- [ ] Готова ли жена сама раз в N делать «обход» или нужен другой способ?

---

### 3.5 Сквозные требования (cross-cutting)

Применяются ко всем блокам.

#### 3.5.1 Human-in-the-loop

- AI **никогда не действует автоматически** в критических операциях (отправка заказа, изменение данных в Esupl без подтверждения)
- AI **показывает данные и аргументы**, человек принимает решение
- Sanity checks на любые автоматические операции (например, >3× отклонение от нормы — требует подтверждения)

#### 3.5.2 Канал взаимодействия

- **Основной:** Telegram-бот (жена уже использует Telegram постоянно)
- **Дополнительный (Phase 2):** web-кабинет для аналитики и настроек
- **Не на MVP:** мобильное приложение (Telegram достаточен)

#### 3.5.3 Локализация

- Интерфейс на русском
- Понимание белорусского в текстах (отзывы, поставщики)
- Поддержка кириллицы во всех источниках данных

## 4. Технические решения

### 4.1 Стек

| Слой | Решение | Обоснование |
|---|---|---|
| **Frontend (Phase 1)** | Telegram Bot API | Жена уже в Telegram. Минимум разработки. |
| **Frontend (Phase 2)** | Next.js / React | Founder знает FE. |
| **Backend** | Java (Spring Boot) или Node.js | Зависит от партнёра. На Phase 1 минимум BE. |
| **БД** | PostgreSQL + Redis | Стандарт. Дёшево на Hetzner/DO. |
| **LLM** | Claude API (Sonnet 4.6 + Haiku 4.5 для дешёвых задач) | Доступ из РБ без VPN. Качество. |
| **OCR** | **TBD** (Claude Vision vs Azure DI — тест в S1) | Решается экспериментом |
| **Vector store** | pgvector в PostgreSQL | Для семантического поиска SKU |
| **Hosting** | Hetzner или Digital Ocean | $20–50/мес для Phase 1 |
| **POS integration** | Esupl API | Используется в нашей кофейне |

### 4.2 Архитектура (Phase 1)

```
[Жена] ←→ [Telegram Bot]
              ↓
         [Application Layer (Node.js/Java)]
              ↓
    ┌─────────┼─────────┬──────────┐
    ↓         ↓         ↓          ↓
[PostgreSQL] [Esupl API] [Claude API] [External APIs]
                                        - Weather
                                        - Google Maps
                                        - OCR vendor
```

### 4.3 Ключевые архитектурные решения

#### 4.3.1 OCR через Claude Vision (предварительно)

**Решение:** для MVP использовать Claude Vision API, не Azure Document Intelligence.

**Обоснование:**
- Доступ из РБ без VPN
- Одна зависимость (тот же API что и для аналитики)
- Стоимость на объёме 20–40 накладных/мес — $0.30–0.50/мес
- Качество 95–98% на структурированных документах достаточно

**Условие проверки:** провести бенчмарк-тест на 5 реальных накладных перед стартом S1. Если Claude показывает <90% точности — пересмотреть.

#### 4.3.2 Esupl как источник истины

- Esupl — главная учётная система кофейни
- Наша система **читает** из Esupl (продажи, остатки, рецепты)
- Наша система **может писать** в Esupl (приход накладных) — но только после подтверждения жены
- Наша БД — это **слой обогащения** (маппинги, поставщики, конкуренты, история, инсайты)

#### 4.3.3 SKU как central entity

- Все основные сущности привязаны к SKU из Esupl
- Маппинги, цены поставщиков, рецепты, прогнозы — всё работает через SKU
- При импорте новой накладной — первое что делаем, маппим позиции на наши SKU

| **Phase 2 / P2** | Стадии S3–S5+. Продукт для рынка. Платно. |
| **MRR / ARR** | Monthly / Annual Recurring Revenue |
| **Human-in-the-loop** | Принцип: AI показывает данные, человек принимает решения |
