# LCOS — Фазы разработки

**Что это:** поэтапный план двух продуктовых фаз, зафиксированных основателем 2026-07-03. План собран из аудита доков и кода (4 независимых аудитора) и трёх независимых проектов плана (линзы: данные-сначала, ценность-сначала, риск-сначала), синтезированных судьёй. Развилки и принятые решения — в §7.

**Терминология (важно):** здесь «Фаза 1/2» — **продуктовые** фазы (какую рутину закрываем). Это другая ось, чем рыночные опции A/B/C из `06_STRATEGY.md` (какому рынку продаём) и чем «Phase 2 = SaaS» из старых planning-доков. Старую нумерацию при конфликте считать superseded этим документом.

**Сквозные принципы** (из `06_STRATEGY.md` §1): фича делает работу, а не показывает цифры; AI готовит — человек подтверждает; одна рутина за раз до боевого ежедневного использования. Соло-разработчик 10–15 ч/нед → этап = 20–60 ч ≈ 1,5–4 недели.

**Каркас плана:** скелет — риск-сначала (дешёвый спайк убивает неизвестные до инвестиций); ритм — ценность-сначала (каждый этап заканчивается боевой рутиной Customer Zero); модель данных — конвенция `organization_id` + `source` + `external_id` на каждой новой таблице с первого дня (это и есть заделы под self-service поставщиков и портирование истории из POS-провайдера).

## Фаза 1 — полностью закрыть боль накладных

Состав боли (по формулировке основателя): ввод данных → маппинг SKU → настройки каждого поставщика (периодичность, объёмы, мин. сумма) → планирование заказов по остаткам ингредиентов. Ступени 1–4 лестницы рутин из `06_STRATEGY.md` §2.

### Э0. Разведка-спайк «убить неизвестные» (~20 ч)

- Read-only пробы по шаблону `.tools/probe-esupl.mjs`: `remains*`, `sales` (team 17957) — формы ответов, единицы, глубина истории, rate limits; результаты фиксировать в `mvp.docs/api/esupl/`.
- Фикс путей `esupl.py:43,65`: `/teams/{id}/following?is_virtual=1`, `/teams/{id}/ingredients`, пагинация.
- Одна контролируемая боевая накладная через `ERP_WRITE_ENABLED=ON→OFF`.
- ADR «источник остатков»: A `remains` / B приход−расход / C ручные инвентаризации.

**Готово когда:** формы remains/sales задокументированы; накладная видна в Esupl; решение A/B/C принято.
**Kill-критерии:** remains 404/пусто → Э3 строится на B/C; глубина sales <3 мес → портирование в Ф2 урезается до «с сегодня вперёд»; запись падает → эскалация в Esupl, конвейер остаётся в `prepared`.

### Э1. Серверная маппинг-память + P0-долг (~30 ч, параллельно Э0)

- Таблица `sku_mappings(organization_id, supplier_id, normalized_description, ingredient_id, packing_id, source, confirmed_at)` + CRUD; перенос `mappingStorage.ts` с localStorage на backend с one-shot импортом локальных записей.
- Закрыть бэклог: DEC-02 (дроп `sku_embedding`), DEC-05 (candidate-set → вариант B), ALIGN-01 (fail-closed шифрование), VER-01 (тесты non-negotiables, включая гейт записи); переименование FE `entities/order → receipts` (снять коллизию имён до появления purchase orders).

**Готово когда:** маппинг, подтверждённый на одном устройстве, применяется на другом; накладная знакомого поставщика маппится автоматом ≥80% строк; тесты зелёные.

### Э2. Настройки поставщика + задел self-service (~25 ч)

- **Отдельная таблица** `supplier_settings(supplier_id, delivery_days, lead_time_days, min_order_amount + currency, contact_phone, notes, is_active)` — не колонки на `suppliers`: `suppliers` остаётся зеркалом Esupl, а обособленная сущность — ровно то, что позже заполняет сам поставщик.
- PATCH-роут + расширение `SupplierService`; FE `/suppliers`, `/suppliers/:id` через локальный тип-оверлей (не трогать `shared/pos/types.ts`).
- **Задел self-service в той же миграции (~2 ч):** роль `supplier` в enum `Role`, nullable `suppliers.portal_user_id → users.id`, ADR «дверь открыта: глобальный supplier-пользователь + invite-токены — потом». В auth/UI ничего не строим.
- НЕ строим: `default_order_volume`, `supplier_prices`/ценовые алерты.

**Готово когда:** 5–7 активных поставщиков кофейни заполнены; карточка реально используется при заказе.

### Э3. Остатки: снимки + список «что заканчивается» (~30 ч)

- Источник по решению Э0: `ErpProvider.get_stock()` (remains) и/или ручной ввод.
- Таблица `stock_levels(organization_id, subdivision_id, ingredient_id, warehouse_id, quantity, as_of, source)`; ручной sync-endpoint (seam как `suppliers/sync`, без Celery); `ingredients.reorder_point` (ручной ввод).
- FE: `PosProvider.listStock()` + страница `/stock` со свежестью снимка, блоком «Мало» (= готовый shopping-list) и ручной корректировкой (`source=manual`).

**Готово когда:** список «мало» совпадает с реальностью на очередной закупке (проверка Customer Zero); обновление ≤1 мин.
**Kill-критерий:** расхождение >50% и не лечится корректировками → Э4b деградирует до «reorder_point + ручной остаток при заказе».

### Э4a. Черновик заказа руками (~30 ч; после Э2)

- Таблицы `purchase_orders(…, status: draft/confirmed/sent_manually/received/cancelled, confirmed_by/at)` + `purchase_order_lines`; **никакой записи в Esupl**.
- `routes/purchase_orders.py`; FE `entities/purchase-draft`, route `/orders`, редактируемая таблица (параметризованный `features/lines-table`), индикатор «сумма vs `min_order_amount`», prefill позиций из `getSupplierProductIds` (что раньше возили).
- Подтверждение → «Скопировать текст для WhatsApp» (канал передачи — текст, авто-отправки нет).

**Готово когда:** реальный заказ собран в LCOS, прошёл проверку мин. суммы, ушёл поставщику.

### Э4b. AI-предзаполнение черновика (~25 ч; после Э3 + Э4a)

- `order_planning_service.py`: `stock_levels` + `reorder_point` + `lead_time_days`/`delivery_days` + последние цены и объёмы из истории накладных → предложенные количества по поставщику; кнопка «Предложить заказ», пометка AI-строк.
- Прогноз спроса НЕ строим — пороги ручные (Э6 позже подскажет их из истории).

**Готово когда:** еженедельный заказ стартует с AI-черновика и правится ≤30% строк.
**Kill-критерий:** правится >70% строк три цикла подряд → откат к чек-листу «что кончается» без количеств, пересмотр модели потребления.

### Э5. Замыкание петли + боевой режим (~30 ч)

- Связь `invoices.purchase_order_id`; авто-матч прихода на открытый PO поставщика; экран расхождений в `InvoiceWorkbench` («заказано 10 — привезли 8»); PO → `received`.
- `ERP_WRITE_ENABLED=ON` постоянно; DEFER-04 (backend-idempotency отправки); метрика S2 (≥95% строк без правок, ≤30 с на накладную); точечные фиксы по журналу боевого использования (multi-page/confidence-gate — только если журнал показывает боль).

**Готово когда:** полный цикл заказ → WhatsApp → приход по фото → расхождения → received пройден на живой поставке; 2 недели цикла без блокнота и без ручного дублирования в Esupl → **Фаза 1 закрыта**.

## Фаза 2 — конкурентное позиционирование + история из POS-провайдера

Ступени 3(частично)–6 лестницы. Портирование истории — через `ErpProvider`-шов: сегодня Esupl, при выходе на другие POS (опция B стратегии) — те же методы у другого провайдера.

### Э6. Портирование истории продаж (~35 ч; буфер — параллелится с Э2–Э5)

- `EsuplErpProvider.list_orders/list_sales` (team-scoped, пагинация); таблицы `sales_history(organization_id, subdivision_id, external_id, occurred_at, normalized lines, source)`, `sync_state(entity, cursor, synced_at)`.
- One-shot backfill на глубину, выясненную в Э0, + инкрементальный ручной триггер; FE-статус импорта в `/settings`.
- Отдача назад в Ф1: фича «предложить `reorder_point` из фактического расхода» на `/stock`.

**Готово когда:** вся доступная история импортирована; повторный запуск идемпотентен; суммы бьются с Esupl-отчётом; ≥10 порогов приняты из подсказок.

### Э7. Конкуренты: фото меню → структурированные позиции (~35 ч; не зависит от Э6)

- Таблицы `competitors(organization_id, name, address, source_urls)`, `competitor_menu_items(competitor_id, name, price, category, captured_at)`.
- OCR-тип `menu` — реюз `shared/ocr`, `prepare-step`, `image-cropper`, двухшагового wizard'а; FE `/competitors`, `/competitors/:id`.
- Отзывы: только ручные заметки; таблицу `competitor_reviews` не создаём до решения вопроса легальности сбора (Q5), скрейпинга Google нет.

**Готово когда:** 3 реальных конкурента введены телефоном, ≤15 мин каждый.

### Э8. Позиционный отчёт (~35 ч; после Э6 + Э7)

- Матчинг своего меню (Esupl dishes + `sales_history`) ↔ `competitor_menu_items` через `shared/match` + LLM-fallback.
- FE `/positioning`: по-позиционное сравнение цен, пробелы ассортимента, конкретные рекомендации («латте дешевле рынка на 12% — поднять на 0,5») с кнопкой принято/отклонено. Отчёт готовит решение, а не таблицу цифр.

**Готово когда:** Customer Zero изменил ≥1 цену по рекомендации и назвал отчёт достоверным.
**Kill-критерий:** авто-матчинг <60% точности → ручная привязка пар вместо автоматики.

| Перед | Что проверить |
|---|---|
| Э0 (внутри) | `remains`, `remains/item`, `remains/item/warehouses` на team 17957: форма, единицы, warehouse-разрез; `sales`: глубина, гранулярность `include=items`; rate limits; боевой `POST /outgoing-invoices` за гейтом |
| Э1 | реальные пути suppliers/ingredients (`/following?is_virtual=1`, team-scoped) — закрывается фиксом в Э0 |
| Э3 | подтверждённая форма remains: отрицательные значения, соответствие `ingredient_id` каталогу |
| Э6 | пагинация `sales` на всю глубину, стабильность `external_id`, `orders` для закупочной истории |
| Э8 | `dishes`/`menu-products` (своё меню) — форма ответа нигде не проверена |

## 4. Заделы: закладываем сейчас, не строим

- **Supplier self-service:** роль `supplier` в enum (Alembic-миграция дешёвая сейчас, дорогая потом), nullable `suppliers.portal_user_id → users.id`, настройки в обособленной `supplier_settings` (та самая сущность, которую позже заполняет сам поставщик), ADR «дверь открыта». В auth/UI — ничего.
- **Портирование истории из POS-провайдера:** конвенция `external_id` + `source` на всех таблицах данных с Э1; `sync_state` и провайдер-методы сразу team-scoped с пагинацией; sync только ручным триггером — фонового планировщика нет. Методы объявляются на `ErpProvider` Protocol → другой POS = другая реализация, не другой продукт.

## 5. Что сознательно НЕ делаем в этих фазах

Авто-отправку заказов и любую запись в Esupl кроме gated `write_invoice`; прогноз спроса/рецептуры (ручные пороги + подсказки из истории); `supplier_prices` и ценовые алерты (F3-B/F4); supplier-портал; отзывы Google/sentiment (F8) до решения Q5; погоду/события (F6), cross-recipe (F9), стратегический диалог (F10); Celery/APScheduler — всё по кнопке; embeddings (Q6).

## 6. Критический путь и календарь

**Критический путь:** Э0 → Э3 → Э4a → Э4b → Э5. Параллелятся: Э1 с Э0; Э2 с Э3; Э6 — буферная задача при блокировке Ф1.
**Оценка:** Фаза 1 ≈ 190 ч ≈ 4 месяца при 10–15 ч/нед; Фаза 2 ≈ 105 ч ≈ +2 месяца.

## 7. Развилки трёх планов и принятые решения

1. **Чем стартовать** (фундамент данных / сразу ценность / спайк рисков) → **Э0-спайк параллельно с Э1-маппингами**: спайк за 20 ч определяет архитектуру Э3–Э4 и объём Ф2, а серверная маппинг-память — самый дешёвый durable-актив, ждать нечего.
2. **Где живут настройки поставщика** (колонки на `suppliers` vs отдельная таблица) → **отдельная `supplier_settings`**: `suppliers` — зеркало Esupl, обособленная сущность и есть задел self-service; цена та же.
3. **История цен** (строить с настройками vs отложить) → **отложить**: для черновика заказа достаточно последней цены из `invoice_lines`; отдельная таблица — оптимизация, не блокер боли.
4. **Планировщик одним куском (55 ч) или сплитом** → **сплит Э4a+Э4b**: ручной черновик с мин. суммой и WhatsApp-текстом даёт рутину на месяц раньше, а kill-критерий применяется к 25-часовому AI-куску, не к монолиту.
5. **Сверка прихода с заказом** (была только в одном плане) → **взять**: превращает набор фич в замкнутую петлю «заказ → приход → расхождения» и даёт честный критерий закрытия Фазы 1.
6. **OCR-закалка отдельным этапом** → **не выделять**: делаем внутри Э5 по журналу реального использования, иначе полируем сценарии, которых нет в бою.
7. **Расход из продаж уже в Фазе 1** → **нет**: Ф1 на ручных порогах; Э6 возвращает подсказки порогов, не прогноз. Держит Ф1 короче и соблюдает «одна рутина за раз».
