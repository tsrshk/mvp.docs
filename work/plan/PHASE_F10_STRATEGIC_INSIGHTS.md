# Фаза F10 — Стратегические инсайты

> Боль: стратегические решения «на ощупь». Продукт: еженедельный стратегический разговор —
> AI формулирует 3 главных ВОПРОСА недели (не отчёт) на данных всех модулей, дальше
> свободный диалог с сохранением истории. AI аргументирует и моделирует — решает человек (G7).
> Завершает Phase 1; после неё — PILOT-GATE (проверка на собственном бизнесе; в ADR-003 — Wife-Gate).
> Сквозные требования — `00_IMPLEMENTATION_PLAN.md` §4.

**Цель:** ≥1 стратегическое решение в месяц на основе разговора; PILOT-GATE:
владелец пилотной кофейни подтверждает «без этого хуже» + сэкономлено ≥3 ч/нед за 4 недели.

**Зависимости:** максимум ценности при F4+F5+F6+F7+F8 (деградирует изящно: отсутствующий
модуль просто не даёт контекста). Формальный минимум: F5. **Оценка:** 3 недели.

---

## 1. Backend

### F10-B1. Контекст-сборка (детерминированная, ядро фазы)
- `InsightContextBuilder` (services): собирает компактный структурированный контекст
  подразделения за окно N недель: продажи/аномалии (F5/F6), изменения цен поставщиков и
  алерты (F3/F4), позиция vs район (F7), тренды отзывов (F8), принятые/отклонённые идеи (F9).
- Бюджет контекста: агрегаты и топ-N, не сырые строки; размер ограничен
  (`insight_context_weeks` REGISTRY default 4). Отсутствующий модуль → секция опущена
  с пометкой в контексте.
- Билдер — чистая функция от БД, покрывается тестами отдельно от LLM.

### F10-B2. Еженедельная сессия «3 вопроса»
- Джоба планировщика (расписание в REGISTRY, по умолчанию совместно с дайджестом F5):
  контекст → `ai_complete` (ОСНОВНАЯ модель — это флагманская фича) → строго JSON:
  `questions[{question, why_now, data_refs[], suggested_angles[]}]` (ровно 3).
- Правило промпта: вопросы и аргументы ссылаются ТОЛЬКО на переданные данные; формат
  «вопрос для обсуждения», не директива («Поднять цену?» — а не «Подними цену»).
- Хранение: `insight_sessions` (`SubdivisionScopedMixin`, uuid pk): `week_start`,
  `context_snapshot JSONB` (что видел LLM — для аудита ответов), `questions JSONB`,
  `created_at`, `read_at?`. Unique `(subdivision_id, week_start)`.
- AI недоступен в момент джобы → сессия не создаётся, ошибка в sync_run; ручная
  генерация `POST /api/v1/insights/generate` доступна всегда (fail-closed, не тихо).

### F10-B3. Диалог
- `insight_messages` (uuid pk, FK→insight_sessions CASCADE): `role` (`user|assistant`),
  `content Text`, `created_at`.
- `POST /api/v1/insights/{session_id}/messages`: сообщение пользователя → LLM-вызов
  с (context_snapshot + история сессии + новое сообщение) → ответ сохраняется и возвращается.
- Ограничения: история сессии обрезается по бюджету токенов (старые ходы суммируются или
  отбрасываются с пометкой); `insight_dialog_daily_limit` сообщений/сутки (REGISTRY,
  default 50) — G11. Моделирование «что если» — только на числах из контекста; промпт
  требует показывать расчёт.
- Гейт: `module_insights_enabled`.

### F10-B4. API
- `GET /api/v1/insights` (список сессий), `GET /api/v1/insights/{id}` (вопросы + сообщения),
  `POST /api/v1/insights/generate`, `POST /api/v1/insights/{id}/messages`,
  `POST /api/v1/insights/{id}/read`.

## 2. Frontend

### F10-F1. Страница «Разговор недели»
- Карточка недели: 3 вопроса с «почему сейчас» и данными-основаниями; бейдж непрочитанного.
- Чат-интерфейс под вопросами: ввод свободной формы, стриминга не требуется (обычный
  запрос-ответ с индикатором «думает»; стриминг — отдельным решением, если UX потребует).
- История прошлых недель — список сессий.
- Мобайл-фёрст (это «понедельничный ритуал» с телефона); mock-провайдер — демо-сессия.

## 3. Вне объёма
- Автодействия по итогам разговора (никогда); голос; push за пределами PWA;
  кросс-организационные бенчмарки (приватность).

## 4. Acceptance Criteria

- [ ] AC-1. `InsightContextBuilder` на фикстурной БД со всеми модулями отдаёт все секции;
      без F7/F8-данных — корректно опускает секции (unit-тесты, без LLM).
- [ ] AC-2. Еженедельная сессия создаётся джобой (тест с ручным триггером): ровно 3 вопроса,
      каждый с `data_refs`, указывающими на существующие элементы context_snapshot;
      невалидный JSON от LLM → сессия не создаётся, ошибка залогирована (тест).
- [ ] AC-3. Диалог: сообщение → ответ с учётом context_snapshot (respx-тест: контекст
      присутствует в запросе к LLM); история персистится и восстанавливается после
      перезагрузки страницы.
- [ ] AC-4. Лимит сообщений/сутки и обрезка истории работают (тест); лимиты — из SQLAdmin.
- [ ] AC-5. AI/VPN недоступны → явные 503 на generate/messages; прошлые сессии читаемы
      (тест + ручная проверка).
- [ ] AC-6. `context_snapshot` сохраняется при генерации — ответы аудируемы «что видела
      модель» (тест наличия + просмотр в SQLAdmin).
- [ ] AC-7. Тенант-изоляция insight_sessions/messages; модульный гейт.
- [ ] AC-8. Реальная проверка: 2 недели подряд сессия генерируется по расписанию, вопросы
      осмысленны (оценка Customer Zero), ≥1 диалог доведён до решения — фиксация
      в `work/phase-f10.md`.
- [ ] AC-9. Общий DoD G10.

**После фазы — PILOT-GATE чек (ADR-003):** 4 недели реального использования всего продукта
на пилотной кофейне, вердикт владельца «без этого хуже», ≥3 ч/нед экономии →
решение о Phase 2 (`PHASE_P2_SAAS_OUTLINE.md`).
