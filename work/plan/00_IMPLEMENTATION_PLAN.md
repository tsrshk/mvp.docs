# LCOS — Общий план реализации

> SSOT по составу фаз имплементации, их порядку, зависимостям и сквозным техническим требованиям.
> Детальные требования каждой фазы — в отдельных файлах `plan/PHASE_*.md` (каждый содержит
> технические требования + Acceptance Criteria для валидации).

Назначение: по этому плану агент-исполнитель реализует фазы по одной; владелец валидирует
каждую фазу по её AC до перехода к следующей.

---

## 1. Точка отсчёта — что уже сделано (не реализовывать заново)

Сверено с `01_ARCHITECTURE.md` (as-built) и кодом. Реализовано и работает:

- **F1 · OCR накладных** — как web PWA (mvp.fe), не Telegram: фото → кроп → нормализация →
  `POST /invoices/recognize` (Claude Vision через бэк) → workbench. Однострочное ограничение:
  уходит только `pages[0]` (см. фазу S2).
- **F2 · Маппинг и приход в Esupl** — supplier auto-match (photo-first), каталог SKU+фасовок
  (`ingredients`/`packings`, org + subdivision override), fuzzy + LLM matching, learning loop
  (mappingStorage), статус-машина накладной, Esupl payload, запись за гейтом `erp_write_enabled`
  (по умолчанию OFF).
- **Платформа**: auth (JWT+refresh+reuse-detection), мультитенантность (org→subdivision→membership),
  трёхуровневая конфигурация (env / system_settings / integration_credentials, Fernet, без env-фолбэка),
  SQLAdmin-суперадминка, VPN-egress fail-closed, модульные гейты (`modules/registry`), сид демо-данных.

**Не сделано**: бэклог стабилизации (`05_BACKLOG.md`), стадии F3–F10, Phase 2 (SaaS).

**Pilot-Gate** — критерий перехода Phase 1 → Phase 2: продукт проверен на собственном
(пилотном) бизнесе — владелец пилотной кофейни (Customer Zero) после 4 недель реального
использования подтверждает «без этого хуже», измеримо сэкономлено ≥3 ч/нед.
(В исходных доках и ADR-003 этот гейт назывался Wife-Gate — термин заменён на нейтральный.)

Нумерация F3–F10 сохранена из `Local_OS_Functional_Stages_v01.md` для трассируемости,
но **все стадии переадаптированы под фактическую архитектуру** (web PWA + FastAPI),
а не Telegram-бот из старой спеки.

## 3. Зависимости и порядок

```
S1 (стабилизация) ─ обязательна первой: чинит принципы + даёт тестовую базу
   ↓
S2 (OCR-качество) ─ независима от F3+, можно параллелить
   ↓
F3 (поставщики/цены) ──→ F4 (сравнение)          ветка A: закупки
F5 (продажи) ──→ F6 (контекст)                    ветка B: аналитика
F7 (конкуренты-меню) ──→ F8 (отзывы)              ветка C: рынок (после S2 — reuse OCR)
F3+F4+F7 ──→ F9 (cross-recipe)
F4+F5+F6+F7+F8 ──→ F10 (инсайты — объединяет всё)
F10 → PILOT-GATE → P2
```

- **Критический путь:** S1 → F3 → F4 → … → F10. Ветки A/B/C взаимно независимы,
  порядок между ними определяет владелец (приоритеты Customer Zero).
- Правило из исходной методологии сохраняется: каждая фаза — самостоятельный релиз,
  после неё продукт используется реально; пауза 3–7 дней на фидбек.

## 4. Сквозные технические требования (обязательны для КАЖДОЙ фазы)

Каждый `PHASE_*.md` ссылается сюда; агент-исполнитель обязан соблюдать без напоминания.
Источник: `LCOS_Conformance_Alignment_GlobalRequirements.md` Часть 3 (R1–R9, до переноса
в `02_REQUIREMENTS.md`), ADR-лог `04_DECISIONS.md`, `mvp.be/CLAUDE.md`.

### G1. Слои и зависимости (backend)
- `api → services → providers/repositories`; services зависят только от `providers/*/base.py` (Protocol).
- Новые внешние интеграции (погода, карты, отзывы) — как провайдеры за Protocol + registry,
  **одна реализация на seam** (ADR-009); кросс-инфра через `ProviderContext`, не в сигнатуры Protocol.
- ORM-модели не пересекают границу провайдеров — только Pydantic-домен (`app/domain/entities.py`).

### G2. Мультитенантность (ADR-008, R5)
- Каждая новая операционная/каталожная таблица несёт `organization_id`
  (`OrganizationScopedMixin`, ondelete=RESTRICT); операционные — также `subdivision_id`
  (`SubdivisionScopedMixin`).
- Новые репозитории **требуют `organization_id` в конструкторе**; скоуп только из JWT
  (`get_tenant_context`), никогда из клиентского ввода.

### G3. Конфигурация и секреты (ADR-005/010/011, R1/R2/R6)
- Новые API-ключи (OpenWeather, Google, Voyage и т.п.) — ТОЛЬКО в `integration_credentials`
  (новые значения `CredentialProvider`), scope по смыслу (`platform` для инфраструктурных,
  `org` для принадлежащих заведению). Без кэша, без env-фолбэка.
- Новые рантайм-настройки (пороги, расписания, флаги) — ТОЛЬКО через `REGISTRY`
  в `core/system_settings.py` (typed SettingSpec) + резолвер. Не в `.env`.
- В `.env` добавляются только статические deploy-значения (URL внешних API — по аналогии
  с `ESUPL_API_BASE`).

### G4. Модульные гейты
- Каждая F-фаза добавляет свой toggle `module_<name>_enabled` в `REGISTRY`
  (default True после релиза фазы) и гейтит свои роуты через `require_module()` — фичу
  можно выключить из суперадминки без редеплоя.

### G5. Fail-closed (ADR-006, R8)
- Все LLM-вызовы — только через `ai_complete()` (VPN-роутинг fail-closed).
- Внешний API недоступен / нет ключа → явная ошибка (503 через error-envelope) или
  честный пропуск с пометкой в данных — никаких тихих фолбэков и деградаций.
- Единый error-envelope `{"error":{code,message,details?}}`.

### G6. Esupl — read-only кроме гейта
- Любое чтение Esupl (продажи, меню, справочники) — через ERP-провайдер, реальный доступ
  read-only (team_id=17957). **Единственная точка записи** — `write_invoice` за
  `erp_write_enabled`. Новых точек записи в Esupl не появляется ни в одной фазе
  (human-in-the-loop, ADR-002).

### G7. Human-in-the-loop (ADR-002)
- AI показывает данные и аргументы — решает человек. Никаких автозаказов, автоответов
  на отзывы, автоизменений меню/цен. Рекомендация ≠ действие.

### G8. Frontend (FSD, ADR-012)
- Строгий FSD: pages → widgets → features → entities → shared; новые API — через
  `baseApi.injectEndpoints` + `queryFn`-обёртки; провайдер-паттерн `backend|mock`
  (mock обязателен — офлайн-демо работает для каждой новой фичи).
- Никаких секретов в браузере; `VITE_*` — только несекретное. UI-текст русский,
  идентификаторы английские. Per-browser хранилища ключуются `orgScopeToken()`.
- Mobile-first: каждая новая страница работает на телефоне (основной сценарий —
  владелец пользуется продуктом с телефона прямо на точке).

### G9. БД и миграции
- Alembic async, ревью после `--autogenerate`, рабочий `downgrade()`.
- Money/quantity — `Numeric` + `Decimal`, никаких float.

### G10. Definition of Done фазы (общая часть, дополняется AC фазы)
- [ ] `docker compose run backend pytest` зелёный (реальный Postgres+pgvector, testcontainers;
      egress через respx); `ruff check .` чистый.
- [ ] FE: `npm run build` (tsc -b && vite build) зелёный.
- [ ] Новые тесты покрывают: tenant-изоляцию новых таблиц, fail-closed новых интеграций,
      гейт нового модуля.
- [ ] Документация: `01_ARCHITECTURE.md` (новые таблицы/роуты/провайдеры) + `03_ROADMAP.md`
      (статус фазы) бампнуты в том же шаге; при новых решениях — ADR в `04_DECISIONS.md`.
- [ ] Ручная browser-верификация основного сценария фазы (Playwright/вручную), скриншоты
      в журнал `work/<фаза>.md`.
- [ ] Изменения НЕ коммитятся агентом — коммитит владелец (существующая конвенция).

### G11. AI-бюджет
- Дешёвые/массовые задачи (sentiment, категоризация, дайджест-текст) — дешёвая модель
  (Haiku-класс) через `system_settings`-настройку модели; дорогие (OCR, стратегический диалог) —
  основная модель. Бюджет Phase 1 на API ≤ $1500 суммарно — новые фичи не должны
  создавать фоновых LLM-вызовов без явного расписания/действия пользователя.

## 5. Валидация фаз (процесс)

1. Агент-исполнитель получает один `PHASE_*.md` + этот файл.
2. Реализует; прогоняет G10 + AC фазы; заполняет чек-боксы AC с доказательствами
   (имя теста / скриншот / вывод команды) в рабочем журнале `work/<фаза>.md`.
3. Владелец проверяет AC независимо (или отдаёт валидирующему агенту), затем реальное
   использование 3–7 дней.
4. Только после этого — следующая фаза. Метрики стадий (точность OCR, auto-mapping rate,
   использование дайджеста) — из `03_ROADMAP.md` / исходных стадий, меряются на живых данных.
