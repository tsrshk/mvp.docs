# Фаза F4 — Сравнение поставщиков и ценовые алерты

> Боль: «у кого лучше заказать?» решается привычкой, не цифрами. Продукт: сравнение цен
> по SKU между поставщиками из данных F3, динамика цен, проактивные алерты о росте.
> Human-in-the-loop: только информация и аргументы — никаких автозаказов (ADR-002, G7).
> Сквозные требования — `00_IMPLEMENTATION_PLAN.md` §4.

**Цель:** 1–3 валидных рекомендации в месяц с расчётом экономии; владелец смотрит сравнение
перед крупной закупкой.

**Зависимости:** F3 (supplier_prices). **Оценка:** 2 недели.

---

## 1. Backend

### F4-B1. Сервис сравнения (`PriceService` / новый `CompareService`)
- `GET /api/v1/ingredients/{id}/compare` →
  для SKU: по каждому поставщику, у которого есть точки цены: текущая цена (последняя),
  дата наблюдения, мин. партия и условия (из карточки F3), тренд (изменение за 90 дней, %),
  флаг «текущий поставщик» (у кого последняя накладная с этим SKU).
- Расчёт «экономии за месяц»: `(цена_текущего − цена_лучшего) × среднемесячный объём`
  (объём = сумма qty по накладным за последние 90 дней / 3). Все расчёты — Decimal,
  чистая бизнес-логика без AI, покрывается unit-тестами.
- Устаревание данных: цена старше `price_staleness_days` (настройка REGISTRY, default 60)
  помечается `stale: true` — UI показывает «цена от <даты>» (никаких тихих сравнений
  по мёртвым ценам).

### F4-B2. Ценовые алерты
- Новая таблица `price_alerts` (`OrganizationScopedMixin`, int pk): `ingredient_id`,
  `supplier_id`, `kind` (enum: `price_increase` | `cheaper_alternative`), `old_price`,
  `new_price`, `delta_pct Numeric(6,2)`, `message` (Text, русский), `created_at`,
  `read_at?`, `source_invoice_id?`.
- Генерация — синхронно в момент записи цен из накладной (F3-B3), НЕ фоновым LLM-джобом:
  - рост цены (supplier, ingredient) > `price_change_alert_pct` → `price_increase`;
  - после обновления: если у другого поставщика активная (не stale) цена ниже новой
    на > тот же порог → `cheaper_alternative` («У B дешевле на 0.50 BYN/кг»).
- `GET /api/v1/alerts?unread=true`, `POST /api/v1/alerts/{id}/read`.
- Текст alert-сообщения — шаблонный (без LLM); LLM-пояснение — только по явному запросу
  пользователя (F4-B3).

### F4-B3. (Опционально) LLM-пояснение сравнения
- `POST /api/v1/ingredients/{id}/compare/explain` — по данным сравнения `ai_complete`
  генерирует короткое русское объяснение-рекомендацию. Только по кнопке пользователя
  (G11 — никаких фоновых LLM-вызовов); модель — дешёвая (Haiku-класс, ключ модели
  в REGISTRY). Промпт обязан запрещать выдумывать цены не из переданных данных.

### F4-B4. Модульный гейт
- Новый toggle `module_compare_enabled` в REGISTRY (default True), `require_module()`
  на роутах compare/alerts.

## 2. Frontend

### F4-F1. Сравнение в карточке SKU / странице поставщиков
- Точка входа: из прайса поставщика (F3) и из справочника SKU — кнопка «Сравнить».
- Экран сравнения: таблица поставщиков (цена, дата, мин. партия, тренд-стрелка,
  бейдж «текущий»), блок «экономия ≈ X BYN/мес» с формулой раскрытия, кнопка
  «Объяснить» (F4-B3, если реализован). Stale-цены — приглушённые с датой.
- Мобильная версия — карточки.

### F4-F2. Алерты
- Индикатор непрочитанных алертов в шапке/сайдбаре; список алертов (страница или
  секция на главной «Накладные»); тап по алерту → экран сравнения соответствующего SKU;
  пометка прочитанным.
- RTK Query + провайдер-паттерн `backend|mock` (mock — демо-алерты).

## 3. Вне объёма
- Автозаказ/связь с поставщиком (никогда — ADR-002); межклиентская агрегация цен
  (не делается — приватность); push-уведомления (канал решается в F5/Q1).

## 4. Acceptance Criteria

- [ ] AC-1. Для SKU с ценами от ≥2 поставщиков `GET .../compare` возвращает всех, помечает
      текущего и лучшего; расчёт экономии совпадает с ручным (unit-тест на Decimal,
      включая пересчёт фасовок и разный unit).
- [ ] AC-2. Цена старше `price_staleness_days` помечена stale и не участвует в
      `cheaper_alternative` (тест + смена порога из SQLAdmin без редеплоя).
- [ ] AC-3. Импорт накладной с ценой, выросшей >порога → создан алерт `price_increase`
      с корректным delta_pct; при наличии более дешёвой свежей альтернативы → второй алерт
      `cheaper_alternative` (интеграционный тест на invoice submit).
- [ ] AC-4. Алерты видны в UI, непрочитанные посчитаны, тап ведёт на сравнение, прочтение
      персистится (браузерная верификация, мобильный вьюпорт).
- [ ] AC-5. Никакой code-path фазы не пишет в Esupl и не вызывает LLM без явного действия
      пользователя (ревью + grep: compare/alerts не зовут `ai_complete`, кроме explain).
- [ ] AC-6. Если explain реализован: ответ строится только на переданных ценах (промпт-тест:
      в ответе нет цен, отсутствующих во входе), недоступность AI → 503 ai_unavailable,
      экран сравнения продолжает работать (fail-closed, но не блокирует не-AI функции).
- [ ] AC-7. `module_compare_enabled=false` → роуты 404, UI-раздел скрыт.
- [ ] AC-8. Тенант-изоляция `price_alerts` покрыта тестом.
- [ ] AC-9. Общий DoD G10.
