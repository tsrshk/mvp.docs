# Фаза S2 — Качество захвата и OCR

> Доводит контур F1 (фото → распознавание) до целевого качества: многостраничные накладные,
> тип документа в промпте, авто-кроп, страховка от ошибок OCR. Источники:
> `scan-preprocessing-plan.md` (Phases 2–4), `recognition-feature.md` (отложенный B4),
> бэклог DEC-07 вариант (A). Сквозные требования — `00_IMPLEMENTATION_PLAN.md` §4.

**Цель:** точность OCR ≥95% позиций на реальных накладных, ≤30 сек на накладную,
многостраничные накладные не теряют данных.

**Зависимости:** S1. **Оценка:** 2 недели. Можно параллелить с F3.

---

## 1. Объём — backend

### S2-B1. Multi-page recognize (DEC-07 → вариант A)
- `POST /api/v1/invoices/recognize` принимает **до 3 файлов** (multipart, поле `files`
  или повторённое `file`) вместо одного; MIME-валидация каждого (`_ALLOWED_MIME`).
- `OcrProvider.extract_invoice` расширяется до множественных страниц:
  `extract_invoice(pages: list[OcrImage], mime..., context) -> InvoiceDraft`
  (домен `app/domain/entities.py`; сигнатура Protocol меняется единожды и осознанно).
- Claude-провайдер шлёт все страницы одним multimodal-запросом (страницы по порядку);
  результат — ЕДИНЫЙ InvoiceDraft (строки со всех страниц, итог с последней).
- Лимиты: суммарный размер запроса контролируется (страницы уже нормализованы клиентом
  до ≤1568px); превышение → 413 в error-envelope.

### S2-B2. Контекст распознавания в промпте (B4 из recognition-feature)
- FE уже собирает `invoiceType` (`paper|electronic`) и `supplierName` — прокинуть их
  в recognize-запрос (multipart-поля) и в OCR-промпт:
  - paper → искать серию/номер бланка накладной (2 буквы + 7 цифр);
  - electronic → искать № документа;
  - подсказка формата дат/валюты (BYN по умолчанию).
- Схема ответа промпта — согласованная с `InvoiceDraft`; правило «null если не уверен,
  не выдумывать» — обязательное (анти-галлюцинация).

### S2-B3. (Опционально, если дёшево) поле `pages_processed` в ответе recognize —
FE показывает «распознано N страниц».

## 2. Объём — frontend

### S2-F1. Multi-page отправка
- `BackendOcrProvider` шлёт ВСЕ held-страницы (до `MAX_INVOICE_PAGES=3`), не `pages[0]`;
  вернуть лимит 3, если в S1 временно ставили 1; убрать предупреждение S1-F3.

### S2-F2. Авто-кроп (scan-preprocessing Phase 2, «auto-first, manual-optional»)
- Автоопределение границ документа: lazy-loaded OpenCV.js (~8MB WASM, грузится только
  при открытии сканера): grayscale → blur → Canny/Otsu → крупнейший quad → warpPerspective.
- Результат показывается с уже расставленными угловыми ручками существующего
  `ImageCropper`; ручная правка — всегда доступный fallback.
- Принудительный confirm-шаг ТОЛЬКО при слабом сигнале качества (низкая уверенность
  краёв, сильный skew, документ <60% кадра); иначе — тихо.
- Порядок конвейера: crop → существующий `normalizeImage` (resize+JPEG один проход).
- Fail-open: ошибка OpenCV/загрузки WASM → текущее поведение (ручной кроп), распознавание
  не блокируется. Бинаризацию/трешолд НЕ вводить (ADR-014).

### S2-F3. Захват с камеры (Phase 3)
- `capture="environment"` на file-input (мобильная камера напрямую).
- Предупреждение о блике/размытии при слабом качестве — если реализуемо дёшево;
  иначе зафиксировать как отложенное.

### S2-F4. Post-recognition confidence-gate (Phase 4)
- После распознавания прогонять валидаторы `shared/ocr/rules.ts`
  (waybillSeries/waybillNumber/totalRow) + межстрочную арифметику (`lineSumMismatch`):
  - всё валидно → обычный workbench (зелёная панель);
  - есть проблемные поля → секция «Проверьте эти поля» наверху workbench с фокус-переходами
    (переиспользовать механизм validation panel), НЕ полноэкранная форма.
- Sanity-check из старой спеки: значение строки с отклонением >3× от типичного
  (qty×price vs sum уже есть) — просто убедиться, что попадает в warn-раздел.

## 3. Вне объёма
- Batch-обработка нескольких НЕЗАВИСИМЫХ накладных за раз (одна накладная = одна сессия).
- Провайдер-aware token-perfect resize (follow-up из scan-preprocessing-plan).
- A/B-харнесс качества (делается вручную при валидации).

## 4. Acceptance Criteria

- [ ] AC-1. Двух- и трёхстраничная накладная (фото) распознаётся в один InvoiceDraft:
      строки со всех страниц присутствуют; тест бэка с 2 фиктивными страницами (respx/mock
      провайдер) + ручная проверка на реальной многостраничной накладной.
- [ ] AC-2. `invoiceType` доходит до промпта: для paper-накладной распознанные серия/номер
      бланка попадают в соответствующие поля draft (ручная проверка на реальном фото);
      unit-тест: промпт содержит хинт выбранного типа.
- [ ] AC-3. Авто-кроп: на 5 реальных фото накладных на столе quad найден и предложен
      автоматически минимум в 4 случаях; ручные ручки всегда доступны; при отключённой
      сети/сломанном WASM поток не ломается (fail-open).
- [ ] AC-4. OpenCV.js не входит в основной бандл (проверка размера чанков `vite build`) —
      грузится лениво при открытии сканера.
- [ ] AC-5. Confidence-gate: накладная с невалидным номером бланка/расхождением сумм
      показывает секцию проблемных полей с переходом к полю; полностью валидная — нет.
- [ ] AC-6. Метрика: на контрольном наборе ≥5 реальных накладных точность распознавания
      позиций ≥95% (число верных ячеек название/кол-во/цена/сумма), время ≤30 сек —
      зафиксировать таблицей в `work/phase-s2.md`.
- [ ] AC-7. Общий DoD G10 (pytest+ruff+build, tenant/fail-closed тесты не сломаны,
      01_ARCHITECTURE бампнут: сигнатура OcrProvider, multi-page).
